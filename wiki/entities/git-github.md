# Git과 GitHub

**Category:** 개발 도구 / 버전 관리
**Status:** stable
**Last updated:** 2026-04-12

## TL;DR

**Git**은 소스 코드의 변경 이력을 기록하는 **버전 관리 시스템**으로, 특정 시점을 저장(commit)해 두면 언제든 과거로 돌아갈 수 있다. **GitHub**는 Git이 적용된 원격 코드 저장소이자, Issue·Pull Request·Actions 등 협업 기능을 제공하는 세계 최대 개발자 플랫폼이다.

## 설명

### Git — 코드의 타임머신

코드를 수정하다 "3시간 전으로 돌아가고 싶다"는 순간이 반드시 온다. PPT에 Ctrl+Z가 있다면, 코드에는 **Git**이 있다. SharePoint의 "버전 기록" 기능의 코드 전문 버전이지만, 자동 저장이 아니라 **내가 직접 의미 있는 시점을 골라서 저장(commit)** 한다는 차이가 있다.

### 핵심 명령어

| 명령어 | 역할 | 쉬운 비유 |
|---|---|---|
| **commit** | 현재 시점을 저장 ("무엇을 바꿨는지" 메시지 포함) | 게임 세이브 |
| **push** | 내 커밋을 원격 저장소(GitHub)에 업로드 | 클라우드에 백업 |
| **pull** | 원격 저장소의 최신 코드를 내 컴퓨터로 가져오기 | 최신 버전 다운로드 |
| **fetch** | 원격 저장소의 변경사항 확인 (병합 전 미리보기) | 업데이트 확인 |
| **fork** | 남의 프로젝트를 복사해서 내 버전으로 | 남의 세이브 파일 복사 |
| **clone** | 원격 저장소를 내 컴퓨터에 복제 | 프로젝트 다운로드 |

### 충돌 (Conflict)

여러 개발자가 하나의 프로젝트에서 **같은 파일, 같은 코드 라인**을 동시에 수정하면 코드 충돌(Conflict)이 발생할 수 있다. Git은 충돌 지점을 명확히 표시해 주며, 개발자가 어떤 코드를 살릴지 선택하여 해결한다.

### 브랜치 (Branch)

Git에는 항상 **마스터(메인) 브랜치**가 존재한다. 개발자들은 다음 흐름으로 작업한다:

1. 메인 브랜치를 복사해서 **새로운 브랜치** 생성
2. 그 안에서 커밋을 쌓으며 작업
3. 완료 후 해당 브랜치를 메인 브랜치로 **머지(병합)**

### Git 브랜치 전략

IT 회사에서는 브랜치를 어떻게 사용·관리할지 정하는 전략을 사용한다:
- **git-flow:** develop, feature, release, hotfix 등 여러 브랜치를 체계적으로 운영
- **trunk-based:** 메인 브랜치 하나에 자주 머지, 짧은 수명의 feature 브랜치만 사용

### GitHub — 원격 코드 저장소

GitHub는 **git이 적용된 원격 코드 저장소**이자, 전 세계에서 가장 많은 개발자가 사용하는 서비스다. 여러 개발자가 git의 커밋을 공유하려면 원격 저장소가 필요하며, 그 역할을 GitHub가 한다.

### GitHub의 주요 기능

| 기능 | 설명 |
|---|---|
| **Repository** | 프로젝트(repo) 단위로 모든 코드와 이력을 보관 |
| **Issue** | 버그 리포트, 기능 요청 등 개발자끼리 논의하는 공간 |
| **Pull Request** | 내 브랜치의 코드 변경을 메인에 병합하기 전 리뷰 요청 |
| **GitHub Actions** | 코드를 올리면 자동으로 빌드·테스트·배포 스크립트를 실행 |
| **README.md** | 프로젝트 소개 문서, repo 첫 페이지에 표시 |

### 오픈소스 문화

GitHub에는 수많은 **오픈소스** 프로젝트가 공개되어 있다. React, Vue, TensorFlow 등 유명 라이브러리가 GitHub에서 공개·관리된다. 누구나 fork해서 자신의 버전으로 수정하거나, Pull Request로 원본 프로젝트에 기여할 수 있다.
