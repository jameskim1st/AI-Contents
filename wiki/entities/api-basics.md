# API (Application Programming Interface)

**Category:** 소프트웨어 개발 / 기초
**Status:** stable
**Last updated:** 2026-04-12

## TL;DR

API는 **프로그램과 프로그램을 연결하는 중개자(매개체)** 다. 클라이언트가 서버의 내부 구조를 몰라도 약속된 규칙(URL + Parameter + Response)에 따라 데이터를 주고받을 수 있게 해준다. 식당에서 손님이 주방에 직접 들어가지 않고 종업원을 통해 주문하는 것과 같은 원리다.

## 설명

### API가 필요한 이유

프로그램끼리 통신할 때 다음 상황에서 API가 중개자 역할을 한다:

1. **접근할 프로그램의 규칙이 복잡한 경우** — API가 복잡한 내부 규칙을 간소화해서 노출.
2. **보안상 간접 접근이 필요한 경우** — 외부에서 데이터베이스에 직접 접근하지 않고, 제한된 기능만 API를 통해 제공.

### 식당 비유로 이해하기

| 식당 | 소프트웨어 |
|---|---|
| 손님 (주문하는 쪽) | **클라이언트** (프론트엔드) |
| 종업원 (주문을 전달하고 음식을 서빙) | **API** |
| 주방 (요리를 만드는 쪽) | **서버** (백엔드) |
| 음식 | **서비스 / 데이터** |

손님은 주방에 직접 들어가지 않는다. 프론트엔드도 백엔드 서버에 직접 접근하지 않고, **반드시 API(종업원)를 통해** 데이터를 요청하고 받는다.

### API 서버

API 서버는 데이터베이스를 대신하여 클라이언트와 통신하는 서버다. 로그인, 상품 정보 조회, 결제 등의 규칙이 정의되어 있으며, 클라이언트는 데이터베이스가 아닌 API 서버와 통신한다.

### API 문서 — 약속의 명세서

API를 사용하려면 **API 문서**를 읽어야 한다. 문서에는 다음이 약속되어 있다:

- **URL** — 어디로 요청할 것인가 (예: `https://api.weather.com/v1/current`)
- **Parameter** — 무엇을 보낼 것인가 (예: `city=Seoul`, `api_key=sk-abc123`)
- **Response** — 무슨 응답이 오는가 (예: JSON 형식의 날씨 데이터)

카카오 로그인 API를 예로 들면, 어디(URL)에 무엇(Parameter)을 보내면 무슨 응답(Response)을 줄지가 문서에 전부 명시되어 있다.

### REST API

가장 널리 쓰이는 API 설계 스타일이다. HTTP 프로토콜의 메서드를 활용한다:

| 메서드 | 역할 | 식당 비유 |
|---|---|---|
| **GET** | 데이터 조회 | "메뉴 보여주세요" |
| **POST** | 데이터 생성 | "주문할게요" |
| **PUT** | 데이터 수정 | "주문 변경이요" |
| **DELETE** | 데이터 삭제 | "주문 취소요" |

### 실제 요청/응답 예시 — 날씨 앱

```
요청(Request):   GET https://api.weather.com/v1/current?city=Seoul
응답(Response):  { "도시": "서울", "온도": 23, "날씨": "맑음" }
```

앱은 이 JSON 응답 데이터를 받아서 화면에 "서울 23도 맑음"으로 표시한다.
