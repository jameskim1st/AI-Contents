# 라이브러리와 프레임워크

**Category:** 소프트웨어 개발 / 기초
**Status:** stable
**Last updated:** 2026-04-12

## TL;DR

**라이브러리(Library)** 는 특정 기능을 미리 짜놓은 코드 뭉치로, 필요할 때 가져다 쓴다. **프레임워크(Framework)** 는 코드의 큰 뼈대(Frame)를 제공하여 그 안에서 개발할 수 있게 해준다. 둘 다 개발 속도를 높이는 핵심 수단이며, 경계가 모호할 때도 있다.

## 설명

### 라이브러리 (Library)

프로그램의 특정 기능을 수행하도록 **미리 짜여진 코드 뭉치**다.

예를 들어 브라우저에서 영상을 재생하는 기능을 직접 구현하려면 브라우저 대응, 해상도 변경, 배속 변경, HLS 대응 등 복잡한 작업이 필요하다. 하지만 영상 라이브러리를 사용하면 이 작업들이 이미 구현되어 있어 훨씬 수월하게 개발할 수 있다.

IT 서비스를 개발할 때는 실제로 많은 라이브러리를 사용한다. 그래야 더 빠르고 쉽게 개발이 가능하기 때문이다.

### 프레임워크 (Framework)

코드의 **큰 뼈대(Frame)를 제공**해서 그 뼈대 안에서 개발할 수 있도록 도와준다. 라이브러리와 마찬가지로 개발자가 미리 만들어 놓은 코드지만, 틀(아키텍처) 자체를 제공한다는 점이 다르다.

### 프론트엔드 프레임워크

사용자가 보는 화면(UI)을 구축하는 데 사용한다.

| 프레임워크 | 언어 | 특징 |
|---|---|---|
| **React** | JavaScript | Meta(Facebook) 개발, 컴포넌트 기반, 가장 큰 생태계 |
| **Angular** | TypeScript | Google 개발, 대규모 엔터프라이즈 앱에 강점 |
| **Vue** | JavaScript | 가벼움, 학습 곡선 낮음 |

### 백엔드 프레임워크

서버 쪽 로직(데이터 저장, 인증, 계산 등)을 처리한다.

| 프레임워크 | 언어 | 특징 |
|---|---|---|
| **Express** | Node.js (JS) | 가볍고 유연, 미니멀리스트 |
| **Django** | Python | "배터리 포함" 철학, 관리자 페이지 자동 생성 |
| **Flask** | Python | 마이크로 프레임워크, 필요한 것만 추가 |

### 라이브러리 vs 프레임워크

둘의 경계가 모호할 때도 있다. React는 공식적으로 "라이브러리"이지만 사실상 프레임워크처럼 쓰인다. **헷갈리면 넓은 의미로 "라이브러리"라 불러도 무방하다.**

핵심 차이는 **제어 흐름의 주도권**이다:
- **라이브러리:** 내 코드가 라이브러리를 호출한다 (내가 주도).
- **프레임워크:** 프레임워크가 내 코드를 호출한다 (프레임워크가 주도, Inversion of Control).

### 패키지 매니저

라이브러리와 프레임워크를 설치·관리하는 도구다.
- **npm** (Node.js / JavaScript 생태계)
- **pip** (Python 생태계)
- **cargo** (Rust), **Maven** (Java) 등 언어별로 존재
