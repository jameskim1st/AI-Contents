01

LLM이란 무엇인가

LLM(Large Language Model)은 방대한 텍스트 데이터를 학습하여 자연스러운 다음 단어를 예측하는 거대한 AI 모델입니다. 마치 세계 모든 요리책을 읽은 쉐프가 새로운 요리를 창작하듯, LLM은 데이터에서 패턴을 배워 새로운 텍스트를 생성합니다.

🍳 쉐프(Chef) 비유 — LLM의 두 단계
📚

학습 단계 (Training)

세계의 모든 요리책, 레시피, 리뷰를 읽고
재료 조합의 패턴을 익힘

인터넷 텍스트 책 논문 코드
"짠맛 + 단맛 → 간장 소스" 같은 패턴을 수조 개 학습
🍽️

생성 단계 (Inference)

새로운 주문(프롬프트)을 받으면
학습한 패턴을 바탕으로 새 요리를 창작

프롬프트 입력 패턴 매칭 텍스트 생성
"한국 수도는?" → 학습 패턴에서 "서울"을 높은 확률로 선택
🎯 LLM의 핵심 정의
LLM = 확률적 다음 토큰 예측기
LLM은 "이해"하는 것이 아니라 "가장 그럴듯한 다음 단어를 확률적으로 선택"합니다.
세계의 모든 텍스트를 읽은 초능력 자동완성 기능이라고 생각하세요.
"한국의 수도는" → "서울" (확률 85%)

02

언어 모델의 역사

오늘날의 LLM은 하루아침에 나온 것이 아닙니다. 수십 년간의 연구가 축적되어 폭발적인 혁신을 만들어냈습니다. N-gram에서 에이전트 시대까지, 핵심 전환점을 따라가 봅시다.

📜 언어 모델 진화의 타임라인
📊 ~2010s 통계적 언어 모델
N-gram 기반 — "이전 N개 단어"를 보고 다음 단어를 확률로 예측. 간단하지만 긴 문맥 이해 불가.
🔄 2014-2017 RNN / LSTM / Seq2Seq
순환 신경망 — 문장을 순서대로 읽으며 문맥을 기억. Google 번역에 적용. 하지만 긴 문장에서 앞부분을 잊는 문제.
⚡ 2017 Transformer 등장 GAME CHANGER
Ashish Vaswani et al. (Google Brain, "Attention is All You Need" 저자 8인)이 발표. Self-Attention 메커니즘으로 문장 전체를 동시에 처리. RNN의 한계를 돌파한 혁명적 구조.
🌿 2018-2019 BERT / GPT / T5 분화
Transformer 기반 세 갈래: BERT(이해 특화, Google), GPT(생성 특화, OpenAI), T5(이해+생성, Google). 각각 다른 강점으로 분화.
🚀 2020-2022 GPT 스케일링 → ChatGPT
GPT-3(175B)가 "크면 클수록 똑똑해진다"를 증명. 2022.12 ChatGPT 출시 → 2개월 만에 1억 사용자 돌파, AI 대중화 시대 개막.
⚔️ 2023 LLM 대전쟁
GPT-4 출시 → Claude, Gemini, LLaMA 2 연이어 등장. 멀티모달(이미지 입력) 시대 시작. 오픈소스 vs 클로즈드 경쟁 본격화.
🧠 2024 Reasoning 모델 + Multimodal 표준화 NEW ERA
OpenAI o1 출시 — "생각하는 모델"의 시작. 수학/코딩에서 인간 전문가 수준. 이미지·오디오·비디오 입력이 기본 기능이 됨. LLaMA 3 (405B) 오픈소스 최대 모델 등장.
🌊 2025 Context 1M · SLM · DeepSeek 충격
컨텍스트 윈도우 1M 토큰이 표준화 (Gemini, Claude). SLM(Small Language Model) 부상 — Phi-4, Gemma 2가 작은 크기로 놀라운 성능. DeepSeek R1이 $5.5M 학습비로 GPT-4급 추론 달성, "중국발 AI 충격" 촉발.
🤖 2026 Thinking Model · MoE · 에이전트 시대 NOW
"Thinking model"이 주류 — Claude, Gemini, GPT 모두 내장 추론 모드 탑재. MoE 아키텍처가 사실상 표준 (LLaMA 4: 128 experts). AI Agent가 코드 작성·배포·검색을 자율 수행하는 "에이전트 시대" 본격 개막.
🔑 역사에서 배우는 3가지 전환점
2017 — Transformer
병렬 처리가 가능해지면서 모델을 거대하게 키울 수 있게 됨
2022 — ChatGPT
일반인이 AI를 직접 사용하기 시작. 산업 전체가 변화
2024 — Reasoning
LLM이 "생각"하기 시작. 단순 예측 → 추론 능력 획득
핵심 포인트

2017년 Transformer가 등장하기 전과 후는 완전히 다른 세상입니다. 오늘날 모든 주요 LLM(GPT, Claude, Gemini, LLaMA)은 Transformer 구조 위에 만들어졌습니다. 그리고 2024년 Reasoning 모델의 등장은 또 한 번의 패러다임 전환입니다.

03

Transformer 아키텍처

2017년 Ashish Vaswani et al. (Google Brain, "Attention Is All You Need" 논문 저자 8인)이 제안한 Transformer는 AI 역사상 가장 영향력 있는 아키텍처입니다. 현재 모든 LLM의 기반이 되는 이 구조를 직관적으로 이해해 봅시다.

⚡ Transformer 아키텍처 — 전체 흐름

Self-Attention — Transformer의 핵심 메커니즘

💡 Self-Attention이 문맥을 이해하는 방법

⚡ Transformer의 Self-Attention

"I saw the bank by the river"

모든 단어가 동시에 서로를 본다. "bank"가 "river"를 한 번에 참조 → 즉시 강변(river bank)으로 해석.

🔑 Query, Key, Value

Query (Q): "내가 누구를 찾고 있나"
Key (K): "나는 이런 정보를 갖고 있어"
Value (V): "내가 전달할 실제 정보"
Q와 K를 비교(dot product) → 점수가 높은 단어의 V를 더 많이 가져옴
🔍 Multi-Head Attention — 여러 관점으로 동시에 보기
Attention을 여러 "헤드"로 분할하여 각각 다른 관계를 포착합니다.
Head 1
문법 관계
(주어-동사)
Head 2
의미 관계
(bank-river)
Head 3
위치 관계
(인접 단어)
Head N
기타 패턴
(대명사 참조 등)
GPT-4: 96~128 헤드가 동시에 서로 다른 관계를 포착 → 결합하여 풍부한 문맥 이해

Transformer의 동작 과정 — 단계별로 열어보기

Transformer가 실제로 문장을 처리하는 과정을 "I am a student" → "je suis étudiant" 번역 예제로 한 단계씩 열어봅시다.

Step 1 — 블랙박스: 일단 결과만 보자
영어 문장이 들어가면 프랑스어 문장이 나온다 — 내부는 아직 모른다.
Step 2 — 상자를 열면: 인코더 + 디코더
Transformer 내부에는 인코더(입력을 이해)와 디코더(출력을 생성), 두 파트가 있다.
Step 3 — 각각 N개의 층이 쌓여 있다
원래 논문(2017)은 인코더 6층 + 디코더 6층. 각 층에서 Self-Attention + FFN이 반복된다.
인코더의 최종 출력이 디코더 각 층에 Cross-Attention으로 전달된다.
Step 4 — 인코더 = 이해(NLU), 디코더 = 생성(NLG)
📥
Encoder
NLU — Natural Language Understanding
입력 문장의 의미를 "이해"하는 역할.
"I am a student"의 모든 단어 관계를 파악하고,
의미 벡터(Context Vector)로 압축한다.
📤
Decoder
NLG — Natural Language Generation
이해한 의미를 새 문장으로 "생성"하는 역할.
인코더가 준 의미 벡터를 받아서,
한 단어씩 출력 문장을 만들어 낸다.
💡 BERT는 인코더만, GPT는 디코더만 사용 — 바로 여기서 갈라진다.

2026년 LLM 대부분은 Decoder-only (GPT 계열). 하지만 원래 Transformer는 Encoder-Decoder였다.
Step 6 — 토큰이 생성되는 과정 (Autoregressive Generation)

인코더는 "I am a student"를 한 번만 처리. 디코더는 한 토큰씩 반복 생성한다.

단계 📥 Encoder 출력 📤 Decoder 입력 생성된 토큰
6a I am a student <sos> → je
6b I am a student <sos> je → suis
6c I am a student <sos> je suis → étudiant
6d I am a student <sos> je suis étudiant → <eos>
🔑
<sos> = Start of Sequence (시작 신호)  |  <eos> = End of Sequence (종료 신호)
디코더는 이전에 생성한 토큰을 다시 입력으로 받아 다음 토큰을 예측한다. 이미 생성된 토큰 = 흐린 색, 새로 생성된 토큰 = 밝은 보라색.
Step 7 — 핵심 인사이트: Autoregressive Generation

이것이 바로 Ch.06 문장 생성에서 배울 핵심 원리입니다. Transformer의 Decoder는 한 번에 하나의 토큰만 생성하고, 이전에 생성한 토큰을 다시 입력으로 사용합니다. 이를 Autoregressive Generation(자기회귀 생성)이라고 합니다.

💡 GPT(Decoder-only)도 같은 원리 — 인코더 없이 디코더만 사용하지만, "이전 토큰들 → 다음 토큰 예측"이라는 autoregressive 메커니즘은 동일합니다. ChatGPT가 답변을 한 글자씩 스트리밍하는 이유가 바로 이것입니다.

MoE (Mixture of Experts) — 2026년의 핵심 아키텍처

🏥 MoE = "전문의 팀 진료 시스템"
🏥
종합병원에 128명의 전문의가 있지만...
환자(입력 토큰)마다 필요한 1~2명의 전문의만 진료합니다. 전체 인력은 거대하지만 한 번에 투입되는 자원은 효율적. AI 모델도 똑같은 원리입니다.
🧱 Dense vs ⚡ MoE — 핵심 비교

🧱 Dense 모델 (기존 방식)

모든 파라미터 활성 (100%)
모든 입력에 모든 파라미터가 계산에 참여.
GPT-3 (175B) → 매번 1,750억 개 전부 동원.
크면 클수록 느리고 비쌈

⚡ MoE 모델 (새로운 방식)

필요한 전문가만 활성 (~4%)
입력에 따라 Router가 전문가 1~2명만 선택.
LLaMA 4 Maverick: 2T total → 매번 ~17B만 동원.

MoE의 핵심 이점 정리

총 파라미터는 크게, 추론 비용은 작게. MoE 덕분에 모델을 수조 단위로 키우면서도 추론 시에는 일부만 사용하여 실용적인 비용을 유지합니다. Noam Shazeer (Google Brain, Transformer 공동 저자이자 MoE의 LLM 적용 선구자, Character.AI 공동 창업자)가 2017년 제안한 원조 MoE 논문이 7년 만에 업계 표준이 되었습니다.

04

토큰과 임베딩

LLM은 글자를 하나씩 읽지 않습니다. 토큰(Token)으로 텍스트를 쪼개고, 임베딩(Embedding)으로 의미를 벡터로 변환합니다. 이 두 단계가 LLM의 비용, 속도, 성능을 모두 결정합니다.

🔤 토큰화(Tokenization) — 문장이 토큰으로 쪼개지는 과정
원본 텍스트:
"인공지능이 세상을 바꿉니다"
↓ Subword Tokenization (BPE)
인공 지능 이 세상 을 바꿉 니다
↓ Token ID 변환
4521 8903 127 6234 89 15678 2345
"인공지능" → "인공" + "지능" (Subword 분할) = 총 7토큰
⚙️ BPE (Byte-Pair Encoding) — 토큰화 알고리즘의 작동 원리
1
시작: 모든 글자를 독립 토큰으로
l o w e r → [l] [o] [w] [e] [r]
2
가장 자주 연속하는 쌍을 합침
[l] [o] [w] → [lo] [w] (l+o가 최빈 쌍)
3
반복 → 어휘 크기 도달까지
[low] [er] → "lower" = 2토큰
자주 쓰는 단어는 통째로 1토큰, 드문 단어는 여러 조각으로. GPT-4o의 어휘: 200,019개 토큰 (o200k_base)
🌏 영어 vs 한국어 토큰화 차이 — 한국어가 더 비싼 이유

English

"Hello, the weather is nice today"
31글자 = ~8 토큰

한국어

"안녕하세요, 오늘 날씨가 좋네요"
13글자 = ~10-17 토큰
같은 내용이라도 한국어는 토큰을 2~3배 더 사용 → 비용이 더 높고, 컨텍스트 윈도우를 더 빨리 소모합니다

임베딩 — 단어를 벡터로 변환하기

🧮 임베딩이란? — 의미를 숫자로 표현
비슷한 의미의 단어는 벡터 공간에서 가까이 배치됩니다.
King - Man + Woman ≈ Queen
임베딩은 의미의 수학적 연산을 가능하게 합니다.
이것이 LLM이 "유사한 개념"을 이해하는 기반입니다.
🔄 Tokenization → Embedding 파이프라인
"안녕"
원본 텍스트
→
[234, 12345]
Tokenization (정수 ID)
→
[0.12, -0.5, ..., 0.8]
Embedding (벡터)
Tokenization = 텍스트 → 정수 (사전 기반)  |  Embedding = 정수 → 고차원 벡터 (학습된 변환)

토큰 = 택시 미터기. 대화가 길수록(토큰 사용량 증가) 비용이 올라가고, 택시가 갈 수 있는 최대 거리(컨텍스트 윈도우)도 정해져 있습니다. 한국어는 영어보다 "미터기가 빨리 올라가는" 셈이죠.
토큰화의 악명 높은 함정들
Leading space: "hello"와 " hello"는 다른 토큰
숫자 문제: "1234"가 ["123","4"]로 쪼개져 수학 성능 저하
05

LLM 학습 파이프라인

LLM이 "세상의 모든 텍스트를 읽은 초능력 자동완성"이 되기까지는 3단계의 체계적인 학습 과정을 거칩니다. 각 단계의 목적, 데이터, 비용이 완전히 다릅니다.

🏗️ LLM 학습 3단계 파이프라인
1
📚 Pre-training (사전학습)
목적: 세상의 모든 텍스트 패턴을 학습하여 "다음 토큰 예측기" 만들기
데이터: Common Crawl, Wikipedia, Books, Code — 15T+ 토큰
비용: $100M+ (1,300억원+) · 수만 GPU × 수개월
결과: 똑똑하지만 아직 "도움이 되는 어시스턴트"는 아닌 상태
2
🎓 Fine-tuning / SFT (지도 미세조정)
목적: "질문에 답하는 어시스턴트"로 스타일과 형식 학습
데이터: (instruction, response) 쌍 — 수천~수십만 개
비용: $10K~$1M · LoRA/QLoRA로 저비용 가능
결과: 질문에 답하지만 가끔 위험하거나 부정확한 답변
3
👤 RLHF / DPO (인간 피드백 정렬)
목적: "도움이 되고, 해롭지 않고, 정직한" Alignment
데이터: (prompt, 좋은 답, 나쁜 답) 선호 쌍 — 수천~수만 개
비용: $100K~$10M · 인간 평가자 비용 포함
결과: 안전하고 유용한 최종 AI 어시스턴트 = ChatGPT, Claude 등

RLHF vs DPO — 인간 피드백 통합의 두 가지 방식

⚖️ RLHF vs DPO 비교

RLHF

Reinforcement Learning from Human Feedback
단계: SFT → Reward Model 학습 → PPO 강화학습 (3단계)
장점: 품질이 압도적. ChatGPT가 이것으로 탄생
단점: 복잡하고 불안정. Reward Model 별도 필요
주 사용: OpenAI, Anthropic (대형 팀)
비용: $1M~$10M+

DPO

Direct Preference Optimization
단계: Preference 데이터로 직접 정책 업데이트 (1단계)
장점: 안정적, 구현 쉬움. Reward Model 불필요
단점: 초대형 모델에서는 RLHF보다 약간 열세
주 사용: 오픈소스 (Llama 3, Mistral, Qwen)
비용: $10K~$100K
Rafael Rafailov et al. (Stanford, 2023, DPO 논문 저자)가 제안. 2024년 이후 오픈소스 모델의 사실상 표준이 됨.

06

문장 생성 원리

LLM이 문장을 만드는 과정은 한 번에 하나의 토큰을 생성하는 자기회귀(Autoregressive) 방식입니다. 이 과정에서 Temperature, Top-K, Top-P가 결과의 창의성과 정확성을 결정합니다.

🔄 Autoregressive 생성 — 토큰 하나씩 순서대로
Step 1: 나는 → 오늘
Step 2: 나는 오늘 → 학교에
Step 3: 나는 오늘 학교에 → 갔다
Step 4: 나는 오늘 학교에 갔다 → [EOS]
매 Step마다 전체 토큰 시퀀스를 다시 입력하여 다음 토큰의 확률을 계산합니다.
[EOS] = End of Sequence, 모델이 "문장이 끝났다"고 판단하면 생성 중단.

🌡️ Temperature가 확률 분포를 바꾸는 원리
T = 0.0

99%
부산
~0%
항상 같은 답
코딩 사실 확인
T = 0.7

15%

Top-K vs Top-P (Nucleus Sampling)

🎯 Sampling 전략 비교

Top-K Sampling

확률 상위 K개의 토큰만 후보로 남기고 나머지 제외.
Top-K = 3 → "서울", "부산", "대한" 만 후보
장점: 구현 단순
단점: 확률 분포 무시. K=3이면 꼬리 확률 50%여도 3개만 선택

Top-P (Nucleus) Sampling

누적 확률이 P%에 도달할 때까지의 토큰만 후보로.
Top-P = 0.9 → 누적 90% 도달까지 포함
장점: 확률 분포에 적응적. 가변적 후보 수
단점: 약간 더 복잡한 구현
실전에서는 Temperature + Top-P 조합이 가장 널리 사용됩니다. (예: T=0.7, Top-P=0.95)
⚙️ 실전 설정 가이드
용도 Temperature Top-P 이유
코드 생성 0.0 1.0 문법 오류 0이 목표
사실 기반 Q&A 0.1~0.3 0.9 정확성 최우선
일반 대화 0.7 0.95 자연스러운 다양성
창작 / 브레인스토밍 1.0~1.5 0.99 의외의 조합 환영

Structured Output — 에이전트 시대의 필수 기술

🔧 Structured Output이란?
LLM이 자유 텍스트 대신 JSON, enum 등 검증 가능한 형식으로 출력을 생성하는 기능. 에이전트가 도구를 호출할 때 파라미터가 정확한 형식이어야 하므로 에이전트 안정성의 기반입니다.

문제: 자유 텍스트 출력

가끔: {"name": "홍길동", "age": 30}
가끔: ```json {"name":"홍길동"}```
가끔: 네, 결과입니다: {이름: 홍길동}
키 이름, 타입, 래핑이 매번 달라 파싱 실패

해결: Constrained Decoding

항상: {"name": "홍길동", "age": 30}

✅ 100% schema 준수 보장
schema 위반 토큰의 확률을 0으로 마스킹

Structured Output의 핵심 철학: LLM이 무엇을 말할지는 제어할 수 없지만, 어떤 형식으로 말할지는 100% 제어할 수 있습니다. "age": 30이 사실인지는 모르지만, 적어도 integer 타입은 보장됩니다. 이것이 LLM을 소프트웨어 시스템에 안정적으로 통합하기 위한 핵심 전제 조건입니다. 2024년 8월 OpenAI가 업계 최초로 공식 도입한 이후 표준이 되었습니다.

Temperature = 주방의 레시피 자유도. T=0은 "레시피를 정확히 따라라" (항상 같은 요리), T=1.5은 "즉흥 요리 OK" (때론 걸작, 때론 재앙). Top-P는 "재료 선택 범위" — 0.9면 상위 90% 검증된 재료만, 0.99면 실험적 재료까지 허용.
07

Context Engineering

"좋은 프롬프트 한 줄"의 시대는 끝났습니다. 2025년부터 AI의 성능을 결정하는 것은 LLM이 받는 전체 입력(Context)을 설계하는 능력 — Context Engineering입니다.

🪑 Context = LLM의 작업 책상
책상이 작으면 메모장 하나만 펼칠 수 있지만, 거대한 책상이라면 설계도, 참고서, 계산기, 이전 작업 기록을 모두 펼쳐놓고 복잡한 프로젝트를 수행할 수 있습니다. Context Window가 LLM의 책상 크기이고, 그 위에 무엇을 어떻게 배치할지 설계하는 것이 Context Engineering입니다.
"Prompt engineering is dead. The real skill is context engineering."
— Andrej Karpathy, 2025 (Tesla AI 전 디렉터, OpenAI 공동창립 멤버, AI 교육 유튜버)

RAG vs Long Context — 상호보완적 관계

⚖️ "Long Context가 RAG를 죽이나?" — 2026년 답: 함께 쓴다

RAG가 여전히 필요한 이유

  • 💰 비용 — 1M 토큰 입력 = ~$3. 매 질문마다 전체를 넣으면 비용 폭발
  • 🔄 최신성 — 실시간 변하는 문서는 검색이 효율적
  • 📚 규모 — 수백만 건 중 관련 문서만 찾는 경우

Long Context가 우세한 영역

  • 📖 고정 문서 — 매뉴얼, 코드베이스처럼 고정된 소스
  • 🔬 정밀 분석 — chunking 시 맥락 손실이 큰 경우
  • 🏗️ 전체 맥락 — 코드 리팩토링, 법률 문서 분석
2026년 표준: Hybrid — RAG로 검색·필터링 → 결과를 긴 컨텍스트에 통째로 주입 → LLM이 풍부한 맥락에서 답변

08

추론 모델 혁명 (Reasoning Models)

기존 LLM은 즉답형 퀴즈쇼였습니다. 추론 모델은 시험 시간을 더 주는 논술 시험입니다. "더 오래 생각하면 더 좋은 답을 낸다" — 이 단순한 원리가 LLM의 새로운 패러다임을 열었습니다.

📝 비유: 퀴즈쇼 vs 논술 시험
기존 LLM은 "3초 안에 답하세요!"라는 즉답형 퀴즈쇼입니다. 추론 모델은 "30분 동안 풀이 과정을 적으며 답하세요"라는 논술 시험입니다. 같은 학생이라도 생각할 시간이 있으면 더 어려운 문제를 풀 수 있습니다.

패러다임 전환: Train-time → Test-time Scaling

⚡ 두 종류의 Compute Scaling
🏋️

Train-time Scaling (2020~2024)

더 큰 모델 + 더 많은 데이터 + 더 많은 학습 연산 = 더 나은 성능

GPT-3(175B) → GPT-4(~1.8T MoE) → Llama 3(405B)
⚠️ 2024년 중반: 수확 체감(diminishing returns)에 직면
🧠

Test-time Scaling (2024~)

추론 시점에 더 많은 연산 투입 = 더 오래 생각 = 더 나은 답

Thinking 토큰 x 탐색 경로 x 검증 연산
✅ OpenAI: "test-time scaling도 power law를 따른다"

Test-time Compute 기법들

🔬 "더 오래 생각하기"의 구체적 방법들
기법 설명 검증 방식
Majority VotingN개 답 생성 후 다수결없음 (다수결)
Best-of-NN개 답 중 검증기가 최고점 선택ORM / PRM
Beam Search + PRM각 추론 단계에서 높은 점수 경로만 유지PRM (단계별)
MCTS게임 AI처럼 트리 탐색으로 추론 경로 탐험PRM
STaR모델이 자신의 reasoning으로 자기를 재학습Self-taught
ORM = Outcome Reward Model (최종 답만 평가)  |  PRM = Process Reward Model (각 풀이 단계를 평가 — OpenAI "Let's Verify Step by Step" 논문, 2023)

DeepSeek R1 — 추론의 민주화

2025년 1월, 중국 AI 스타트업 DeepSeek(량원펑 창업, 헤지펀드 High-Flyer 설립)이 공개한 오픈소스 reasoning LLM입니다. 가장 놀라운 발견은 R1-Zero 실험 — SFT(지도 미세조정) 없이 RL(강화학습)만으로 모델이 자발적으로 "생각하는 법"을 학습했다는 것입니다.

🏗️ DeepSeek R1 — 5단계 파이프라인
1
Cold Start Data
R1-Zero 출력에서 가독성 좋은 CoT 수천 개 수집 → 소량 SFT 워밍업
2
Reasoning-oriented RL
수학/코딩/과학 등 정답 검증 가능한 도메인에서 강화학습 (GRPO 알고리즘)
3
Rejection Sampling
~80만 개 reasoning 샘플 + 20만 개 일반 SFT 데이터 확보
4
SFT (Supervised Fine-Tuning)
Reasoning 능력 + 일반 대화 능력 동시 확보
5
RL for All Scenarios
Reasoning + 일반 도메인 최종 RL. 안전성·유용성 보상 추가
훈련 비용 약 $5.5M — OpenAI o3 추정($500M~$1B)의 1/100

저지연

비용 제로

LLM의 한계와 미래

LLM은 놀라운 도구이지만 완벽하지 않습니다. 6가지 한계를 정확히 이해하고, 이를 극복하는 방법과 LLM이 향하는 미래를 살펴봅니다.

LLM의 6대 한계

🌀
