DEFINITION

하네스 엔지니어링 (Harness Engineering)

AI 모델(LLM)이 실제 업무를 수행할 수 있도록, 모델 외부에서 도구, 지식, 검증, 제약, 피드백을 하나의 실행 구조로 설계하는 활동.

AI 팀원 한 명이 조직에서 일하려면, 그를 둘러싼 시스템(지침 문서, 도구 접근권, 결재선, 품질 기준, 회고 절차)이 반드시 함께 있어야 합니다. 하네스는 바로 이 "둘러싼 시스템"의 공학적 명명입니다.

WHY "HARNESS"?

왜 "Harness"라는 이름이 붙었을까

"Harness"는 영어로 마구(馬具). 말의 힘을 억누르는 것이 아니라, 올바른 방향으로 집중시켜 빠르고 정확하게 달리게 하는 장비입니다. 초원을 자유롭게 뛰어다니던 야생말을 경마장에서 의도대로 달리게 하려면 반드시 필요하죠. AI 모델도 똑같습니다.

1-3. 시대의 중첩

Prompt ⊂ Context ⊂ Harness (안쪽 정의가 바깥에 포함됨)
Harness Engineering
에이전트가 일할 수 있는 전체 시스템을 어떻게 설계하는가
Context Engineering
모델이 어떤 정보를 기반으로 판단하도록 설정하는가
Prompt Engineering
모델에게 무엇을 어떻게 질문하는가
각 시대가 이전 시대를 대체가 아닌 포함합니다. 직원이 쌓은 프롬프트 기술은 컨텍스트 엔지니어링에서, 컨텍스트 설계 역량은 하네스 엔지니어링에서 그대로 활용됩니다. 안쪽 능력 없이 바깥만 잘하는 것은 불가능합니다.

1-4. 핵심 용어 사전

기술 용어의 원래 정의 + HR 협업 시나리오 번역
기술 용어 원래 정의 HR 협업 예시
CLAUDE.md
AGENTS.md		 프로젝트 루트에 두는 마크다운 파일로, AI 에이전트가 새 세션을 시작할 때마다 가장 먼저 읽는 프로젝트별 지침 문서. 아키텍처 규칙, 금지사항, 실행 명령어가 담긴다. HR이 작성한 "AI 채용 에이전트의 기본 지침 문서". 공정성 기준, 금지 질문, 평가 가중치가 여기 명시됨.
Hooks 에이전트의 작업 생명주기 중 특정 시점에 자동 실행되는 스크립트. 예: 코드 저장 직전 타입체크, 테스트를 자동 실행하고 실패 시 저장 자체를 차단. AI가 "연봉 데이터를 외부로 보내는 순간" 자동으로 차단하고 경고하는 규칙. 사람 결재 없이도 사전 차단.
Skills 에이전트가 특정 작업을 수행할 때 참조하는, 필요 시 로드되는 가이드 및 절차 문서. 평소엔 비활성, 필요할 때만 불러 씀. AI가 "경조사 휴가 처리" 업무에 들어갈 때만 참조하는 SOP 문서. 평소 컨텍스트를 차지하지 않음.
Sub-agents 특정 영역에 특화된 하위 에이전트. 전체 작업 중 한 분야를 전담하여 메인 에이전트의 컨텍스트를 격리하고 보호함. "급여 계산 전담 AI", "복리후생 전담 AI". 메인 HR 에이전트가 각 전담 AI에게 위임 후 결과만 받음.
Feedback Loop 에이전트가 같은 실패를 반복할 때 이를 감지하고 하네스(지침 및 규칙) 자체를 업데이트하는 자기 교정 메커니즘. AI가 같은 유형의 잘못된 승진 추천을 두 번 하면, 승진 가이드라인 문서에 해당 케이스가 자동 추가되어 재발 방지.
Context Rot 에이전트의 컨텍스트 창(한 번에 볼 수 있는 정보량)이 채워질수록 이전 정보를 점진적으로 잊거나 판단이 흐려지는 현상. (Act 2에서 상세)
Guides & Sensors Martin Fowler 제안. Guides는 에이전트가 행동하기 전에 방향을 잡아주는 사전 통제. Sensors는 행동 후에 결과를 검증하고 교정하는 사후 통제. (Act 2에서 상세)
Ralph Loop Geoffrey Huntley (오픈소스 개발자, DevRel 분야 저명 기고자) 제안. 에이전트가 단일 프로세스 안에서 실행 → 검증 → 수정을 반복하며 스스로를 진화시켜 목표에 도달할 때까지 문제를 해결하는 루프 모델. (Act 3에서 상세)

1-5. 코딩을 넘어 업무 전반으로

하네스는 코딩 에이전트에서 시작된 개념이지만, 본질은 "업무 에이전트 전반의 운영 구조"입니다. 조직에서 이미 존재하던 업무 분장, 표준 절차(SOP), 결재선, 품질 기준, 회고 루프가 곧 업무 하네스의 구성요소입니다. HR이 수십 년간 쌓아온 조직 설계 역량이 그대로 하네스 엔지니어링의 언어로 번역됩니다.

왜 모델만으로는 부족한가
🧠
모델만

  • 상태 유지 불가 (세션 간 기억 없음)
  • 실행 불가 (파일, 시스템 접근 없음)
  • 환경 접근 불가 (외부 도구 호출 없음)

🧠+🔧
모델 + 하네스
실제 업무 수행 가능.
  • 지식 문서로 상태 유지
  • 샌드박스로 안전 실행
  • 도구 호출로 환경 접근
  • 검증 게이트로 품질 보장

2-2. 2대 근본 문제와 해결책

하네스가 존재해야 하는 두 가지 구조적 이유
문제 ①

Context Rot (컨텍스트 부패)

정의: 컨텍스트 창이 꽉 차면 AI가 앞 내용을 잊거나 판단이 흐려짐.
해결책
CLAUDE.md를 매 세션 첫 읽기
🤝 협업 시나리오: 새로 투입된 AI 담당자가 "팀의 헌법이자 최우선 원칙"을 한 장 문서로 매번 확인한 뒤 일을 시작. 이 한 장만 있으면 다른 것이 흐려져도 원칙은 흔들리지 않음.
문제 ②

규칙과 울타리 부재

정의: 정보는 있어도 위험한 행동을 막는 구조가 없으면 사고 발생.
해결책
Hooks로 작업 완료 직전 자동 차단
🤝 협업 시나리오: 승인권 없는 직원이 결재 없이 대외송금 못 하도록 시스템이 강제하는 것과 같은 원리. 본인의 의지나 기억에 의존하지 않는 안전장치.

2-3. 프롬프트는 부탁, 하네스는 강제

🙏
프롬프트 = 부탁
"이거 하지 마세요"
AI가 잊거나 오해하거나 또 실수할 수 있음. 반복 발생 가능.
🛡️
하네스 = 강제
실수 자체가 불가능한 구조
시스템이 원천 봉쇄. 재발 불가.
🏭
공장 안전 시스템 비유
안전모를 안 쓰면 출입문 자체가 열리지 않는 공장. 작업자가 "안전모 써주세요"라는 공지를 잊어도 사고는 발생하지 않습니다.
HR TRANSLATION
사내 규정을 "공지"하는 것과 "시스템으로 자동 강제"하는 것의 차이. 하네스는 후자입니다. 규정 교육 대신 규정 위반 자체가 불가능한 환경을 설계하는 일.

3-2. 단계별 협업 풍경

각 단계에 HR 직무 3건 예시
🔹

In the Loop (현재)

원 정의: 사람이 AI의 매 단계 출력을 확인하고 수정하는 협업 방식.
💻 개발자 예시: 엔지니어가 AI 출력을 한 줄씩 "이렇게 고쳐" 지시.
📊 HR Analyst
AI 결과를 매번 검토하고 수정
🎯 채용 담당자
AI가 요약한 이력서를 하나하나 재확인
📝 L&D 매니저
AI가 초안한 교육 자료를 문장 단위 편집
결정적 순간: 결과물을 직접 수정.
🔸

On the Loop (전환 지점) ⭐

원 정의: 사람이 하네스(지침, 규칙, 검증)를 설계하고, AI는 그 안에서 자율 실행. 사람은 모니터링과 예외 처리를 담당.
💻 개발자 예시: CLAUDE.md + Hooks 설계 → AI가 자율 구현.
📊 HR Analyst
분석 SOP + 데이터 품질 규칙(Hooks) 설계 → AI가 매주 자동 리포트
🎯 채용 담당자

결정적 순간: 하네스(시스템) 수정. 직무 본질의 전환점.

Out of the Loop (미래)

원 정의: AI가 스스로 Ralph Loop(실행 → 검증 → 수정 반복)로 하네스를 진화시킴. 사람은 최초 목표와 초기 하네스만 제공.
🤝 HR 협업 예시
AI 에이전트가 자기 실패 케이스를 분석해 자기 온보딩 문서를 자동 업데이트. 예: 신입 사원 FAQ 챗봇이 답변 만족도 저조 질의를 주간 단위로 집계하여 자기 지식 문서를 스스로 보강.
HR 함의: 직원 역할이 설계자에서 큐레이터이자 감독으로 한 단계 더 이동.

KEY INSIGHT

3단계는 선형 발전이 아니라 업무 특성별 공존

🔹 In the Loop 유지: 고위험 업무 (해고, 보상 결정, 법적 대응)
🔸 On the Loop: 반복 업무 (데이터 리포트, 1차 스크리닝, 정기 공지)
🔷 Out of the Loop: 지속 개선 영역 (FAQ 챗봇, 문서 관리, 온보딩 자료)
CHRO 질문의 전환: "언제 전환하지?"가 아니라 "어느 업무를 어느 단계에 배치하지?"로 바뀝니다. 포트폴리오 관점.

하네스 엔지니어링은 개발자만의 이야기가 아닙니다

조직의 직무를 재정의하는 원리이자, HR이 주도해야 할 변화입니다. 엔지니어링의 언어로 쓰여 있지만, 그 실천의 본질은 조직 설계이자 직무 설계입니다.

채용, 온보딩, 평가, 학습, 보상. 모든 HR 영역이 "시스템 설계"의 언어로 다시 쓰입니다. 이 변화를 IT에 맡기는 조직과 HR이 주도하는 조직의 미래는 다를 것입니다.
