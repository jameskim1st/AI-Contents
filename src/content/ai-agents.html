01

LLM에서 Agent로 — 왜 진화가 필요했나

Part 0에서 LLM이 다음 토큰을 확률로 예측하는 모델임을 배웠습니다. 그렇다면 왜 "AI 에이전트"라는 개념이 등장했을까요? 답은 단순합니다 — 순수 LLM만으로는 할 수 없는 일이 너무 많기 때문입니다.

🔗 Part 2 간단 리뷰: LLM은 토큰(텍스트 최소 단위)을 하나씩 예측하여 문장을 만듭니다. "서울"이라는 답도 사실은 확률 85%로 "뽑힌" 결과. 자세한 원리는 Part 2 LLM 완전 이해에서 다뤘습니다.
💡 이 챕터의 초점 — LLM의 한계를 정확히 이해하자
Part 0가 "LLM이 어떻게 작동하는가"였다면, Part 3의 출발점은 "LLM 혼자서는 무엇을 못하는가"입니다. 이 한계들이 바로 Memory · Tools · Planning — 즉 에이전트 구성 요소의 등장 배경입니다.

🚫 LLM의 한계 ②: 정확한 계산

LLM은 복잡한 수학 연산에서도 자주 실패합니다. 이는 LLM이 실제 계산을 하는 것이 아니라, 가장 그럴듯한 다음 토큰을 예측하기 때문입니다.

▎ 수학 계산의 실패

⚠ 정답은 724.8이지만, 확신 없는 확률적 추측일 뿐!
🤔 왜 다음 토큰 예측이 계산 오류로 이어지나

"245 × 37 ÷ 12.5"는 학습 데이터에 거의 없는 패턴입니다. 모델이 본 수학 예제 대부분은 다른 숫자 조합이고, "= 724.8"이 따라오는 경우는 거의 없습니다.

따라서 모델은 실제 계산을 하지 않고, 학습 데이터의 통계적 가중치를 기반으로 "722 정도면 비슷하지 않을까?"라고 확률적으로 추측합니다. 이게 바로 LLM의 계산 오류입니다.

CoT(Chain of Thought)가 효과적인 이유: "먼저 245×37=9,065를 계산하고, 다음에 9,065÷12.5=724.8" 같은 단계별 표현은 학습 데이터(교과서, 튜토리얼, Stack Overflow)에 매우 흔한 패턴입니다. 한 번에 푸는 것보다 단계별로 분해하면 각 단계의 확률이 높아져 정답 도달률이 극적으로 상승합니다. (다음 챕터에서 자세히)

정리: LLM 혼자서는 부족합니다
지금까지 본 것처럼, LLM은 "다음 단어 예측"에는 놀라울 정도로 뛰어나지만 —
기억을 못 하고, 정확한 계산도 못 하고, 외부 정보를 직접 찾아보지도 못합니다.

그렇다면 이런 한계를 어떻게 극복할까요?
답은 간단합니다 — LLM에게 "도구"와 "기억"을 달아주면 됩니다. 계산은 계산기에게, 기억은 메모장에게 맡기는 거죠.

그런데, Augmented LLM은 "Agent"일까요?
아직은 아닙니다. Augmented LLM은 도구를 "사용할 수 있는" 상태이지만, 여전히 사람이 시키는 것만 합니다.
진짜 에이전트는 여기서 한 발 더 나아갑니다 — 스스로 판단하고, 계획을 세우고, 필요한 도구를 골라서 실행합니다.
다음 챕터에서 이 차이를 정확히 짚어보겠습니다.
⚠️ 사실 이 경계는 더 회색지대입니다

교과서적 구분은 "Augmented LLM = 사람이 시키는 것만, Agent = 스스로 판단"이지만, 실무에서는 더 미묘합니다.

더 정확한 구분은:
• Augmented LLM = "수동 도구 선택" — 사람이 "calculator를 써"라고 명시
• Agent = "자동 도구 선택" — LLM이 "이건 계산이니까 calculator를 써야겠네"라고 스스로 판단

하지만 양쪽 모두 사람의 최상위 목표(프롬프트) 없이는 작동하지 않습니다. Agent는 "더 나은 자율성"을 갖춘 것이지, 완전 자립적이 아닙니다. "시작 버튼"은 여전히 사람이 누릅니다.

에이전트(Agent)란 센서를 통해 환경을 인지하고, 액추에이터를 통해 그 환경에 작용하는 모든 시스템을 의미한다.
— Russell & Norvig, AI: A Modern Approach (2016)

에이전트는 환경과 상호작용하며, 네 가지 핵심 구성 요소로 이루어져 있습니다.

LLM도 똑같습니다 — "중간 과정을 글로 쓰게 하면" 정확도가 크게 올라갑니다.

프롬프트에 "단계별로 생각해봐 (Let's think step by step)"라는 한 줄만 추가했더니, 수학 문제 정답률이 17% → 78%로 뛰었습니다.
▎ 왜 "단계별로 생각해봐"가 효과적인가?
핵심: LLM이 생성한 중간 단계 텍스트가 다음 예측의 입력(컨텍스트)이 됩니다.
1장에서 배운 "다음 토큰 예측"과 똑같은 원리예요 — 다만 자기가 쓴 풀이를 보면서 다음 단계를 예측합니다.
질문: "사과 한 봉지에 6개씩 들어있어. 3봉지를 사서 친구에게 4개를 줬으면 남은 건?"
1단계
LLM이 읽는 입력:
"사과 한 봉지에 6개, 3봉지 사서 4개 줬으면?"
→
"6×3 = 18개"
↳ "18개"라는 새로운 정보가 다음 입력에 붙음
2단계
LLM이 읽는 입력:
"...3봉지 사서 4개 줬으면?"
+ "① 6×3 = 총 18개" ← 1단계에서 계산한 결과
→
"18-4 = 14개"
답변
"사과 14개가 남았습니다" ✓
핵심: 1단계에서 계산한 "18개"라는 결과가 없었다면, 2단계에서 "18-4"를 할 수 없었겠죠?
이처럼 각 단계의 출력이 다음 단계의 입력이 됩니다. LLM이 마치 연습장에 풀이를 쓰면서 푸는 것과 같은 효과!
💡 그런데 토큰은 어떻게 소비될까?
위 도식을 보면 매 단계마다 입력이 점점 길어지는 게 보이시죠? LLM은 "이어서 쓰기"가 아니라, 매 단계마다 전체를 처음부터 다시 읽습니다.
• 1단계: 질문(20토큰) → 20토큰 소비
• 2단계: 질문(20) + 1단계 출력(10) → 30토큰 소비 (질문이 다시 카운팅됨!)

그래서 추론 단계가 길어질수록 Context Window를 빠르게 소비합니다. 이게 바로 다음에 배울 Memory 챕터의 "Context Window 한계"와 직결되는 부분이에요.

Chain-of-Thought (CoT) — "생각의 사슬"

이 발견에서 나온 기법이 Chain-of-Thought(CoT)입니다. LLM에게 답만 내놓지 말고, 풀이 과정을 단계별로 쓰게 유도하는 방법이에요.

🔬 왜 이게 가능해졌을까?
모델이 충분히 커지면서 "출현(emergence)"한 능력입니다. 작은 LLM에서는 "단계별로 생각해봐"라고 해도 효과가 없었는데, 모델 크기가 임계점을 넘자 갑자기 추론 능력이 나타났습니다.

최근에는 여기서 더 나아가, 답하기 전에 "생각하는 시간"을 더 주는 모델(OpenAI o1, Claude의 extended thinking 등)이 등장했습니다. CoT를 모델 내부에서 자동으로 수행하는 거죠. 이런 모델을 Reasoning Model이라 부릅니다.
🧠 왜 CoT가 효과적인가 — 메커니즘

CoT 효과의 원리를 한 줄로: "어려운 문제를 학습 데이터에 자주 등장하는 쉬운 부분 문제들로 쪼개기"입니다.

모델은 다음 단어 예측만 훈련되었기 때문에, 복잡한 추론을 한 번에 하려면 각 단계의 확률이 빠르게 떨어집니다. P(정답) = P(step1) × P(step2) × ... — 한 단계만 0.3이면 전체가 무너집니다.

하지만 "단계별 설명"은 학습 데이터(교과서, 위키, Stack Overflow, 튜토리얼)에서 매우 흔한 패턴입니다. 각 단계가 학습된 패턴과 매칭되므로 P(step_i) ≈ 0.9 이상으로 유지됩니다. 즉, CoT는 "확률 곱셈에서 각 항을 최대화하는 트릭"입니다.

언제 CoT가 특히 효과적? ① 다단계 산수 ② 논리 추론 ③ 코드 디버깅 ④ 법률·정책 해석. 언제 효과 없음? 단순 정의 질문, 사실 검색.

이러한 추론 능력 덕분에 LLM 에이전트는 복잡한 작업을 실행 가능한 단계들로 분해하여 계획을 수립할 수 있습니다.

LLM을 에이전트에 적용하면?

"Augmented" LLM을 에이전트 구조에 결합하면, LLM은 텍스트 입력을 통해 환경을 관찰(Sensor)하고, 웹 검색 같은 도구를 사용하여 행동(Actuator)할 수 있습니다.

이러한 계획(Planning) 능력을 통해 에이전트는 상황을 이해하고(LLM), 다음 단계를 계획하며(Planning), 도구(Tool)를 사용해 행동을 취하고, 그 결과를 메모리(Memory)에 기록합니다.

Memory (기억)

LLM은 상호작용 중에 어떤 것도 기억하지 않습니다. 질문 하나를 하고, 이어서 다른 질문을 하면, 이전 질문을 전혀 기억하지 못합니다. 에이전트에게 기억력을 부여하는 것이 핵심 과제입니다.

🔗 Part 2 연결: Part 0에서 배운 할루시네이션 문제를 기억하나요? LLM은 "모르는 것도 그럴듯하게 지어냅니다." 이 문제의 근본 원인은 기억 부재(이전 대화 못 기억) + 외부 정보 접근 불가(최신 데이터 없음). Memory · Tools · Planning이 바로 이 한계들을 해결하는 3대 도구입니다.

그렇다면 에이전트에게 어떤 종류의 기억이 필요할까요? 심리학에서는 인간의 기억을 여러 유형으로 분류합니다. 하나의 시나리오로 네 가지 기억을 모두 이해해봅시다.

이러한 기억들을 실제로 구현할 때는 크게 단기 기억(Short-term Memory)과 장기 기억(Long-term Memory) 두 가지로 나눌 수 있습니다. 단기 기억은 "지금 대화에서 무슨 말을 했는지"를 기억하는 것이고, 장기 기억은 "며칠 전, 몇 주 전에 무슨 이야기를 나눴는지"까지 기억하는 것입니다.

Short-term Memory: Context Window 활용

💡 Context Window란? LLM이 1번의 요청에서 읽고 쓸 수 있는 최대 텍스트 크기(입력+출력 합산)입니다. LLM 자체는 "대화"라는 개념이 없어요 — 여러분이 ChatGPT에서 엔터를 누를 때마다, 앱(ChatGPT, Claude 등)이 지금까지의 대화 전체를 하나의 텍스트로 묶어서 LLM에게 매번 새로 보냅니다. 이 묶는 작업은 LLM이 아니라 앱 쪽에서 처리합니다. 대화가 길어지면 이 텍스트가 Context Window 한계를 넘기게 되고, 그때 앞부분을 잘라내야 해서 "잊어버리는" 것처럼 보이는 거예요.

💰 왜 1M이 "표준"이 됐나 — 가격이 같아졌기 때문

2026년 기준, 컨텍스트 윈도우와 가격의 관계가 무너졌습니다.

2024년까지: "큰 컨텍스트 = 비싸다" 등식이 성립했습니다. GPT-4 32K는 비싸고, Claude 100K는 더 비쌌습니다.

2026년 현재: Claude Opus 4.6의 1M 컨텍스트도 표준 가격($15/M output)과 동일하게 처리됩니다. 200K 초과 요청에 별도 베타 헤더도 필요 없습니다. 가격 체계가 "사용한 토큰 수"만 기준이지, "사용 가능한 최대 윈도우"가 아닙니다.

결과: 추가 비용 없이 1M을 쓸 수 있으니, 새 프로젝트 대부분이 처음부터 1M으로 시작합니다. 다만 "1M을 다 채워서 매번 보내면" 토큰당 비용이 누적되어 비쌉니다. 1M은 "필요할 때 다 들어갈 수 있는 그릇"이지, "매번 다 채우는 그릇"이 아닙니다.

Context Window가 점점 커지고 있지만, 프롬프트에 넣는 정보가 많아질수록 LLM이 모든 내용에 골고루 주의를 기울이기 어려워지고, 비용도 증가합니다. 따라서 전체 대화 이력을 매번 그대로 넣는 것은 비효율적이며, 다음 두 가지 기법이 주로 사용됩니다.

기법 ① 대화 이력 트리밍 (Trimming)

💡 그렇다면 ChatGPT와 Claude는 실제로 어떻게 할까?
ChatGPT (OpenAI) — 트리밍 + 요약 조합
대화가 길어지면 오래된 부분을 자동으로 요약해서 시스템 메시지로 압축합니다. 최근 대화는 원문 유지, 오래된 대화는 요약본으로 교체하는 방식입니다.
Claude (Anthropic) — 큰 Context Window + 트리밍
Claude는 한계를 넘으면 오래된 메시지부터 잘라냅니다. 자동 요약 기능은 공식적으로 제공하지 않습니다.
접근법이 다른 이유: ChatGPT는 상대적으로 작은 Context Window(272K, 단 Codex 사용자는 1M까지 설정 가능)를 요약으로 보완하고, Claude는 큰 Context Window(1M)로 대화가 길어지면 트리밍으로 감당하는 전략입니다.

기억의 진화 — Memory Bank

기존 LLM 기반 에이전트는 매번 새로운 대화를 시작하는 것처럼 행동했습니다. 이전 대화 내용을 잊고 같은 질문을 반복하거나, 사용자의 선호를 기억하지 못하는 것이 큰 한계였습니다. 이 문제를 해결하기 위해 등장한 것이 Memory Bank — 대화에서 중요한 정보를 자동으로 저장하고, 다음 대화에 활용하는 관리형 장기 메모리 서비스입니다.

🎯 맞춤형 상호작용
사용자 선호도와 과거 행동을 기억하여 개인화된 대화를 제공합니다.
🔗 세션 연속성
며칠, 몇 주 전의 맥락도 이어서 자연스럽게 대화할 수 있습니다.
🧹 지능형 기억 관리
새로운 정보가 생기면 기존 기억과 자동으로 통합·업데이트합니다.
⚡ 성능 영향 없음
기억이 비동기로 원격 생성되어 에이전트의 응답 속도에 영향을 주지 않습니다.
▎ 에빙하우스 망각 곡선과 Memory Bank의 원리

Memory Bank는 심리학의 에빙하우스 망각 곡선(Ebbinghaus Forgetting Curve) 이론에서 영감을 받았습니다.
자주 회상(retrieval)할수록 기억이 오래 유지되듯, 자주 사용되는 정보는 강화하고 오래 사용되지 않는 정보는 점진적으로 삭제합니다.

Memory Bank는 자주 사용되는 기억은 강화(reinforce)하고, 오랫동안 참조되지 않는 기억은 점진적으로 약화(decay)시킵니다. 마치 시험 준비를 할 때 간격 반복(spaced repetition)으로 학습하면 기억이 오래 유지되는 것과 같은 원리입니다.

▎ Memory Bank는 어떻게 작동하는가? — 저장 · 검색 · 관리
1
저장 (Store)
대화가 끝나면, Memory Bank가 대화에서 핵심 정보를 자동 추출하여 DB에 저장합니다.
대화: "저는 아메리카노를 좋아하고, 팀은 5명이에요"
→ 추출 & 저장: 선호: 아메리카노 / 팀 인원: 5명
2
검색 (Retrieve)
다음 대화가 시작되면, 현재 질문과 관련된 기억을 DB에서 검색해서 Context Window에 주입합니다.
새 대화: "커피 좀 시켜줘"
→ DB 검색: "아메리카노 좋아함", "팀 5명" 을 찾아서 프롬프트에 포함
3
관리 (Manage)
자주 참조되는 기억은 강화하고, 오래 안 쓰인 기억은 점진적으로 약화·삭제합니다. 새 정보가 들어오면 기존 기억과 자동 통합합니다.
강화: "아메리카노 좋아함" (3번 참조됨 ↑)
업데이트: "팀 5명" → "팀 6명" (새 정보 반영)
약화: "작년 프로젝트 일정" (6개월간 미참조)
이 사이클이 대화할 때마다 반복됩니다
대화 → ① 저장 → 다음 대화 → ② 검색 & 주입 → ③ 관리 → 🔄
💡 실제 예시: ChatGPT의 "Memory" 기능이 바로 이 Memory Bank의 단순 버전입니다. "이 사용자는 Python을 선호합니다"처럼 핵심 사실을 저장해뒀다가, 다음 대화에서 자동으로 활용합니다. 설정에서 저장된 기억을 확인하고 삭제할 수도 있어요.

지식 출처 외부 문서·데이터베이스에서 범용 지식을 가져옴 사용자와의 대화에서 얻은 개인 경험·선호를 축적
활용 목적 모델이 모르는 내용에 대해 근거 있는 답변 생성 사용자별 지속적 맥락 유지 및 개인화된 경험 제공
구현 방식 개발자가 직접 데이터 파이프라인 구축 핵심 정보 추출·저장·업데이트를 자동 처리
비유 📚 백과사전을 찾아보는 것 🧠 친한 친구가 나를 기억하는 것
💡 RAG와 Memory Bank는 서로를 대체하는 것이 아니라 보완하는 관계입니다. RAG가 "지금 모르는 것을 찾아오는" 역할이라면, Memory Bank는 "예전에 나눈 이야기를 기억하는" 역할입니다. 실무에서는 필요에 따라 두 기술을 조합하여 사용합니다.

Function Calling의 한계 — 'M × N' 통합 문제

Function Calling은 강력하지만, 현실에서는 큰 문제가 있습니다. AI 모델이 여러 개이고 연결할 도구도 여러 개라면, 모델마다 도구마다 각각 별도의 연결을 만들어야 합니다. 이것이 바로 'M × N' 통합 문제입니다.

😰 Function Calling 규격이 앱마다 제각각 (OpenAI, Anthropic, Google 방식)
😰 보안·인증 방식이 통일되지 않아 매번 새로 구현

Model Context Protocol (MCP)

Anthropic이 개발한 MCP는 다양한 서비스의 API 접근을 표준화하여, 어떤 에이전트 프레임워크에서도 도구를 쉽게 사용할 수 있게 해줍니다.

💡 앞서 배운 식당 비유로 다시 보면: Function Calling에서는 AI가 웨이터(API)에게 직접 주문서를 건넸습니다. 그런데 식당이 10곳이라면? 각 식당마다 주문서 양식이 다릅니다. MCP는 모든 식당이 쓰는 통일된 주문서 양식을 만든 것입니다.

MCP는 AI에게 외부 도구를 쓸 수 있는 권한을 줍니다. 마치 신입사원에게 첫날부터 모든 시스템 권한을 주지 않는 것처럼, AI에게도 신중하게 권한을 설정해야 합니다.

②

③
데이터 노출 주의 — 연결된 도구를 통해 개인정보나 내부 데이터가 의도치 않게 노출될 수 있음
💰 비용 — "도구를 많이 연결할수록 비용이 급증"

📋 실제 MCP 도구 사용설명서 예시 — 날씨 도구
{
  "name": "get_weather",
  "description": "도시의 현재 날씨를 조회합니다",
  "inputSchema": {
    "type": "object",
    "properties": {
      "location": { "type": "string", "description": "도시 이름" },
      "unit": { "type": "string", "enum": ["celsius", "fahrenheit"] }
    },
    "required": ["location"]
  }
}
쉽게 읽으면:
• name — 도구 이름 (get_weather)
• description — 이 도구가 뭘 하는지 설명
• inputSchema — 어떤 입력값이 필요한지 (location은 필수, unit은 선택)
이런 설명서가 도구마다 하나씩 있고,
AI에게 질문할 때 연결된 모든 도구의 설명서가 전부 Context Window에 들어갑니다.
가 있는데, AI에게 질문할 때마다 연결된 모든 도구의 설명서가 함께 전달됩니다.

에이전트가 어떤 도구를 언제 사용할지 결정하려면, 계획 수립 능력이 필수입니다. Planning이란 주어진 작업을 실행 가능한 단계들로 분할하는 과정입니다.

▎ 작업 분해 (Task Decomposition)

조사해줘."
→
Output (계획)
수행할 작업:

☐ 학술 DB에서 관련 논문 탐색
☐ 결과 요약 및 핵심 정리

Reasoning: Chain-of-Thought

계획을 세우려면 추론(Reasoning) 능력이 필요합니다. Chain-of-Thought는 LLM이 단계별로 사고하도록 유도하는 기법입니다.

일반 LLM
바로 답변
"도서관에 책 15권이 있다.
5권을 빌려주고 3권을 새로 받았다.
몇 권이 남았나?"
↓
10권입니다. ❌ 오답
Reasoning LLM
단계별 사고 후 답변
같은 문제 + "단계별로 생각해봐"
↓
처음: 15권
5권 빌려줌 → 10권 남음
3권 추가 → 13권
13권입니다. ✅ 정답

ReAct: Reasoning + Acting

ReAct는 추론(Thought)과 행동(Action), 관찰(Observation)을 반복하여 복잡한 작업을 수행하는 기법입니다.

🔗 용어 연결: ReAct 패턴은 파트마다 다른 이름으로 등장합니다.
• Part 5의 "3단계 작업 흐름" (지시 → 결과 확인 → 재지시)
• Part 6 Ch.02의 "에이전트 루프" (센서 → 판단 → 행동)
• Part 6 Ch.05의 "Plan-Critic-Build" (계획 → 비평 → 구현)
모두 같은 "ReAct" 사이클의 변형입니다. 이름만 다를 뿐, 핵심 원리는 동일.
▎ ReAct 사이클 예시: "최신 갤럭시 S 시리즈의 가격은?"
Thought
최신 가격 정보가 필요하니, 웹 검색을 해야겠다.
Action
search("갤럭시 S26 가격 2026")
Observation
검색 결과: 갤럭시 S26 출고가 1,199,000원…
Thought
가격 정보를 찾았다. Ultra 모델 가격도 확인하자.
Action
search("갤럭시 S26 Ultra 가격")
Observation
S26 Ultra 출고가 1,698,400원…
Thought
충분한 정보를 모았다. 최종 답변을 구성하자.
Answer
갤럭시 S26: 1,199,000원 / S26 Ultra: 1,698,400원

Reflexion: 실패로부터 학습하기

ReAct로도 완벽하지 않습니다. Reflexion은 에이전트가 이전의 실수를 돌아보고 개선하는 기법입니다.

▎ Reflexion 프레임워크
Actor
행동 선택·실행
(ReAct 기반)
Evaluator
출력물 점수화
(성공/실패 판단)
Self-Reflection
행동+점수 돌아보기
(개선 방향 도출)
Actor — 실행자
LLM이 ReAct 방식으로 "생각 → 행동"을 반복하며 실제 작업을 수행합니다.
예: "날씨 검색해야겠다" → API 호출 → 결과 받아오기
Evaluator — 채점관
Actor의 결과물을 실제로 테스트하거나, LLM에게 "이 결과 맞아?" 하고 점수를 매기게 시킵니다.
예: 코드 실행 → 테스트 통과 ✓ / 실패 ✗ 판정
Self-Reflection — 반성
실패 시, LLM에게 "행동 기록과 실패 원인을 분석해봐"라고 시켜서 교훈을 도출하고 메모리에 저장합니다.
예: "변수명 오타 → 다음엔 변수명부터 확인" (장기 메모리 저장)
Actor가 행동 → Evaluator가 평가 → Self-Reflection이 반성 → 개선된 다음 행동
단기 메모리(행동 기록) + 장기 메모리(반성 기록)를 활용하여 에이전트가 실수로부터 학습
💡 SELF-REFINE — 스스로 고치는 AI
유사한 기법으로 SELF-REFINE이 있습니다. 하나의 LLM이 세 가지 역할을 번갈아 수행합니다:

① 초안 작성
→
② 스스로 피드백
→
③ 개선된 출력
↺ 반복
예시: "보고서 초안을 작성해줘" → AI가 초안 작성 → 스스로 "논리가 약한 부분이 있다"고 판단 → 해당 부분을 보강하여 재작성. 마치 본인이 쓴 글을 본인이 퇴고하는 것과 같습니다.

Reflexion이 외부 환경의 피드백을 활용하는 반면, SELF-REFINE은 외부 도구 없이 자체적으로 품질을 높이는 점이 다릅니다.

그런데, 서로 다른 에이전트끼리는 어떻게 대화할까? — A2A

위의 멀티 에이전트 패턴은 같은 시스템 안에서의 협업입니다. 하지만 현실에서는 다른 회사, 다른 플랫폼, 다른 기술로 만들어진 에이전트들이 서로 협력해야 하는 경우가 생깁니다. 이때 필요한 것이 A2A (Agent-to-Agent) 프로토콜입니다.

▎ A2A — Agent to Agent Protocol
에이전트 A

에이전트 B
◄─ A2A ─►
에이전트 C
에이전트끼리 "협업/대화"하는 방법을 표준화 — Google이 2025년 4월 Cloud Next에서 발표
💡 A2A의 핵심 가치: 단순히 메시지를 주고받는 것을 넘어, 상대 에이전트의 존재를 인지하고, 능력을 파악하며, 작업을 요청하고 결과를 교환하는 전 과정을 표준화합니다. 서로 다른 기술/플랫폼/개발사에 의해 만들어진 에이전트들이 표준화된 '언어'를 배워 대화하고 협력할 수 있도록 지원합니다.

▎ Agent Directory — 에이전트를 어떻게 찾을까?

에이전트가 수천 개로 늘어나면 "어떤 에이전트가 있는지" 일일이 알 수 없겠죠? 이를 위해 Agent Directory(에이전트 디렉토리) — 에이전트 카드를 모아둔 검색 서비스를 구축할 수 있습니다. 앱스토어에서 앱을 검색하듯, 필요한 에이전트를 검색하고 연결하는 겁니다.

📋 Agent Directory / Registry
등록된 에이전트 카드를 검색할 수 있는 중앙 카탈로그
✈️ 항공권 에이전트 🏨 호텔 에이전트 🌐 번역 에이전트 🎨 이미지 생성 에이전트 ... 수천 개
클라이언트 → Directory "항공 예약 가능한 에이전트 찾아줘"
Directory → 클라이언트 "항공권 에이전트 3개 찾았어" + URL 전달
→ 이후 해당 URL로 직접 A2A 연결하여 작업 요청
실제 사례: Google Cloud에서 AI Agent Marketplace를 운영하며, 파트너사들이 자신의 A2A 에이전트를 등록하고, 고객이 필요한 에이전트를 검색·연결할 수 있습니다. 앱스토어의 AI 에이전트 버전이라고 보면 됩니다.
