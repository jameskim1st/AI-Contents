01

운영체제와 하드웨어

컴퓨터를 켜면 가장 먼저 실행되는 프로그램, 그리고 그 안에서 일하는 핵심 부품들을 이해합니다.

🏢 비유로 먼저 잡기
컴퓨터를 공장이라고 생각해 보세요.
OS(운영체제) = 공장 관제탑 — 누가 언제 무슨 작업을 할지 총괄 지휘
CPU = 작업자 — 실제 계산을 수행하는 핵심 인력
메모리(RAM) = 작업대 — 지금 작업 중인 재료를 펼쳐 놓는 공간
디스크(SSD/HDD) = 창고 — 재료를 영구 보관하는 곳

이 네 가지만 이해하면, "컴퓨터가 왜 느려지는지", "메모리를 왜 늘려야 하는지" 같은 질문에 명쾌하게 답할 수 있습니다.

운영체제(OS) — 프로그램 실행의 메인 프로그램

운영체제(Operating System)는 하드웨어 자원을 관리하고 응용 프로그램(카톡, 브라우저, Excel 등)을 실행하는 메인 프로그램입니다. 우리가 컴퓨터를 사용하는 데 필요한 거의 모든 것 — 프로그램 실행, 네트워크 통신, 키보드·마우스 입력 — 을 OS가 총괄합니다.

OS의 3대 역할

프로그램 실행 관리
앱을 메모리에 올리고
CPU 시간을 배분
🔧
하드웨어 자원 관리
CPU, 메모리, 디스크를
프로그램에 할당·회수
🌐
네트워크·입출력 관리
네트워크 통신, 키보드
마우스·모니터 제어
💡 커널(Kernel) — OS의 핵심
커널은 OS의 핵심 구성요소로, 프로그램들이 하드웨어 자원을 사용할 때 이를 중재합니다. OS를 만들 때 커널을 기본으로 하고, 그 위에 셸(Shell), 파일 시스템, 드라이버 등을 올립니다. CPU·메모리 할당 조율, 인터럽트 처리, 프로세스 스케줄링 모두 커널의 일입니다.

하드웨어 핵심 4요소

CPU
🧠
Central Processing Unit
컴퓨터의 모든 계산을 수행하는 핵심 칩. 프로그램이 실행된다 = CPU가 명령어를 하나씩 읽어 계산하고 있다는 뜻입니다.

클럭 속도: 1초에 처리하는 박자 수 (3.5GHz = 초당 35억 박자)
코어: 독립된 처리 장치 수 (8코어 = 동시에 8가지 작업)

예: Intel Core i7, AMD Ryzen 9, Apple M4
메모리(RAM)
📋
Random Access Memory
전기적 신호를 임시로 저장하는 휘발성 공간. 프로그램이 실행되면 메모리 위에 올라가고, 종료되면 사라집니다.

8GB — 웹 브라우징, 문서 작업
16GB — 프로그래밍 최소 권장
32GB+ — 데이터 분석, 로컬 LLM

전원 끄면 데이터 소멸 (휘발성)
디스크(SSD/HDD)
💾
영구 저장소
정보를 영구적으로 저장합니다. 메모리가 프로그램 실행 과정을 책임진다면, 디스크는 프로그램을 컴퓨터에 저장하는 과정을 책임집니다.

전원 꺼도 데이터 유지 (비휘발성)
GPU
🎨
Graphics Processing Unit
대량 병렬 연산에 특화된 칩. 원래는 그래픽용이었지만, AI 시대에는 딥러닝 학습·추론의 핵심이 되었습니다.

CPU: 수학 교수 1명이 어려운 문제 풀기
GPU: 초등학생 1만 명이 구구단 동시에 풀기

예: NVIDIA H100, RTX 4090

02

프로그램의 세계

개발자가 짠 코드가 어떻게 실행 가능한 프로그램이 되고, 메모리 위의 프로세스가 되는지 — 소프트웨어의 생명주기를 따라갑니다.

코드 → 프로그램 → 프로세스 — 소프트웨어의 생명주기

프로그램은 코드(명령어)의 집합이 컴퓨터가 이해할 수 있는 형태로 컴파일된 것이고, 프로세스는 그 프로그램이 메모리 위에서 실제로 실행 중인 상태입니다. 카톡, 브라우저, Excel을 동시에 켜면, 각각이 하나의 프로세스가 됩니다. 작업관리자(Task Manager)에서 프로세스 목록을 확인하고 강제 종료할 수 있습니다.

프로그래밍 언어 — 고급 vs 저급

고급 언어
사람이 읽기 쉬운 언어
Python, Java, JavaScript, C++ 등

print("Hello World")
영어와 비슷한 문법
저급 언어
CPU가 직접 이해하는 언어
기계어(0과 1), 어셈블리어

01001000 01100101
사람이 읽기 어려움

CPU는 0과 1(기계어)만 읽을 수 있으므로, 고급언어를 저급언어로 변환하는 과정이 반드시 필요합니다. 이 변환을 담당하는 시스템 프로그램이 컴파일러(Compiler)입니다.

컴파일러와 바이트코드 — 변환 방식의 차이
🔨 전통적 컴파일
고급 언어 → 기계어
CPU가 직접 해석·실행

예: C, C++, Rust
☕ 바이트코드 방식
고급 언어 → 바이트코드 → 가상 머신이 해석
"한 번 컴파일하면 어디서든 실행"

예: Java(JVM), Python(CPython)
빌드 (Build)
소스코드를 실행 가능한 산출물로 만드는 전체 과정입니다. 컴파일은 빌드 과정의 일부입니다. 우리가 다운받아 클릭으로 바로 실행하는 프로그램은 모두 빌드를 거친 결과물입니다.

바이브코딩에서 npm run build 명령어가 바로 이 빌드 과정입니다.
IDE
Integrated Development Environment — 코딩, 디버깅, 컴파일, 실행을 하나의 프로그램 안에서 할 수 있게 해주는 통합 개발 환경입니다.

VS Code — 가장 보편적, 확장 기능 풍부
Cursor — VS Code 기반 + AI 코딩 내장
IntelliJ — Java/Kotlin 전문 (JetBrains)
🔬 심화 — 프로세스의 메모리 구조와 인터럽트
프로세스가 생성되면 메모리 안에 네 가지 영역이 할당됩니다:
Stack — 함수 호출 정보, 지역 변수 저장  |  Heap — 동적으로 할당되는 데이터
Code — 프로그램의 명령어 자체  |  Data — 전역 변수, 상수

인터럽트(Interrupt): 여러 프로그램이 동시에 실행 중일 때 키보드 입력이나 마우스 클릭 같은 갑작스러운 신호가 발생하면, OS가 현재 CPU 작업을 잠시 멈추고 해당 신호를 먼저 처리합니다. 처리 속도가 매우 빨라서 프로그램이 멈추는 느낌을 받지 않습니다.
03

네트워크

카카오톡 메시지가 친구에게 도착하는 이유, 브라우저에 주소를 치면 페이지가 뜨는 이유 — 모두 네트워크 덕분입니다.

📮 한 문장 정의
네트워크 = 컴퓨터와 컴퓨터 간의 연결. 우리가 사용하는 거의 모든 IT 서비스(카톡, 네이버, 넷플릭스)는 네트워크를 통해 동작합니다.

네트워크 통신의 3대 요소

"어떤 방식(프로토콜)으로, 어디 컴퓨터(IP)에, 어떤 프로그램(PORT)을 연결할 것인가?" — 이 세 가지가 네트워크 통신의 핵심입니다.

프로토콜 + IP + PORT = 네트워크 통신
📜
프로토콜
Protocol
"어떤 방식으로?"
HTTP, HTTPS, SSH, SMTP
📍
IP 주소
Internet Protocol Address
"어느 컴퓨터에?"
192.168.0.1
🚪
PORT
프로세스 주소
"어떤 프로그램에?"
80, 443, 3000

주요 프로토콜

프로토콜(Protocol)은 컴퓨터 간 네트워크 통신을 위한 공통 규격(규칙)입니다. 전 세계적으로 네트워크 통신이 가능한 이유가 바로 이 공통 규격 덕분입니다.

프로토콜 종류와 기본 PORT
프로토콜 용도 기본 PORT
HTTP 웹 정보 송수신 80
HTTPS 보안 웹 통신 (암호화) 443
SSH 원격 컴퓨터 접속 22
SMTP 이메일 전송 25 / 587
FTP 파일 전송 21
💡 HTTP vs HTTPS
HTTP는 웹에서 가장 많이 쓰이는 프로토콜이지만, 보안에 취약합니다. HTTPS는 HTTP에 보안(S = Secure)을 더한 프로토콜로, 통신 데이터를 암호화합니다. 브라우저에서 google.com을 입력하면 자동으로 https://google.com이 적용됩니다. 대부분의 웹사이트가 HTTPS를 표준으로 사용합니다.

IP 주소 — 컴퓨터의 네트워크 주소

IP는 컴퓨터의 고유 주소가 아니라, 연결된 네트워크(와이파이, 랜선 등)를 기준으로 부여받는 네트워크 주소입니다.

IPv4
전통적인 IP 형식
192.168.0.1
주소 개수: 약 43억 개 (255^4)
현재 거의 포화 상태
IPv6
주소 부족 문제 해결
2001:0db8:85a3::8a2e
주소 개수: 사실상 무한
스마트폰 유심 IP가 대표적

PORT — 컴퓨터 안의 프로세스 주소

IP가 "어떤 컴퓨터인지"를 가리킨다면, PORT는 "그 컴퓨터 안의 어떤 프로그램인지"를 가리킵니다. 하나의 컴퓨터에 여러 서버가 돌아갈 수 있는 이유가 바로 각 서버가 서로 다른 PORT를 사용하기 때문입니다.

도메인과 DNS — 사람이 읽는 주소

도메인 → DNS → IP 변환 과정

도메인 네임(Domain Name)은 IP 주소를 사람이 읽기 쉽게 바꾼 영문 주소입니다. naver.com을 입력하면 DNS(Domain Name System)가 이를 IP 주소(223.130.195.200)로 변환해 줍니다. 서버에 도메인을 붙이려면 호스팅 서비스(가비아, Cloudflare 등)에서 도메인을 구매하고 DNS에 등록해야 합니다.

🛡️ 방화벽 (Firewall)
IP + PORT 기반 네트워크 보안
네트워크 보안을 위해 컴퓨터 간 연결을 제한하는 시스템입니다. IP 주소와 PORT 번호를 기반으로 허용/차단 규칙을 설정합니다. 서버 운영 시 필요한 PORT만 열어두는 것이 기본 보안 원칙입니다.
🔍 "네이버에 접속한다" — 실제로 무슨 일이 일어나는가?

🔬 심화 — OSI 7계층
컴퓨터 OS가 네트워크 신호를 특정 프로그램으로 전달하는 과정은 7개의 계층(Layer)으로 나뉩니다. 이를 OSI 7계층이라 하며, 각 계층마다 역할이 정해져 있습니다. 네트워크 전공 도서에서 필수적으로 다루는 개념이지만, 비전공자는 "네트워크 통신에도 여러 단계가 있구나" 정도로 이해하면 충분합니다.
📡 네트워크의 물리적 수단
광케이블 — 0과 1의 전기 신호를 빛의 물리적 신호로 변환하여 전송 (랜선)
와이파이 — 전자기파를 이용한 무선 통신

🍳 요리 비유

라이브러리 (Library)
🧂

핵심: 내 코드가 라이브러리를 호출한다
(내가 주도)

📐

API

프로그램과 프로그램이 대화하는 규칙 — API(Application Programming Interface)를 이해하면, AI 시대의 모든 연결이 보입니다.

🤵 한 문장 정의
API = 프로그램과 프로그램을 연결하는 중개자 (규칙들의 집합)
손님(클라이언트)이 주방(서버)에 직접 들어가지 않듯, 프론트엔드도 백엔드 서버에 직접 접근하지 않고 반드시 API(종업원)를 통해 데이터를 요청하고 받습니다.
🍽️ API를 식당으로 이해하기

API 문서 — 약속의 명세서

API를 사용하려면 API 문서를 읽어야 합니다. 문서에는 세 가지가 약속되어 있습니다:

API 문서의 3대 요소
🔗
URL
어디로 요청할 것인가
api.weather.com/v1/current
📨
Parameter
무엇을 보낼 것인가
city=Seoul, api_key=sk-...
📬
Response
무슨 응답이 오는가
{ "온도": 23, "날씨": "맑음" }

REST API — 가장 널리 쓰이는 API 설계 스타일

HTTP 메서드와 역할 (CRUD)
메서드 역할 식당 비유
GET 데이터 조회 "메뉴 보여주세요"
POST 데이터 생성 "주문할게요"
PUT 데이터 수정 "주문 변경이요"
DELETE 데이터 삭제 "주문 취소요"
📡 실제 API 요청/응답 예시 — 날씨 앱
1. 요청 (Request) — 앱 → API 서버
날씨 앱이 기상청 API에 "서울 날씨 알려줘"라고 요청
GET https://api.weather.com/v1/current

  city = "Seoul"
  api_key = "sk-abc123..."
GET = "데이터를 가져와줘"
api_key = 내 신분증 (API 키)
2. 응답 (Response) — API 서버 → 앱
API 서버가 JSON 형식으로 날씨 데이터를 돌려줌
{
  "도시": "서울",
  "온도": 23,
  "날씨": "맑음",
  "습도": 45
}
앱은 이 JSON 데이터를 받아서
화면에 "서울 23° 맑음"으로 표시
🔑 API 키 — 인증과 보안
대부분의 API는 사용자를 확인하기 위해 API 키(Key)를 요구합니다. API 키는 일종의 신분증으로, 누가 얼마나 API를 사용하는지 추적하고 악용을 방지합니다. API 키는 절대 코드에 직접 넣지 않고, .env 파일에 안전하게 보관해야 합니다.

Git & GitHub

코드를 수정하다 "3시간 전으로 돌아가고 싶다" — 그 소원을 들어주는 코드의 타임머신을 소개합니다.

🎮 게임 비유
PPT에 Ctrl+Z가 있다면, 코드에는 Git이 있습니다.
SharePoint의 "버전 기록"과 비슷하지만, 자동 저장이 아니라 내가 직접 의미 있는 시점을 골라서 저장(commit)한다는 차이가 있습니다.

Git 핵심 명령어

Git 명령어 한눈에
명령어 역할 쉬운 비유
commit 현재 시점 저장 (메시지 포함) 🎮 게임 세이브
push 커밋을 원격 저장소에 업로드 ☁️ 클라우드에 백업
pull 원격의 최신 코드 가져오기 📥 최신 버전 다운로드
fetch 원격 변경사항 확인 (병합 전) 🔍 업데이트 확인
fork 남의 프로젝트를 내 버전으로 복사 📋 남의 세이브 파일 복사
clone 원격 저장소를 내 PC에 복제 💻 프로젝트 다운로드
🔄 Git 워크플로우 — commit → push → fetch/pull
⚡ 충돌 (Conflict)
같은 파일, 같은 라인을 동시에 수정하면 발생
여러 개발자가 같은 파일의 같은 코드 라인을 동시에 수정하면 코드 충돌이 발생합니다. Git은 충돌 지점을 명확히 표시해 주며, 개발자가 어떤 코드를 살릴지 선택하여 해결합니다. 바이브코딩에서도 AI가 만든 코드와 내가 직접 수정한 코드가 충돌할 수 있으니 알아두면 유용합니다.

브랜치 (Branch) — 안전한 작업 공간

🌿 브랜치 시각화 — 메인에서 분기 → 작업 → 머지

Git에는 항상 메인(main) 브랜치가 존재합니다. 개발자들은 메인 브랜치를 복사해서 새 브랜치를 만들고, 그 안에서 커밋을 쌓으며 작업한 뒤, 완료되면 메인 브랜치로 머지(병합)합니다. IT 회사에서는 git-flow, trunk-based 등의 브랜치 전략을 사용합니다.

GitHub — 세계 최대 개발자 플랫폼

GitHub는 Git이 적용된 원격 코드 저장소이자, 전 세계에서 가장 많은 개발자가 사용하는 서비스입니다. 여러 개발자가 커밋을 공유하려면 원격 저장소가 필요하며, 그 역할을 GitHub가 합니다.

컨테이너와 가상화 — 서버를 나누는 기술

🏢 건물주 비유로 먼저 잡기

여러분이 1층짜리 건물주가 됐다고 상상해보세요. 건물 전체를 임차인 한 명에게만 빌려주고 있습니다.

🏗️ 건물 = 서버(물리 서버)

VM vs 컨테이너 — 핵심 차이

🔍 VM(가상 머신)과 컨테이너, 뭐가 다를까?
🖥️
VM (가상 머신)
✦ 각각 OS를 포함 (Windows, Linux 등)
✦ 무겁다 — 수 GB 크기
✦ 부팅 느리다 — 수 분
✦ 더 강한 격리 (OS 수준 분리)
✦ 자원 많이 사용
📦
컨테이너
✦ 호스트 OS 공유 (커널 공유)
✦ 가볍다 — 수십 MB
✦ 시작 빠르다 — 수 초
✦ 약간 약한 격리 (프로세스 수준)
✦ 자원 효율적

컨테이너는 Guest OS가 없어서 훨씬 가볍고 빠릅니다. 대신 호스트 OS의 커널을 공유합니다.

Docker = 컨테이너를 관리하는 도구

🐳 Docker가 해주는 세 가지

(Dockerfile → Image)
▶️
2. 컨테이너 실행
"이 컨테이너를

(docker run)
🚢
3. 컨테이너 배포
"이 컨테이너를

Kubernetes = 여러 서버를 총괄하는 매니저

건물 1채 = Docker로 충분 / 건물 3채 이상 = 매니저(Kubernetes) 필요

Kubernetes가 해주는 일 3가지
🎯
1. 자동 배치
어떤 서버에 컨테이너를 놓을지
자동으로 결정
(리소스 상황 고려)
🔄
2. 자동 복구
컨테이너가 죽으면
자동 재시작
(서비스 중단 최소화)
📈
3. 자동 확장
트래픽 많으면 컨테이너 수
자동 증가
(Auto Scaling)
🎯 핵심 정리 3줄

1. 컨테이너 = 한 대 서버에서 여러 소프트웨어를 안전하고 효율적으로 운영하는 기술
2. Docker = 컨테이너를 만들고 실행하는 도구 (프로그램)
3. Kubernetes = 여러 서버의 Docker 컨테이너를 총괄하는 오케스트레이션 도구
