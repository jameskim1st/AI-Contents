01

데이터란 무엇인가 — DIKW 피라미드와 AI 시대

"데이터"는 AI 시대에 가장 자주 쓰이지만 가장 오해받는 단어입니다. 1989년 Russell Ackoff가 제안한 DIKW 피라미드는 37년이 지난 지금도 여전히 유효합니다 — 아니, LLM 시대에 오히려 더 중요해졌습니다.

DIKW 피라미드 — 데이터가 지혜가 되기까지
Wisdom (지혜)
지식을 바탕으로 올바른 행동·판단을 내릴 줄 아는 능력
Knowledge (지식)
정보들을 엮어 발견한 패턴·규칙·인사이트 — "이럴 땐 저래진다"
Information (정보)
데이터에 맥락·의미를 부여한 것 — 누가·언제·무엇을·얼마에
Data (데이터)
가공되지 않은 날것의 사실·측정값 — 아직 의미는 없는 숫자/문자
🍜 예시 (편의점 매출):
Data → "2026-04-11 14:32, 상품#A103, ₩1,800"
Information → "오늘 14시 대에 A103(컵라면) 한 개가 ₩1,800에 팔렸다"
Knowledge → "점심 직후 컵라면 수요가 매일 30% ↑"
Wisdom → "12시~15시에 컵라면 진열을 2배로 늘리자 (ROI 검증 후)"
AI 시대에 DIKW가 여전히 유효한 이유
① LLM은 Information 층에 산다
LLM이 생성하는 것은 대부분 정보(조합·요약·변환)입니다. Wisdom(의사결정)은 여전히 사람의 몫 — 적어도 2026년까지는.
② Data 층이 모든 것을 결정한다
피라미드의 바닥이 흔들리면 위의 Knowledge/Wisdom은 모두 환각입니다. Garbage in, garbage out은 1957년부터 지금까지 불변.
③ AI는 각 층의 속도를 바꾼다
Data → Information은 과거 수일·수주가 걸렸지만, LLM으로 수초가 되었습니다. 하지만 층 자체는 사라지지 않았습니다.
④ 품질은 층마다 전파된다
데이터 1%의 오류가 Knowledge 층에서는 10%, Wisdom 층에서는 100% 잘못된 의사결정이 됩니다. 하위 층일수록 품질 투자 레버리지 ↑.
"Garbage in, garbage out" — AI 시대의 진짜 의미
"LLM은 마법이 아닙니다. 훈련 데이터의 평균 품질을 따라갑니다. 훈련 데이터가 엉망이면 모델은 엉망이고, 프롬프트 데이터가 엉망이면 답이 엉망입니다."
— Andrew Ng, Data-Centric AI 캠페인 (2021~)

04

데이터 인터페이스 투어 — SQL부터 노트북까지

데이터는 "저장" 다음에 "꺼내기"가 필요합니다. 2026년 현재 데이터를 만지는 도구는 5가지 계층으로 나뉩니다 — SQL, NoSQL 쿼리, Vector 쿼리, 노트북, BI. 각각이 무슨 역할인지, 누가 쓰는지를 빠르게 투어합니다.

① SQL — 50년째 만국 공용어
1974년 IBM에서 탄생한 SQL은 여전히 데이터 세계의 lingua franca입니다. 2026년에도 데이터 엔지니어·분석가의 80%가 가장 많이 쓰는 언어 — Stack Overflow Developer Survey 기준. LLM조차 결국 SQL을 생성합니다.
-- "4월 가장 많이 팔린 상품 TOP 5"
SELECT product_name, SUM(quantity) AS sold
FROM orders o
JOIN products p ON o.product_id = p.id
WHERE o.order_date BETWEEN '2026-04-01' AND '2026-04-30'
GROUP BY product_name
ORDER BY sold DESC
LIMIT 5;

LLM Pre-training과 데이터 — LLM은 데이터로 만들어진다

"LLM이 어떻게 만들어지느냐"의 답은 놀랍도록 단순합니다 — 엄청난 양의 텍스트로 다음 단어를 맞추는 학습. 알고리즘보다 중요한 것은 데이터입니다. 2026년에도 "누가 더 좋은 데이터를 가지고 있느냐"가 LLM 경쟁력의 핵심입니다.

주요 Pre-training 데이터 출처
🕷️
Common Crawl
전 세계 웹 페이지를 자동 수집한 공개 아카이브. ~250B 웹페이지, ~수 PB. 거의 모든 LLM의 기본 훈련 데이터. 노이즈가 많아 필터링이 관건.
📚
Wikipedia
약 60M 문서. 고품질·백과사전식 문체. 거의 모든 LLM이 "gold data"로 과잉 가중치(up-sampling)를 줍니다.
📖
Books (Books3 등)
약 200k~수백만 권의 책. 긴 맥락과 서사 구조의 원천. 저작권 소송의 중심(Ch.05 하단).
💻
Code (GitHub)
수십 TB의 공개 코드. 흥미로운 발견: 코드를 많이 학습시키면 일반 추론도 향상. 구조적 사고의 효과.
📄
arXiv / PubMed
학술 논문. 수학·과학·의학의 전문 지식. LaTeX 수식까지 포함되어 모델의 전문성을 올림.
🤖
Synthetic Data ⭐
LLM이 LLM을 위해 생성한 데이터. 2025~2026년 폭증. Phi-3, Llama 3의 핵심. "고품질 → 적은 양으로 충분".

Tokenization — 텍스트를 숫자로 쪼개기
LLM은 문자가 아니라 토큰(단어 조각)으로 텍스트를 다룹니다. 대표 알고리즘: BPE (Byte-Pair Encoding), WordPiece, SentencePiece. GPT는 cl100k_base/o200k_base.
# 영어
"The cat sat on the mat" → ["The", " cat", " sat", " on", " the", " mat"]
→ 6 tokens (단어 = 토큰 거의 1:1)

# 한국어
"고양이가 매트에 앉았다" → ["고", "양", "이", "가", ...]
→ 16 tokens (음절/바이트 단위로 쪼개짐)
⚠️ 한국어 토큰 비효율: 영어 대비 2~3배의 토큰이 필요합니다. 같은 API 호출에 2~3배의 비용이 드는 셈. 2024년 GPT-4o의 o200k_base가 한국어 토큰을 크게 개선 (1.6x → 1.2x).
데이터 오염과 저작권 — 2025~2026 뜨거운 이슈
🦠 Data Contamination
훈련 데이터에 벤치마크 답이 섞여 있는 문제. "MMLU 90점"이 진짜 실력인지 답을 외운 건지 알 수 없음. 2026년에는 contamination-free eval이 필수.
⚖️ 저작권 소송
NYT vs OpenAI(2023~), Authors Guild vs OpenAI, Getty Images vs Stability AI 등. 2026년에도 결판이 나지 않았고, 업계는 "licensed data"로 이동 중 (Reddit, FT, AP 딜).
💡 더 깊이: pre-training-data, tokenization wiki 엔티티에서 RedPajama, SlimPajama, FineWeb 등 오픈 데이터셋과 BPE 내부 동작을 자세히 다룹니다.

Fine-tuning 데이터 — 모델을 내 문제에 맞추기

Pre-training이 "일반 지능"을 만든다면, Fine-tuning은 "내 도메인·내 말투·내 안전성"에 맞춥니다. 2026년 현재 3가지 방법이 표준이며, 놀랍게도 양보다 품질이 결정적입니다.

Fine-tuning 3가지 방법 — SFT · RLHF · DPO
📝
SFT
Supervised Fine-Tuning
"instruction → response" 쌍을 라벨 데이터로 삼아 학습. 가장 단순·기본적. 대부분의 fine-tune이 여기서 시작.
{
 "instruction": "이메일 요약",
 "input": "...",
 "output": "3줄 요약"
}
👥
RLHF
RL from Human Feedback
"A vs B 중 어느 답이 더 나은가"를 사람이 비교 라벨링. Reward 모델을 훈련하고 RL로 정책 개선. ChatGPT를 만든 바로 그 방법.
prompt: "A 설명해줘"
A: ... (정중)
B: ... (장황)
→ human: A > B
⚡
DPO ⭐
Direct Preference Optimization
Reward 모델 없이 preference pair로 직접 최적화. RLHF보다 훨씬 간단·안정. 2024~2026 오픈소스에서 사실상 표준.
{
 "prompt": "...",
 "chosen": "좋은 답",
 "rejected": "나쁜 답"
}

🔥 LIMA 논문 — "1000개면 충분하다"
"Less Is More for Alignment. 65B LLaMA를 단 1,000개의 엄선된 instruction pair로 SFT했을 때, 학습 없이 RLHF한 모델과 경쟁 가능한 성능."

✓ LIMA 이후
"엄선된 1,000개면 충분하다. 양 << 품질. Base model이 이미 지식을 갖고 있고, SFT는 포맷/스타일만 가르친다."
💡 Superficial Alignment Hypothesis: 모델의 지식은 pre-training에서 거의 다 학습됨. fine-tuning은 "출력 형식"을 가르치는 것에 가까움. 이것이 적은 데이터로 충분한 이유.

UltraFeedback
64k DPO 데이터. GPT-4가 비교 라벨링. Zephyr, Starling 등 2024 오픈 모델의 핵심.

Allen AI, 2024. SFT + DPO 혼합 최대 오픈 레시피. Llama 3 기반 Tulu 3를 재현 가능하게 공개.
내부 도메인 데이터
2026년 엔터프라이즈의 실제 선택. 사내 QA 로그, 문서, 업무 대화를 1~5k개 정제 → 전용 모델.
품질 > 양 — 2026 Fine-tuning 원칙 5가지
① 다양성 > 중복
같은 형태 10,000개보다 다양한 형태 1,000개가 낫다.
② 포맷 일관성
응답 스타일·길이·톤이 일관되어야 모델이 흉내낼 수 있다.
③ Edge case 포함
쉬운 사례만 있으면 어려운 케이스를 배우지 못한다.
④ 사람 검수 필수
synthetic 데이터도 최종은 사람이 샘플링 검수를 거쳐야 한다.
⑤ "언제 fine-tune이 필요한가?" 자문
Prompt engineering, few-shot, RAG로 안 되는 이유가 명확할 때만. 2026년 90% 케이스는 fine-tune 없이 해결 가능합니다.

Cosine Similarity — 두 벡터가 얼마나 비슷한가
수식
cos(θ) = (A · B) / (‖A‖ × ‖B‖)
두 벡터 사이의 각도 θ의 코사인. 방향만 보고 길이는 무시하기 때문에 "긴 문서"와 "짧은 문서"도 공정하게 비교됩니다.
해석
+1.0 — 완벽히 동일 방향 (같은 의미)
+0.7~0.9 — 매우 유사
0.0 — 직교 (무관)
-1.0 — 정반대

cos("연차 규정", "피자 주문")         = 0.09 ← 완전 무관

벡터 DB — AI 시대의 저장소

임베딩을 만들었다면 어디에 저장해서 어떻게 검색할까요? PostgreSQL로도 할 수 있지만, 10억 벡터를 millisecond에 검색하려면 전용 엔진이 필요합니다. 벡터 DB는 "유사도 검색"을 1급 연산으로 지원하는 새로운 종의 데이터베이스입니다.

왜 전통 DB로는 부족한가 — Exact Match vs Similarity
전통 RDBMS
Exact Match
SELECT * FROM docs
WHERE title = '연차 규정';
정확히 "연차 규정"이 있어야 매칭. "휴가 신청"은 안 됨. ❌
Vector DB
Similarity Search (k-NN)
query_vec = embed("연차 규정")
db.search(query_vec, top_k=5)
"휴가 신청", "PTO policy", "연차 사용법" 모두 매칭. ✅
💡 핵심 차이: 전통 DB는 "B-Tree + WHERE절"로 정확한 조건을 찾습니다. 벡터 DB는 "고차원 공간에서 가장 가까운 이웃"을 찾습니다 — 완전히 다른 알고리즘이 필요합니다.
ANN 알고리즘 — 왜 "Approximate"인가
10억 벡터 전수조사 (brute force)는 초당 1건도 못 합니다. 그래서 Approximate Nearest Neighbor(ANN) — "거의 정확한" 대신 "수 ms 안에" 답을 반환하는 자료구조를 씁니다.
HNSW ⭐
Hierarchical Navigable Small World
다층 그래프. 상층부터 "스킵 리스트"처럼 내려가며 찾음. 2026년 사실상 표준. Pinecone, Weaviate, Qdrant 기본.
Recall: ~99% | Speed: ⚡⚡⚡⚡⚡
IVF
Inverted File Index
k-means로 공간을 여러 클러스터로 나눔. 쿼리가 들어오면 가까운 몇 개 클러스터만 검색. Facebook FAISS.
Recall: ~95% | Speed: ⚡⚡⚡
PQ
Product Quantization
벡터를 압축해서 메모리 절약. 1536차원 float32 → 수십 바이트. IVF-PQ 조합이 대규모에 필수.
Memory: ↓↓↓ | Accuracy: trade-off
HNSW 작동 원리 — 다층 그래프 탐색
Layer 2
Layer 1
Layer 0
상층(Layer 2)은 "고속도로", 하층(Layer 0)은 "골목". 상층에서 시작해 단계적으로 내려가며 log(N) 시간에 최근접을 찾음.

  • 벡터 수 < 수천만
  • 관계형 데이터와 JOIN이 필요하다
  • 트랜잭션·백업·권한이 중요
  • 운영 부담을 늘리기 싫다

  • p99 latency < 50ms 필수
  • Hybrid search·reranker 네이티브
  • Multi-tenancy, 네임스페이스 관리
  • 진정한 분산·샤딩 필요
📌 2026년의 현실: 90%의 RAG 프로젝트는 pgvector로 충분합니다. "일단 Pinecone부터"의 시대는 끝났고, 필요해지기 전엔 단순한 걸로 시작하는 게 새 정석입니다.

데이터 편향 — AI가 왜 편향되는가

AI는 중립이 아닙니다. 데이터가 편향되면 모델도 편향됩니다. 그것도 은밀하게, 통계적 패턴으로 포장된 채. 2026년 EU AI Act가 "고위험 AI"에 특별 규제를 걸어둔 이유도 여기에 있습니다. 편향은 기술 문제가 아니라 사회적·법적 문제입니다.

편향의 3가지 핵심 유형
TYPE 1
Selection Bias
누구를 데이터에 포함시켰는가?
"지원자만" 보면 "탈락자"를 모름.
표본이 모집단을 대표하지 못함.
TYPE 2
Historical Bias
과거 사회의 불공정이 데이터에 박힘.
"과거 승진자 = 남자"였다면
모델이 남자를 선호하도록 학습.
TYPE 3
Measurement Bias
측정 자체가 왜곡됨.
얼굴 인식 훈련에
밝은 피부 사진이 90%라면?
실제 사례 ① — Amazon 채용 AI (2018)
HISTORICAL BIAS
Amazon은 이력서 자동 점수 매기기 AI를 10년간 개발했습니다. 훈련 데이터는 "지난 10년간 합격한 엔지니어 이력서". 문제는 — 2008~2018년 테크 업계 지원자·합격자는 압도적으로 남성이었습니다.
# 모델이 학습한 패턴
"여자 체스 동아리 회장" → 감점
"여자 대학교 출신" → 감점
"women's coding camp" → 감점
결과: 2018년 Amazon은 프로젝트를 전면 폐기. "편향을 제거해도 안심할 수 없다"는 뼈저린 교훈.
실제 사례 ② — COMPAS 재범 예측 (2016)
HISTORICAL + MEASUREMENT BIAS
미국 법원은 COMPAS라는 AI로 피고인의 재범 가능성을 점수화하여 보석 결정에 사용했습니다. 2016년 ProPublica의 탐사 보도로 폭발적 문제가 드러났습니다.
False Positive (무고한데 위험 판정)
흑인: 45%  ↔  백인: 23%
False Negative (위험한데 안전 판정)
백인: 48%  ↔  흑인: 28%
결과: 모델은 race를 직접 안 봤지만, 우편번호·이전 체포 횟수 같은 "proxy 변수"가 race와 상관있었습니다. 편향 제거는 "민감 속성 빼기"만으로 안 됩니다.

🇪🇺 EU AI Act 연결 — 편향이 법적 리스크가 된 시대
2024년 발효된 EU AI Act는 고용·교육·신용·법 집행 등을 "고위험 AI"로 분류하고, 이들에 대해 다음을 요구합니다:
  • 훈련 데이터의 품질·대표성 증명
  • 편향 평가 및 완화 조치 문서화
  • 지속적인 모니터링과 결과 로깅
  • 인간 감독(human oversight) 메커니즘
  • 위반 시 전 세계 매출의 최대 7% 과징금 (GDPR보다 높음)
💡 2026년 현재 "편향 감사"는 compliance 필수 체크리스트. 개발 단계부터 품질+공정성을 측정해야 합니다. "나중에 고치지"가 안 통합니다.

Data-Centric AI — Andrew Ng의 혁명

2021년 3월, 딥러닝의 대부 Andrew Ng가 폭탄 선언을 했습니다 — "모델은 그만 만들어라. 데이터를 고쳐라." 수년간 모델 경쟁에 지쳐있던 업계가 충격에 빠졌고, 이 철학은 지금 AI 실무의 사실상 표준이 되었습니다. Part 1의 마지막 핵심 메시지.

💡 모델을 바꾸면 1% 오르고, 데이터를 고치면 10% 오른다는 게 Ng 팀의 실증 관찰. 무작위 에러 분석보다 "잘못된 라벨만 수정"이 더 큰 효과를 내는 경우가 흔합니다.
Small Clean vs Big Noisy — 어느 쪽이 이기는가
Big Noisy Data
500,000 images, 30% mislabeled
Training accuracy: 99.2%
Production accuracy: 62%
→ "잘 외웠지만, 현실에선 실패"
Small Clean Data ⭐
10,000 images, 99% labeled correctly
Training accuracy: 96.5%
Production accuracy: 91%
→ "덜 외웠지만, 일관되게 잘 함"
📌 LLM 시대 예외 아님: Anthropic Claude, DeepSeek R1 모두 "고품질 소규모 SFT/RLHF 데이터"의 효과를 강조합니다. Pre-training은 대량이지만, 모델 성격을 결정하는 post-training 데이터는 quality first.
실전 사례 — Landing AI의 제조·의료 현장
🏭
제조 — 결함 검출
문제: 불량품 사진이 50장밖에 없음 (희귀).
전통 ML: "데이터 10만 장 모아야 합니다".
Landing AI: 50장을 매우 정확히 라벨링 + 라벨러 가이드 표준화 → 95%+ 정확도 달성.
🏥
의료 — 영상 판독
문제: 방사선과 의사 5명이 같은 영상에 다른 라벨.
Data-Centric 접근: "라벨 일관성"을 수치화 → 가이드라인 재작성 → 재라벨링 → 15%p 성능 개선. 모델 변경 없이.
Data-Centric AI Competition (NeurIPS 2021)
Andrew Ng팀이 주최한 역사적 대회. "모델은 고정, 데이터만 개선해서 정확도를 올려라."
고정된 부분
  • ResNet50 아키텍처 (고정)
  • 하이퍼파라미터 (고정)
  • Training loop (고정)
참가자가 바꿀 수 있는 것
  • 데이터 추가·제거
  • 라벨 수정
  • Augmentation 선택

데이터 파이프라인 — ETL, ELT, Streaming

데이터는 저장소에 자동으로 나타나지 않습니다. 수집하고, 변환하고, 적재하는 파이프라인이 24/7 돌아가야 합니다. 파이프라인은 지난 30년간 ETL → ELT → Streaming으로 진화했고, 2026년 Modern Data Stack이 그 결과입니다.

ETL vs ELT — 순서가 바뀌면 세상이 바뀐다
1990~2015
ETL (전통)
Extract → Transform → Load
"저장하기 전에 변환한다"
소스 → ETL 서버(Informatica) → DW
변환이 전용 서버에서
🏢 Informatica, SSIS, Talend, DataStage
💰 라이선스 비용 ↑, 유연성 ↓
2015~ 현재
ELT ⭐ (현대)
Extract → Load → Transform
"원본 먼저 저장, 나중에 변환"
소스 → Fivetran → Snowflake/BQ → dbt
변환이 DW 안에서 SQL로
🚀 Fivetran, Airbyte, dbt, Snowflake, BigQuery
☁️ Cloud-first, Git-friendly, 저렴
🎯 왜 바뀌었나? Cloud DW(Snowflake, BigQuery)가 저장·컴퓨팅을 분리하면서 "DW 안에서 SQL로 변환"이 더 빠르고 싸졌습니다. 굳이 별도 ETL 서버를 둘 이유가 사라진 것. 이것이 Modern Data Stack의 출발점입니다.
Batch vs Streaming — 시간의 축
📦
Batch
"하루에 한 번, 전날치 데이터 일괄 처리".

장점: 단순, 싸다, 디버깅 쉬움
단점: 최신성 떨어짐
도구: Airflow, Dagster, Prefect
사용 사례: 야간 ETL, 월간 리포트
🌊
Streaming ⭐
"이벤트가 발생하면 millisecond에 처리".

장점: 실시간, 운영 대응
단점: 복잡, 비쌈, state 관리 어려움
도구: Kafka, Flink, Pulsar
사용 사례: 사기 탐지, IoT, 실시간 추천
CDC — Change Data Capture, 실시간의 다리
트랜잭션 DB의 WAL(Write-Ahead Log)을 읽어, 변경된 row만 다운스트림에 흘려보내는 기법. 2026년 실시간 파이프라인의 사실상 표준.
# PostgreSQL WAL
INSERT orders (id=4812, amount=$250)  →  Debezium  →  Kafka topic: orders-cdc
UPDATE orders SET status='paid'  →  Debezium  →  Kafka topic: orders-cdc
DELETE orders WHERE id=4812  →  Debezium  →  Kafka topic: orders-cdc

# 소비자들
→ Snowflake (분석용)
→ Elasticsearch (검색용)
→ Redis (캐시)
→ Vector DB (임베딩 재계산)
💡 핵심 도구: Debezium, Airbyte CDC, Fivetran HVR. 하나의 소스 변경을 여러 다운스트림에 "팬아웃"하는 것이 강점.
