01

엔터프라이즈 데이터 스택 2026 — 전체 아키텍처

Part 8에서 "데이터가 어디에 사는가"(DB·DW·Lake·Lakehouse)를 배웠습니다. Part 9 Ch.01은 여기서 한 단계 나아갑니다 — "2026년 엔터프라이즈는 그 위에 무엇을 얹었는가". 답은 Lakehouse + Vector DB + Feature Store + MLOps의 4층 구조, 그리고 이 전체를 묶는 Data-as-a-Product 모델입니다.

🔥 2026 우선순위 전환 — "더 많이"에서 "더 믿을 수 있게"로
~2023
"Data Volume War"
• "우리는 PB급이다"
• Data Lake에 전부 덤프
• 사용률은 보지 않음
• "나중에 분석" 희망
• 품질은 후순위
→
2026
"Decision-Ready Data"
• 결정 가능한 (trusted)
• 거버넌스되는 (lineage)
• 감사 가능한 (audit log)
• 적을수록 좋음 (essential)
• AI-ready format
🎯 한 줄 요약: 데이터 리더 90%가 "데이터가 많은 게 문제가 아니라 어떤 데이터를 믿을 수 있는지 모르는 게 문제"라고 답했습니다 (Monte Carlo State of Data Quality 2026). AI가 의사결정을 자동화할수록, 그 아래 데이터의 "신뢰 가능성"이 가장 비싼 자원이 됩니다.

🔢 Layer 3 — Feature Store + Vector DB
"AI가 쓸 수 있게 가공"
📌 Part 8 복습: 임베딩과 벡터 DB 개념이 처음이라면 먼저 복습하세요. 여기서는 Feature Store와 Vector DB의 운영 차이를 다룹니다.
📖 Part 8 Data 기초로 돌아가기 — 임베딩/벡터 DB 복습 →
📊 Feature Store
용도: 전통 ML (tabular 데이터) — 고객 LTV, 이탈 예측, 사기 탐지
핵심: Feature 재사용 + 일관성 (Online = Offline)
대표 도구: Feast (오픈소스), Tecton, Databricks Feature Store
서빙: Online (p99 < 10ms) + Offline (배치 학습)
🔍 Vector DB
용도: RAG / 시맨틱 검색 (비정형 데이터) — 문서, 이미지, 코드
핵심: ANN(근사 최근접 이웃) 검색 + 메타데이터 필터링
대표 도구: Pinecone, Weaviate, Qdrant, pgvector
서빙: 유사도 검색 → LLM 컨텍스트 주입
Feature Store 워크플로
Vector DB 워크플로 (RAG)
📈 2026 트렌드: Feature Store가 Vector DB를 흡수하는 추세. Tecton은 vector feature를 네이티브 지원하고, Databricks Feature Store는 벡터 인덱스를 내장. "정형 feature와 벡터 feature를 하나의 서빙 레이어에서" 가 방향.
💥 실패 패턴: 유령 벡터
임베딩 모델을 교체(v1→v2)했는데 기존 벡터를 re-index하지 않음 → v1 벡터와 v2 쿼리가 다른 공간에 존재 → 검색 품질 급락.
✅ 해결: Re-indexing 전략
Blue/Green 인덱스 — 새 모델로 전체 re-embed → 새 인덱스에 적재 → 품질 검증 후 트래픽 전환 → 구 인덱스 삭제. 다운타임 제로.

Data-as-a-Product (DaaP) — 2026의 새로운 계약
Zhamak Dehghani가 2019년에 제안한 Data Mesh의 핵심 개념이 2026년 엔터프라이즈의 기본 운영 모델로 자리 잡았습니다. 데이터는 더 이상 "중앙 팀의 자산"이 아니라 도메인 팀이 오너십을 갖는 제품입니다.
① Discoverable (검색 가능)
Data Marketplace에서 "고객 신용 평가용 테이블 있나요?"를 검색하고 구독 가능. 메타데이터 · 오너 · SLA가 공개.
② Addressable (주소 있음)
고유 URI로 참조 가능: data://finance/daily-revenue/v3. 버전 관리는 모델처럼.
③ Trustworthy (신뢰 가능)
Data Contract로 스키마·품질·freshness SLA를 명시. 위반 시 소비자에게 자동 알림. Great Expectations, Soda가 검증.
④ Self-describing (자기 설명)
스키마 + 샘플 + 예시 쿼리 + 오너 + 용도가 문서화. AI 에이전트가 읽고 "이 데이터로 답할 수 있는지" 판단.

Chunking 전략 — 숨은 승부처

"청크를 어떻게 자르느냐"는 RAG 성능의 30-40%를 결정합니다. 그런데 대부분의 튜토리얼은 "1000자씩 자르세요"로 끝납니다. 이 챕터는 2026년 현장에서 검증된 chunking의 전략별·도메인별 답입니다.

🏆 2026 기본값 — "몰라서 시작한다면 이렇게"
Recursive · 256-512 토큰 · 10-25% overlap
Prem AI RAG Benchmark 2026 기준 대부분 도메인에서 안정적 상위권. 시작은 여기서.
📐 왜 이 숫자들: 256-512 토큰 ≒ 1-2 단락. 임베딩 모델이 가장 "의미 조밀"하게 인코딩하는 구간. Overlap 10-25%는 경계에서 잘린 문장을 복구하기 위한 safety margin.
4가지 Chunking 전략 — 상세
① Fixed-Size
"N자/N토큰씩 무식하게"
가장 단순. 500자/500토큰 단위로 기계적으로 자름.
장점: 빠름, 일관됨.
단점: 문장/단락 중간에서 잘림.
쓸 때: prototype, 로그 데이터.
② Recursive ⭐
"계층적 구분자 우선"
우선순위 구분자 ["\n\n","\n",". "," "]를 재귀적으로 시도. 단락 → 문장 → 단어 순으로 깨뜨림.
장점: 자연 경계 존중.
쓸 때: 기본값 — 일반 텍스트·FAQ·위키.
③ Semantic
"의미 유사도로 자른다"
문장별 임베딩 → 인접 문장 유사도 급락 지점에서 컷. Greg Kamradt가 유행시킴.
장점: 주제 경계 포착.
단점: 임베딩 비용 2배.
쓸 때: 법률·학술 논문.
④ Page-level / Structural
"문서 구조를 그대로"
페이지·섹션·헤딩 단위로 자름. PDF/슬라이드처럼 구조가 있는 문서에 강함.
장점: 테이블/차트 보존.
단점: 크기 불균일.
쓸 때: 금융 보고서·IR·슬라이드.

from langchain_text_splitters import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
  chunk_size=512, # 토큰 기준
  chunk_overlap=80, # 약 16% overlap
  separators=["\n\n", "\n", ". ", " "],
  length_function=tiktoken_len, # 토큰 카운터
)

chunks = splitter.split_documents(docs)
# → 각 chunk는 <=512 토큰, 경계는 단락→문장→공백 우선

Hybrid Search + Reranking — Vector-only의 한계 극복

"임베딩 모델만 좋으면 된다"는 2023년의 신화였습니다. 2026년의 합의는 분명합니다 — Hybrid Search + Reranker가 기본값. Vector-only RAG는 이제 "기초 튜토리얼"의 영역이고, 프로덕션은 3단 구성을 씁니다.

💡 먼저 짚고 가는 것: "RAG = 임베딩 + 벡터 DB"라는 이미지는 2022-2023년의 기억입니다. 실제로 PoC를 돌려보면 곧 한계가 드러납니다 — 제품 코드 같은 정확한 단어는 못 찾고, 신제품명은 무시되며, 비슷해 보이는 답이 자주 1순위로 올라옵니다. 이 챕터는 "왜 이런 일이 생기는가"부터 짚고, 2026 표준 해법인 BM25 + Vector + Reranker 3단 구성이 어떻게 그 한계를 메우는지 단계별로 풀어냅니다. 각 단계가 무엇을 책임지는지, 비용은 얼마나 드는지, 어떤 모델을 골라야 하는지까지 한 번에 봅니다.
왜 Vector-only는 부족한가 — 3가지 약점
벡터 검색은 "의미가 비슷한 문서를 찾는" 능력이 강력합니다. "자동차"로 검색하면 "차량", "vehicle"이 나오죠. 그런데 이 강점이 곧 약점이 되기도 합니다 — 의미가 비슷한 것을 찾느라 정확히 일치하는 것을 놓칩니다. 아래 3가지가 실전에서 가장 자주 부딪히는 함정입니다.
🎯
Exact match 약함
"SKU# A-3099-XL" 같은 정확한 ID는 벡터 유사도가 흐려짐. BM25는 즉시 정확 매치.
🔤
희귀어 취약
신제품명·사람 이름·약어는 훈련 데이터에 없어 임베딩이 "의미 없는 평균"으로 수렴.
📏
거리 ≠ 관련성
코사인 0.85가 0.84보다 꼭 좋은 답은 아님. cross-encoder로 재정렬 필요.
📌 현장에서 자주 보는 시나리오: "AS-2024-01 모델 사양표 알려줘"라고 물었는데 검색 결과 1순위가 "AS 시리즈 일반 소개 PDF"로 나옵니다. 의미적으로는 가깝지만 사용자가 원한 건 정확한 모델 코드의 사양표였죠. 임베딩은 "모델 코드 ID" 같은 토큰의 정확도에 약합니다 — 그래서 BM25가 도와줘야 합니다.
Hybrid Search — BM25 + Vector + RRF
핵심 아이디어: 두 검색 방식이 서로 다른 영역에서 강하다면, 둘을 합치면 약점이 메워집니다. 키워드 검색(BM25)은 정확한 단어를 잘 찾고, 벡터 검색은 의미를 잘 잡습니다. 이 둘의 결과 리스트를 RRF로 합쳐 하나의 순위표를 만드는 것이 Hybrid Search입니다.

아래에서 각 방식의 강·약점을 본 뒤, 어떻게 합치는지(RRF)를 봅니다.
🔤 BM25 (Sparse)
키워드 TF-IDF 기반. 1994년부터 검증된 클래식.
강점: exact match, 희귀어, 숫자/ID
약점: 동의어 무시 ("car" vs "automobile")
동작 원리: "이 단어가 문서에서 얼마나 자주 등장하는가(TF)" + "이 단어가 전체 코퍼스에서 얼마나 흔한가(IDF)"의 조합. 희귀하면서 자주 등장하는 단어 = 그 문서의 핵심 키워드로 판단.
🧠 Vector (Dense)
임베딩 기반 의미 검색. 2020년 이후 주류.
강점: 의미 유사성, 동의어, 다국어
약점: exact match, 희귀어
동작 원리: 텍스트를 768~3072 차원 벡터로 변환 → 코사인 유사도 계산. "차"와 "자동차"가 다른 단어라도 벡터 공간에서 가까이 있어 함께 검색됨.
🔀 RRF (Reciprocal Rank Fusion) — 결합 방법
두 결과 리스트를 순위 기반으로 결합. 공식: score(d) = Σ 1/(k + rank_i(d)), 보통 k=60.
점수 스케일이 달라도 순위만 쓰니 튜닝이 쉬움. 2026년 사실상 표준 결합 알고리즘.
왜 RRF가 표준이 됐나: 점수 합산(weighted sum)은 BM25 점수(예: 0~50)와 코사인(0~1)의 스케일이 너무 달라 가중치 튜닝이 어려웠습니다. RRF는 순위(rank)만 쓰므로 스케일이 항상 동일 — 어떤 검색 엔진을 섞어도 동작. 그래서 Elastic·OpenSearch·Weaviate 등 주요 엔진이 모두 채택했습니다.
2026 표준 파이프라인 — 3단 검색
실전 RAG 파이프라인은 "넓게 잡고 → 좁히고 → 답하기"의 3단 깔때기 구조입니다. 각 단계가 다른 책임을 맡습니다 — Hybrid는 recall(놓치지 않기), Reranker는 precision(정확도), LLM은 합성(생성). 한 모델에 다 맡기지 않고 책임을 분리한 이유는 비용·속도·품질의 균형 때문입니다.

📐 왜 100 → 10인가? — 숫자의 근거
100을 뽑는 이유: Hybrid가 너무 좁게(top-10) 뽑으면 reranker가 봐야 할 정답 후보가 빠질 수 있음(recall ↓). 너무 넓게(top-1000) 뽑으면 reranker 비용이 10배. 100은 recall 95%+ 보존하면서 reranker 비용을 감당 가능한 균형점.

10으로 좁히는 이유: LLM 컨텍스트 윈도가 길어졌어도 여전히 "관련 없는 5개를 끼워넣으면 답이 흔들린다"는 Lost in the Middle 현상이 있음(Liu et al., 2023). 그래서 가장 관련 높은 5-10개만 LLM에 넘기는 것이 사실상 표준. Anthropic·OpenAI 공식 가이드도 이 범위 권장.
2026 Reranker 카탈로그

🎯 top-100만 보면 된다

RAG Evaluation — 측정하지 않으면 개선할 수 없다

Part 4 Ch.08에서 "AI 시스템 평가"의 일반론을 배웠습니다. 여기서는 그것의 RAG 특화 버전입니다. RAG는 "검색이 잘 됐나" + "답이 잘 됐나"의 두 축이 얽혀 있어 일반 LLM 평가보다 복잡합니다. 2024-26년 표준 프레임워크로 자리 잡은 RAGAS와 동료들을 봅니다.

왜 RAG를 평가해야 하나 — "보이지 않는 실패"
🙈
Silent Failure
RAG는 "모르겠습니다" 대신 그럴듯한 헛소리를 생성합니다. retrieval이 빈약해도 LLM은 자신 있게 답합니다. 에러 로그에 안 잡힙니다.
🔀
두 축 얽힘
답이 틀렸을 때 검색이 잘못인지 생성이 잘못인지 분리해야 고칠 수 있습니다. 하나의 "정답률"로는 원인 분석이 불가능합니다.
📉
Regression
청크 크기 바꿨더니 어제 잘 되던 질문이 오늘 안 됩니다. 회귀 테스트셋 없이는 배포가 공포입니다.
💡 2026년 현장의 교훈: "RAG PoC는 금요일까지, 프로덕션은 6개월". 이 6개월의 대부분은 평가 루프를 돌리며 청크·임베딩·reranker·프롬프트를 튜닝하는 시간입니다. 평가가 없으면 PoC는 영원히 PoC로 남습니다.

GraphRAG — 지식 그래프의 재부상

Vector RAG는 "한 청크에 답이 있는 질문"에 강합니다. 하지만 여러 문서·여러 사실을 엮어 추론해야 하는 질문엔 무력합니다. 2024년 Microsoft Research가 공개한 GraphRAG는 이 한계를 지식 그래프로 풀어냈고, 2025-26년 LinkedIn·Neo4j 사례로 실전 검증됐습니다.

왜 Vector RAG만으론 부족한가 — "Multi-hop의 저주"
질문: "우리 자회사 B가 만드는 Y 제품에 들어가는 C 부품의 공급사 D가 혹시 러시아 제재 명단에 있나?"
VECTOR RAG
"B 자회사 Y 제품 C 부품 D 공급사 러시아 제재"를 한 청크에서 같이 언급한 문서가 존재할 가능성에 의존. 그런 문서는 거의 없음. 실패.
GRAPH RAG
B →produces→ Y →contains→ C →supplied_by→ D →in_list→ 제재명단. 그래프 순회로 4-hop 추론. 성공.
💡 원리: Vector RAG는 "의미적으로 가까운 것"을 찾고, GraphRAG는 "관계로 연결된 것"을 찾습니다. 전자는 빠르고 단순, 후자는 느리고 복잡하지만 다단계 추론에 강합니다.
🏗 Microsoft GraphRAG — 3가지 검색 모드
2024년 7월 Microsoft Research가 오픈소스 공개. 핵심 아이디어: LLM이 문서에서 엔티티·관계를 추출하고, Leiden 알고리즘으로 커뮤니티 클러스터를 만든 뒤, 커뮤니티별 요약을 미리 생성해 저장. 질문 유형에 따라 3가지 모드로 검색.
MODE 1
🌍 Global
"전체 문서 코퍼스의 주요 테마는?" 같은 요약형 질문. 커뮤니티 요약을 map-reduce로 합침. 비쌈 but 강력.
MODE 2
🎯 Local
"X 엔티티에 대해 알려줘" 같은 특정 대상 질문. 엔티티 중심 1-2 hop 그래프 + 텍스트 청크 결합.
MODE 3
🌊 Drift
혼합 질문. Global로 시작해 관련 커뮤니티로 "drift"하며 local까지 내려감. 2025 추가, 가장 범용.
⚙️ 구현 스택: MS GraphRAG 공식 repo는 Python. 저장은 Parquet + LanceDB. 2025년 Neo4j + LangChain GraphRAG 통합이 기업에서 더 인기 — 이미 Neo4j를 쓰는 곳이 많아서입니다.

온톨로지와 지식 그래프 — 의미 계층의 귀환

Ch.09에서 GraphRAG를 만났습니다. 이 장은 그 한 층 아래 — "그래프에 의미를 얹는 레이어"입니다. 단어는 어렵지만 원리는 직관적입니다. "김아무개는 PwC에 근무한다"처럼 세상을 작은 문장(트리플) 단위로 적고, 여기에 "사람은 회사에 근무한다"라는 규칙을 얹어 놓으면 컴퓨터가 스스로 새로운 사실을 도출합니다. 2010년대에 시들었다가 2024-2025 LLM 시대에 극적으로 되돌아온 이유와 함께, 기초부터 그림으로 풀어봅니다. 16개 서브섹션을 4개 흐름으로 이해하면 됩니다 — (1) 왜 필요한가 → (2) 기본 원리 → (3) 표준·생태계 → (4) 실전·설계·역사.

❓ 10-1. 왜 지금, 왜 다시? — 같은 대상, 여러 이름의 문제
기술 용어보다 먼저, 문제부터 봅시다. 현장에서 LLM·AI 에이전트가 가장 많이 틀리는 순간은 "같은 것을 서로 다른 이름으로 부를 때"입니다. 온톨로지는 이 어긋남을 해결하는 "지능형 지도"입니다 — 여러 표현을 하나의 의미로 묶어주는 기반 계층.
🏥 의료 · 감기약의 세 이름
일반인: "감기약"
의사 차트: "URI 치료제" (Upper Respiratory Infection)
약국 POS: "타미플루"
→ 온톨로지 없이는 AI가 "이들이 같은 것"임을 모름. 처방 알림·상호작용 체크가 엇나감.
🏗️ 건설 · 철근의 세 이름
현장 작업자: "리바" (일본어 rebar에서)
설계 스펙: "SD400" (항복강도 400MPa)
구조 도면: "주근" (역할 명칭)
→ 같은 자재가 세 스택에 흩어져 있음. BIM·시공 AI 자동 연계의 최대 걸림돌.
🍅 식품 · 토마토의 양면성
식물학적으로는 과일(씨앗 포함 자방)
요리 맥락에서는 채소(짠 요리에 씀)
1893년 미 대법원 판결: 관세 목적상 "채소"
→ 맥락에 따라 분류가 달라짐. 단일 전역 분류가 아닌 "관점별 분류 체계"가 필요.
🛒 마트 비유 · 분류 원칙 자체
같은 "토마토"도 마트마다 매대 위치가 다름 — 채소 코너 / 샐러드 재료 / 이탈리안 식재료 / 유기농. "분류 원칙"이 매장 운영 철학을 드러냅니다.
→ 온톨로지 = "분류 원칙", 지식 그래프 = 그 원칙대로 배치된 실제 물건들의 네트워크.
🧭 핵심 메시지: LLM 환각의 상당수는 "같은 대상을 서로 다른 이름으로 부르는 현실"을 모델이 의미 수준에서 잇지 못해서 생깁니다. 온톨로지는 그 간극을 메우는 "의미 계층의 지도"입니다. 정규표현식으로는 못 잇고, 벡터 유사도로도 부족합니다 — 명시적·형식적 선언이 있어야 합니다.
"An ontology is a specification of a conceptualization."
— Tom Gruber, Stanford KSL (Knowledge Systems Laboratory), Knowledge Acquisition 5(2):199-220, 1993
풀어 쓰면 — "이 도메인엔 어떤 종류가 있고, 어떤 관계가 성립하고, 어떤 규칙이 따라오는가"를 컴퓨터가 추론할 수 있는 형태로 적은 것. 30년 넘게 이 한 문장이 학술 온톨로지 정의의 뿌리입니다.
📐 10-2. 가장 기본 — 세상을 "트리플(Triple)"로 적기
온톨로지 세계의 원자 단위는 트리플(triple)입니다. 모든 지식을 (주어 · 술어 · 목적어) 3요소의 문장으로 쪼개 적습니다. 학술 용어로는 Subject - Predicate - Object. 이 구조가 단순해서 어떤 지식이든 표현할 수 있고, 컴퓨터가 다루기도 쉽습니다.
💡 트리플을 여러 개 쌓으면 자연스럽게 그래프가 됩니다. "김아무개-근무한다-PwC", "PwC-소재한다-서울", "PwC-자회사다-PwC Global" ... 노드(주어/목적어)가 재사용되면서 네트워크가 만들어집니다. 이것이 곧 지식 그래프(Knowledge Graph)의 원형입니다.
① CLASSES
개념(종류)
Person, Company, City … "어떤 종류가 있는가"
② PROPERTIES
관계·속성
worksAt, locatedIn … "어떤 관계가 성립하는가"
③ AXIOMS
공리·규칙
"Person은 Mammal이다" 같은 항상 성립하는 규칙
온톨로지 = ① + ② + ③  ·  지식 그래프 = 여기에 구체적 인스턴스(트리플)까지 얹은 것
✨ 10-3. 핵심 마법 — "자동 추론(Reasoning)"이 대체 뭔가
온톨로지가 단순 "관계 저장소"와 다른 결정적 차이는 자동 추론 엔진(OWL reasoner)입니다. 몇 가지 규칙(공리)만 선언해 두면, reasoner가 알아서 적어 두지 않은 새로운 사실을 도출해 줍니다. 작은 예시로 감을 잡아봅시다.
📝 우리가 적어둔 것 (선언)
Mammal ⊂ Animal  (공리)
Dog ⊂ Mammal  (공리)
Fido : Dog  (인스턴스)
hasOwner(Fido, 김아무개)  (인스턴스)
⊂ 기호는 "하위 집합" — "Mammal은 Animal의 하위"라는 뜻.
OWL
REASONER
⟹
(규칙 적용)
✨ Reasoner가 자동으로 도출
Mammal ⊂ Animal
Dog ⊂ Mammal
Fido : Dog
Dog ⊂ Animal  ← 새 사실!
Fido : Mammal  ← 새 사실!
Fido : Animal  ← 새 사실!
아무도 "Fido는 동물이다"라고 입력한 적이 없지만, 규칙으로부터 자동 도출됩니다.
🔑 왜 중요한가? ① 일관성 검증 — "Fido는 Dog이면서 동시에 Plant다"처럼 모순을 reasoner가 자동 발견. ② 설명 가능성 — "Fido가 동물인 이유"를 추론 경로로 제시. ③ LLM 환각 억제 — LLM의 답변을 온톨로지로 검증하면 근거 없는 주장을 걸러낼 수 있음. OG-RAG의 +40% correctness가 이 메커니즘에서 나옵니다.
🍷 10-4. 실제 온톨로지 해부 — Wine 예시로 보는 4-Layer 구조
Noy & McGuinness의 "Ontology Development 101" (Stanford, 2001) 논문이 쓴 Wine 예시는 지금도 교과서입니다. 추상적 원리(클래스·속성·제약)를 "잡을 수 있는 와인 한 병"까지 내려와서 확인하는 방식. 4개 계층을 한 번에 봅시다.
① CLASSES — 개념 계층
도메인의 "종류". 상위→하위 is-a 관계.
Wine ⊃ Red Wine ⊃ Bordeaux ⊃ Pauillac
Pauillac의 모든 인스턴스는 자동으로 Bordeaux·Red Wine·Wine의 인스턴스 (추이성).
② SLOTS — 속성·관계
각 클래스가 가질 수 있는 속성·관계.
Wine 클래스의 slot: flavor, body, sugar, color, maker, grape
maker는 단순 속성이 아닌 Winery 클래스와의 관계.
③ FACETS — slot 제약
slot의 값 유형·개수·제약.
flavor: Enumerated(strong/moderate/delicate) · 1개
grape: 최소 1개(블렌드는 여러 개)
Dessert Wine의 sugar: 항상 "SWEET" (하위 클래스에서 강제)
④ INSTANCES — 실제 개체 ⭐
클래스의 구체 사례. "잡을 수 있는" 개별 와인.
Chateau-Morgon-Beaujolais · Body=Light · Color=Red · Flavor=Delicate · Tannin=Low · Grape=Gamay · Maker=Chateau-Morgon · Region=Beaujolais
🔁 역슬롯(Inverse Slot) — 양방향 자동 동기화
Wine의 maker와 Winery의 produces는 서로 역관계입니다. 하나만 기록해도 반대편이 자동 갱신 — 입력 중복 제거 + 정합성 보장.
:Sterling-Merlot  :maker  :Sterling-Vineyard   ← 하나만 선언
⟹ :Sterling-Vineyard  :produces  :Sterling-Merlot   ← 자동 도출
💡 실무 포인트: 이 4-Layer 구조는 SNOMED CT·FIBO·Gene Ontology 같은 어떤 온톨로지를 뜯어봐도 같은 패턴입니다. 단어만 다를 뿐 — OWL 표기에선 Class/Property/Restriction/Individual. Noy & McGuinness "Ontology Development 101" Stanford KSL-01-05 (2001)이 가장 널리 인용되는 입문서.
🧭 10-5. 가장 많이 헷갈리는 4가지 — Taxonomy / Schema / Ontology / KG
"Taxonomy classifies. Schema validates. Ontology means. Knowledge graph remembers."
① TAXONOMY
분류 체계
대상을 카테고리로 분류. 계층 구조(tree)가 전형.
예: 서점 장르 분류, 생물 분류학, 듀이 십진 분류법
② SCHEMA
스키마
데이터 구조를 검증(유효/무효). "맞는 형식인가"만 판정.
예: JSON Schema, SQL DDL, Avro, Protobuf
③ ONTOLOGY ⭐
온톨로지
개념·관계의 의미를 명세 + 자동 추론. "왜 그런가"까지.
예: OWL 공리 "Mammal ⊂ Animal", FIBO, SNOMED CT
④ KNOWLEDGE GRAPH
지식 그래프
사실들의 집합을 기억(인스턴스 트리플). 실제 데이터.
예: Wikidata 1억+ 엔티티, Google KG 5천억+ 사실
💡 실무 포인트: "온톨로지 vs 지식 그래프"는 대립이 아닙니다. "문법 vs 글"의 관계 — 쓸만한 KG는 반드시 인스턴스 그래프 + 스키마/온톨로지가 함께 있어야 합니다.
🔀 10-6. 두 세계 — Property Graph(Neo4j) vs Ontology(RDF/OWL)
"그래프 DB"라고 불리는 두 진영이 있습니다. 실용 그래프 진영(Neo4j 대표)과 의미론 그래프 진영(Stardog·GraphDB·Virtuoso). Ch.09 GraphRAG에서 주로 쓰는 건 전자, 본 챕터의 주인공은 후자.
관점 Property Graph (Neo4j) Ontology (RDF/OWL)
기본 단위노드·엣지·속성(k-v)Subject-Predicate-Object 트리플
스키마유연·선택적엄격·선언적
자동 추론없음 (앱이 쿼리로 해결)있음 (OWL reasoner)
쿼리 언어Cypher / Gremlin / GQL(ISO)SPARQL (W3C)
표준화벤더 주도 (GQL로 수렴 중)W3C 공식 표준
대표 사용처소셜·사기 탐지·운영 그래프의료·법률·과학·공공 데이터
대표 제품Neo4j, TigerGraph, Memgraph, NeptuneStardog, GraphDB(Ontotext), Virtuoso, Fuseki
CYPHER · Property Graph
"김아무개가 근무하는 회사의 도시는?"
MATCH (p:Person {name:'김아무개'})
      -[:WORKS_AT]->(c:Company)
      -[:LOCATED_IN]->(city:City)
RETURN city.name
SPARQL · RDF/OWL
같은 질문 — 트리플 패턴으로
SELECT ?cityName WHERE {
  :김아무개 :worksAt ?company .
  ?company :locatedIn ?city .
  ?city rdfs:label ?cityName .
}
표기가 다를 뿐 개념은 같습니다. SPARQL은 reasoner와 결합하면 "worksAt의 상위 속성"까지 자동으로 고려 — Cypher는 그 확장이 수동입니다.
🏛️ 10-7. W3C 4대 표준 스택 — RDF / RDFS·OWL / SPARQL / SHACL
시맨틱 웹 생태계는 W3C가 정의한 4개 표준을 쌓아 올린 구조입니다. 각 계층이 무엇을 하는지 한 장으로 이해하면, 이후 어떤 도구·제품을 만나도 "어디에 속하는가"를 바로 짚을 수 있습니다.
RDF — 데이터 모델
모든 사실을 트리플로. URI로 고유 식별.
:김아무개  :worksAt  :PwC .
:PwC  :locatedIn  :Seoul .
OWL — 스키마·추론 규칙
클래스 포함 관계·속성 제약을 선언.
:Dog  rdfs:subClassOf  :Mammal .
:worksAt  rdfs:domain  :Person .
SPARQL — 질의
트리플 패턴 매칭. SQL의 RDF판.
SELECT ?p WHERE {
  ?p :worksAt :PwC .
}
SHACL — 검증 제약
"Person은 반드시 이름이 있어야" 같은 규칙.
:PersonShape a sh:NodeShape ;
  sh:targetClass :Person ;
  sh:property [ sh:path :name ;
                sh:minCount 1 ] .

🔀 10-11. 같은 단어, 두 계보 — 학술 온톨로지 vs Palantir 운영 온톨로지

A. 학술 온톨로지
W3C OWL / Protégé 계보

📋 10-14. 온톨로지를 직접 만든다면? — Noy & McGuinness 7단계
"직접 만들려면 어디서 시작하나?" Stanford KSL의 Noy & McGuinness가 2001년에 정리한 7단계 프로세스가 여전히 표준입니다. v2 시대엔 2·5·6단계에 LLM을 들여와 공수를 크게 줄일 수 있지만, 뼈대는 동일합니다.
STEP 1
🎯
도메인·범위 결정 + 역량 질문(Competency Questions)
"온톨로지가 답해야 할 질문들"을 먼저 씀. 예: "Bordeaux는 레드인가 화이트인가?", "구운 고기에 어울리는 와인은?". 이 목록이 곧 범위 결정의 기준.
STEP 2 ⭐
♻️
기존 온톨로지 재사용 — 맨땅에서 시작하지 말 것
SNOMED·FIBO·Schema.org·Wikidata·UNSPSC 등을 먼저 검토. 이미 검증된 30년 어휘가 있는데 새로 만드는 건 대부분 잘못된 선택. LLM v2 시대에도 이 원칙은 유지.
STEP 3
📝
중요 용어 나열 — 일단 쏟아내기
이 단계에선 클래스인지 속성인지 구분하지 않음. 도메인 용어를 포괄적으로 쓰는 데 집중. 와인 예: wine, grape, winery, body, flavor, maker, sugar, color, food pairing...
STEP 4
🌳
클래스 · 계층 정의 — Top-down / Bottom-up / Hybrid
Top-down: 일반 → 구체 (Wine → Red Wine → Bordeaux)
Bottom-up: 구체 → 일반 (Pauillac · Margaux → 공통 상위 Medoc 도출)
Hybrid: 중심 개념부터 양쪽으로 확장 (실무에서 가장 흔함)
원칙: 형제 클래스는 동일 일반성 수준 · 1개뿐이면 재설계 · 12개 넘으면 중간 카테고리 도입.
STEP 5 ⭐
🔗
속성(slot) 정의
Step 3 용어 중 "독립 객체가 아닌 것"을 slot으로 변환. maker·grape는 다른 클래스와의 관계 slot. v2 시대엔 LLM이 텍스트에서 slot 후보를 자동 추출(AutoSchemaKG).
STEP 6
🔒
패싯(facet) 정의 — 값 유형 · 카디널리티 · 도메인·레인지
카디널리티: Wine.body=1, Winery.produces=여러 개
값 유형: String / Number / Boolean / Enumerated / Instance
도메인·레인지: maker의 domain=Wine, range=Winery
STEP 7
🍇
인스턴스 생성 — 실제 데이터 채우기
클래스에 구체 개체를 만듦. Chateau-Morgon-Beaujolais의 모든 슬롯 값 입력. v2에선 LLM이 문서에서 자동 추출(10-9 KG 구축 파이프라인 참조).
🧭 3가지 대원칙 (Noy & McGuinness)
① "단 하나의 정답은 없다" — 애플리케이션·확장 방향·설계자의 관점에 따라 올바른 온톨로지가 달라집니다.
② 반복 프로세스 — 1차 초안 → 실사용 → 전문가 검토 → 수정. 도메인이 바뀌면 온톨로지도 바뀝니다.
③ 객체(명사) + 관계(동사) — "와인이 음식과 어울린다" → 객체: 와인·음식, 관계: pair-with. 자연어 문장을 분해하면 온톨로지 구성요소가 드러납니다.

Privacy & PII — LLM 시대의 실전 방어

GDPR·PIPA·HIPAA가 다루는 핵심은 PII (Personally Identifiable Information)입니다. LLM 시대에 PII 리스크는 새 모양으로 돌아왔습니다 — 모델이 학습 데이터를 기억하고, prompt에 섞인 PII를 그대로 로그에 남기고, tool 사용으로 추가 PII를 긁어옵니다. 2026년 엔터프라이즈는 PII를 "가림"이 아니라 "아키텍처"로 다룹니다.

PII 정의 — 직접 vs 간접 (Quasi)
DIRECT PII
"혼자서도 누군지 안다"
• 이름 (full name)
• 주민등록번호 / SSN
• 여권·운전면허 번호
• 이메일 · 전화번호
• 신용카드 번호
• 얼굴 사진 · 지문
• 계좌번호 · IBAN
QUASI / INDIRECT PII
"조합하면 누군지 안다"
• 생년월일 + 우편번호 + 성별
• 직장 + 직책 + 지역
• IP 주소 · 쿠키
• 구매 이력 · 위치 히스토리
• 브라우저 fingerprint
• 익명 의료 기록 + 희귀질환 코드
🔬 Sweeney 1997: 미국인의 87%는 생년월일+5자리 우편번호+성별만으로 유일하게 식별 가능. "익명화된" 데이터가 전혀 익명이 아닐 수 있습니다.
🛡 PII 처리 6가지 기법
①
Redaction
가림·삭제. "홍길동의 연봉" → "[이름]의 연봉". 원복 불가.
Presidio Anonymizer
②
Tokenization
고유 토큰으로 치환. "홍길동" → "USR_4f8a". vault에 원본 보관, 역매핑 가능.
Skyflow
③
Masking
부분 가림. "010-1234-5678" → "010-****-5678". 확인용은 OK, ID로는 부족.
④
k-Anonymity
최소 k명이 같은 조합을 갖도록 일반화. "34세 서울" → "30대 수도권". k=5, k=20 선택.
⑤
Differential Privacy
쿼리 결과에 수학적 noise. ε(epsilon)으로 privacy-utility 트레이드오프 정량화. Apple·Google 인구조사.
⑥
Federated Learning
데이터를 모으지 않고 각 디바이스에서 모델만 학습. 그래디언트만 전송. 의료·금융.
💡 실전 조합: ①+② (redaction + tokenization) → LLM 프롬프트에 토큰만 전달, 응답 후 vault에서 원본으로 복원. "Zero PII to LLM" 패턴.

Synthetic Data + Data Security — 가짜 데이터, 진짜 공격

Part 9의 마지막 챕터는 두 주제를 엮습니다 — 없으면 만들어내는 합성 데이터와, 있는 데이터를 지키는 보안입니다. 둘은 짝입니다. 합성 데이터는 privacy를 해결하는 듯 보이지만, 잘못 쓰면 Model Autophagy Disorder로 모델을 망칩니다. 데이터 보안 공격은 2026년 들어 학습 파이프라인 자체를 노립니다.

🌱 왜 Synthetic Data가 뜨는가 — 3가지 이유
REASON ①
Privacy 해결
"Fake but statistically realistic". 실제 고객 데이터 없이 같은 분포를 재현해 개발·테스트 가능. GDPR·HIPAA 우회의 정공법.
REASON ②
희귀 케이스 증강
사기 탐지·희귀 질환·안전사고 — positive sample이 부족. 합성으로 minority class를 늘려 모델 학습.
REASON ③
LLM Fine-tuning
데이터 고갈이 현실. Anthropic·OpenAI 모두 내부 학습에 LLM 자체 생성 데이터 사용(Constitutional AI, RLAIF 등).
📊 Gartner 2024 예측: "2030년까지 AI 모델 학습 데이터의 60%+가 합성". 2024 Cosmopedia, 2025 Phi-4, 2026 Llama-4 — 모두 합성 데이터 대량 사용.
🏭 생성 방법 — LLM이 LLM 데이터를 만든다
▸ Instruction 생성: 100개 씨앗 → LLM이 유사 작업 10,000개 생성 → 필터링 → 학습
▸ Constitutional AI: 모델이 자기 답변을 원칙 체크리스트로 비평·수정 → 학습 신호
▸ RLAIF: RLHF의 H(human)를 LLM(AI)로 대체. 평가자 LLM이 rank
▸ Distillation: 큰 모델이 작은 모델용 고품질 데이터 생성 (Phi 시리즈)
▸ Tabular GAN/VAE: 정형 데이터용 (Mostly AI, Gretel)
🧪 실전 관찰: Anthropic·OpenAI 공개 자료에 따르면 합성 데이터 품질의 핵심은 "다양성"과 "필터링". 단순 재샘플링은 효과 없음. Reward 모델·rubric으로 나쁜 샘플을 걸러내는 파이프라인이 성패를 가름.
⚠️ 위험 — Mode Collapse & MAD
MODE COLLAPSE
다양성 감소
생성 모델이 안전한 소수 패턴만 반복. 평균 품질은 괜찮아 보이지만 분포 꼬리가 잘립니다. 희귀 케이스(원래 증강 목적!)가 오히려 사라집니다.
MODEL AUTOPHAGY DISORDER
"모델이 자기 꼬리를 먹는다"
Alemohammad et al. 2023 (Rice Univ). 합성→학습→합성→학습 반복 시 품질·다양성이 지수적으로 붕괴. MAD로 명명. 5세대 만에 얼굴 생성 모델이 거의 동일한 얼굴만 뱉음.
🛡 해독제: ① 실제 데이터와 섞어서(보통 1:1~1:3) 학습 · ② 세대 수 제한(≤2) · ③ 다양성 메트릭(distinct-n, self-BLEU) 모니터링 · ④ 인간 검수 샘플링.

🎯 데이터 보안 공격 3대장 — 학습 파이프라인을 노린다
ATTACK ①
☠️ Data Poisoning
학습 데이터에 악성 샘플을 심어 모델 행동을 왜곡. Backdoor: 특정 trigger 입력 시만 공격자 의도대로. 오픈소스 scraping 파이프라인에 취약.
ATTACK ②
🔍 Membership Inference
"이 샘플이 학습에 쓰였는가?"를 외부에서 추론. 의료·금융 모델에선 그 자체가 PII 유출. Shokri et al. 2017로 체계화.
ATTACK ③
🪞 Model Extraction
API 호출만으로 동등 성능의 복제 모델을 만듦. 상용 모델의 IP 침해. distillation 기법 + 수십만 query.
🛡 방어: ① 학습 데이터 출처 검증 + 해시 체크 · ② Differential Privacy(Membership inference 방어) · ③ API rate limit + 쿼리 로깅 · ④ Canary 샘플로 중독 조기 탐지 · ⑤ MLOps에 supply chain security(SLSA).
