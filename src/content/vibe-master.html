TDD (Test-Driven Development) — 테스트 주도 개발

테스트를 먼저 쓰고, 통과하는 코드를 나중에 쓴다. AI 시대에도 원칙은 동일.

1. RED — 인간이 실패하는 테스트 작성
2. GREEN — AI가 통과 코드 구현
3. REFACTOR — 인간이 리뷰 → AI가 정리
버그 40~80% 감소 — 2025 DORA
AI의 치팅에 주의
· 테스트 삭제/skip으로 통과
· assert 약화 (값 → 타입 검증)
· 구현 먼저, 거기 맞춘 테스트
→ 테스트 수정 금지 필수 규칙
SDD (Spec-Driven Development) — 스펙 주도 개발

코드가 아닌 스펙이 진실의 원천. 무엇을(WHAT) 먼저 정의하고, 어떻게(HOW)는 AI에게.

1. SPECIFY — 인간이 "무엇을" 정의
요구사항 · 수용 기준 · 제약조건
언제 쓰나: 신규 기능 · 대규모 설계 · 여러 세션에 걸친 작업 · 팀 협업 (스펙 = 커뮤니케이션)
2. PLAN — AI가 "어떻게" 제안
기술 스택 · 아키텍처 → 인간 승인
3. TASKS — 분해 후 병렬 실행
작업↔요구사항 매핑 유지. 모호한 부분 → [NEEDS CLARIFICATION] 마커로 구현 중단.
핵심: 각 단계가 파일로 저장 · 커밋 · 참조 → 컨텍스트 소실 방어의 정석
결론: Spec + TDD

Chapter 05

조합하는 패턴 — Plan-Critic-Build, Ralph Loop

도구를 알았으니, 조합하는 패턴입니다. 에이전트가 가장 많이 실패하는 지점: 계획 없이 바로 코드 작성. 해결책은 단순 — 계획 단계를 쓰기 권한 없는 모드로 분리.

Plan–Critic–Build — 계획과 실행의 분리
1. PLAN
읽기만 · 쓰기 금지
파일 읽기·검색·질문만. 결과: 편집 가능한 계획 문서.
2. CRITIC
다른 눈으로 본다
리뷰 전용 에이전트. 작성자의 blind spot을 드러내기.
3. BUILD
승인 후 실행
승인된 계획 링크만 컨텍스트로. 쓰기 권한 개방.
핵심: Planner와 Critic을 같은 컨텍스트에 두지 않기. 자기 계획 → 자기 비평 시 blind spot 잔존.
Claude Code의 Plan Mode(Shift+Tab ×2) — 1단계를 구조적으로 강제.

"뭘 보여줘야 하지?" — 답은 이미 머릿속에. 리뷰할 때 말하던 것, 버그 겪고 반사적으로 체크하던 것 → 파일로 추출.

멀티 에이전트 활용

Part 1 Ch.07에서 멀티 에이전트의 3가지 아키텍처(Supervisor · Swarm · Hierarchical)를 배웠다면, 여기서는 "실제로 어떻게 쪼개고 조합하는가"를 봅니다. 핵심 원칙: 하나의 에이전트 = 하나의 일.

01
컨텍스트의 벽

02
역할의 벽
"쓰고, 구현하고, 리뷰하고, 리팩터하라"를 한 에이전트에게 → 전부 어설프게
자기 코드 자기 리뷰 = blind spot
03
신뢰성의 벽
단계별 95% × 20단계 연쇄 = 전체 성공률 36%
긴 체인 = 수학적 실패
Subagent vs Agent Team
Subagent
▎구조: 메인이 생성한 격리된 워커
▎컨텍스트: 자체 윈도우, 요약만 반환
▎통신: 주 에이전트에게만 보고
적합: 많은 파일 읽기, 병렬 조사
Agent Team
▎구조: 여러 독립 세션이 협업
▎컨텍스트: 완전히 독립, 각자 별도 윈도우
▎통신: 팀원끼리 직접 메시지
적합: 보안/성능/테스트 동시 검토

01
컨텍스트 격리
