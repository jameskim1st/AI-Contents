🤔 왜 이것이 역설인가 — 3가지 이유
①
검증 능력
AI 제안을 검증하려면 코드를 읽을 수 있어야 합니다. AI가 미묘하게 틀린 걸 못 잡아내면 그대로 프로덕션에 들어갑니다. SQL 인젝션, 메모리 누수, 동시성 문제 — 모두 "그럴듯해 보이지만 잘못된" 코드의 함정입니다.
②
설계 능력
아키텍처 설계(DB 스키마, 캐싱 전략, API 경계, 보안 모델)는 AI가 못합니다. 기초가 있어야 "AI가 무엇을 해야 하는가"를 명확히 정의할 수 있습니다. 정의가 모호하면 AI는 모호한 결과만 생성합니다.
③
방향잡이 가치
결국 기초 있는 사람이 AI를 방향잡고 검증합니다. AI는 도구일 뿐이고, 도구를 잘 쓰는 사람의 가치는 도구가 강력해질수록 함께 올라갑니다. "AI가 코딩을 해주니 코딩 못 해도 된다"는 정확히 그 반대입니다.
🔑 한 줄 요약: "AI가 코드 작성을 해주니 우리는 코드 이해를 더 잘해야 한다." 작성에서 해방된 시간을 검증·설계·방향잡이에 투자하는 사람이 새 시대의 승자가 됩니다.

TDD (Test-Driven Development) — 테스트 주도 개발

테스트를 먼저 쓰고, 통과하는 코드를 나중에 쓴다. AI 시대에도 원칙은 동일.

1. RED — 인간이 실패하는 테스트 작성
2. GREEN — AI가 통과 코드 구현
3. REFACTOR — 인간이 리뷰 → AI가 정리
버그 40~80% 감소 — 2025 DORA
AI의 치팅에 주의
· 테스트 삭제/skip으로 통과
· assert 약화 (값 → 타입 검증)
· 구현 먼저, 거기 맞춘 테스트
→ 테스트 수정 금지 필수 규칙
SDD (Spec-Driven Development) — 스펙 주도 개발

코드가 아닌 스펙이 진실의 원천. 무엇을(WHAT) 먼저 정의하고, 어떻게(HOW)는 AI에게.

1. SPECIFY — 인간이 "무엇을" 정의
요구사항 · 수용 기준 · 제약조건
언제 쓰나: 신규 기능 · 대규모 설계 · 여러 세션에 걸친 작업 · 팀 협업 (스펙 = 커뮤니케이션)
2. PLAN — AI가 "어떻게" 제안
기술 스택 · 아키텍처 → 인간 승인
3. TASKS — 분해 후 병렬 실행
작업↔요구사항 매핑 유지. 모호한 부분 → [NEEDS CLARIFICATION] 마커로 구현 중단.
핵심: 각 단계가 파일로 저장 · 커밋 · 참조 → 컨텍스트 소실 방어의 정석
결론: Spec + TDD

하네스 엔지니어링 — AI 에이전트 성패의 핵심 ⭐

Agent = Model + Harness. 모델이 아닌 것은 전부 하네스입니다. 마구(馬具, horse tack)처럼, 강력하지만 예측 불가능한 말(= LLM)을 올바른 방향으로 이끄는 장비 전체가 하네스입니다.

💡 Part 1과의 연결

Part 1에서 배운 Memory · Tools · Planning — 이것들이 바로 하네스의 구성요소입니다. Part 1이 "에이전트 안에 뭐가 있나"를 배웠다면, 여기서는 "그것들을 어떻게 설계하나"를 배웁니다.

왜 모델만으로는 부족한가

LLM의 본질 = 텍스트 입력 → 텍스트 출력 함수. 그 자체로는 다음을 할 수 없다:

❌ 세션 간 상태 유지
❌ 코드 실행
❌ 실시간 정보 접근
❌ 환경 구성 · 패키지 설치
심지어 "채팅"조차 하네스의 산물이다. 이전 메시지를 추적하고 이어붙이는 while 루프 — 이것이 가장 기본적인 하네스. 우리가 ChatGPT와 "대화"한다고 느끼는 모든 경험은 모델 위에 씌워진 하네스 덕분이다.

💡 모델이 멍청해서가 아니다. 교대 근무 엔지니어를 상상해보라 — 매번 새 엔지니어가 이전 근무자의 기억 없이 출근하는데, 인수인계 문서도, 진행 보드도, 테스트 환경도 없다면? 아무리 뛰어난 엔지니어라도 제대로 일할 수 없다.
하네스의 5대 핵심 축
1
파일시스템과 지속적 저장소
에이전트에게 작업 공간을 주고, 세션을 넘어 상태를 유지. Anthropic의 claude-progress.txt — 각 세션이 자기 일을 기록하고, 다음 세션이 읽고 이어받는다.
2
코드 실행과 샌드박스
모든 행동을 미리 도구로 만들 수 없다 → bash와 코드 실행이 범용 도구. 단, 격리된 샌드박스에서 실행 — 허용된 명령만, 네트워크 제한.
3
컨텍스트 관리 — 핵심 전장 ⭐
컨텍스트가 채워질수록 에이전트가 멍청해진다 (Context Rot). 대응 전략: 컴팩션(요약·덜어내기) · 도구 출력 오프로딩(앞뒤만 남기고 파일로) · 스킬(필요할 때만 로드)
"성공은 조용히, 실패만 시끄럽게" — HumanLayer 팀의 핵심 원칙
4
서브 에이전트 = 컨텍스트 방화벽
"역할 분담"이 아니라 컨텍스트 격리가 진짜 가치. 조사·탐색의 노이즈를 서브가 흡수하고, 상위에겐 결과만. "큰 컨텍스트 윈도우 = 더 큰 건초더미일 뿐. 바늘 찾는 능력이 좋아지는 게 아니다."
5
훅과 백프레셔 메커니즘
에이전트 생명주기의 특정 시점에 자동 실행되는 스크립트. 타입체크·테스트를 자동 실행 → 에러 시 되돌려보냄. Anthropic: Puppeteer로 에이전트가 직접 브라우저 테스트 → 코드만 봐서는 못 찾는 버그 발견.
📋 하네스 레이어 & 부재 시 리스크

실무에서 반복 등장하는 5가지 레이어. 각각이 없을 때 어떤 리스크가 발생하는지가 설계의 출발점.

레이어 역할 부재 시 리스크
지식 베이스
CLAUDE.md · AGENTS.md
 레포지토리 안에 저장된 아키텍처 결정, 프로젝트 컨텍스트, 실행 규칙. 에이전트의 "헌법" 에이전트가 팀 컨벤션을 무시하거나, 이미 결정된 사항을 다시 질문
아키텍처 제약
Custom Linter · Structure Test
 에이전트가 건드려서는 안 되는 코드나 시스템에 물리적으로 접근하지 못하도록 차단 에이전트가 모델 경계를 넘어서는 규칙을 '자연스럽게' 무시 → 아키텍처 붕괴

📌 무슨 벤치마크? 어떤 "다른 하네스"?

벤치마크: SWE-bench — 1,900개 이상의 실제 GitHub 이슈를 해결하는 능력을 측정합니다. 단순 코딩 문제가 아니라, 실제 오픈소스 프로젝트의 버그·기능 요청을 가져와 "이걸 고쳐봐"라고 시킵니다. 2026 Q1 기준 순위입니다.

"다른 하네스" 예시:
• Codex (OpenAI 참조 구현) — TDD 강제 + 린터 자동 실행 + 작업 분해 우선
• Aider (오픈소스) — 대화형, Git 자동 커밋, repo-map 자동 생성
• SWE-agent (학술) — 파일 편집/검색/실행 도구 최소 세트 + 명시적 plan 단계
vs Claude Code (자기 하네스) — 범용 도구 다수, 자유도 높음, plan은 권장 사항

결론: 같은 모델(Opus 4.6)인데도 하네스 설계만으로 28등 차이가 납니다. 모델 성능의 한계는 1~2등급 정도지만, 하네스는 경쟁 우위를 역전시킵니다. 이게 "엄밀함의 재배치" 명제의 실증입니다.

하네스 엔지니어링 실전 원칙 5가지
01
실패에서 시작하라
이상적인 하네스를 미리 설계하지 말고, 실패할 때마다 구조적으로 방지하는 장치를 추가하라 — Hashimoto 원칙
02
적게 넣어라
ETH Zurich: 138개 설정 파일 테스트 → LLM 생성 파일은 성능↓ 비용 20%↑. 코드베이스 개요나 디렉터리 목록은 무용지물 — 에이전트는 직접 탐색한다
03
도구를 과하게 연결하지 마라
MCP 서버 많이 연결 → 도구 설명이 인스트럭션 예산을 잡아먹음. CLI가 훈련 데이터에 있는 도구라면, MCP 대신 CLI를 쓰라
04
점진적 작업을 강제하라
Anthropic: 가장 큰 개선 = 한 번에 한 기능만 + 매번 Git 커밋 + 진행 노트. 다음 세션이 깨끗한 상태에서 시작.
05
엄밀함의 재배치 (Relocating Rigor)
코드를 한 줄 한 줄 정확하게 짜던 엄밀함이 → 에이전트가 올바르게 작동하도록 시스템을 설계하는 엄밀함으로 이동 — Chad Fowler
🔬 왜 디렉터리 목록이 "무용지물"인가

두 가지 원리가 작동합니다:

① 토큰 낭비 + 주의 산만 — 디렉터리 목록 1만 파일을 컨텍스트에 넣으면 약 30,000 토큰을 차지합니다. 이 30K가 실제 작업에 도움이 되지 않을 뿐 아니라, 모델의 attention을 분산시켜 정작 중요한 부분에 집중을 못 합니다.

② 에이전트의 직접 탐색이 더 효율적 — 에이전트는 find, grep, ls 같은 도구로 필요할 때만 탐색합니다. 미리 다 보여주는 것보다 "필요한 순간 찾는" 게 100배 효율적입니다.

결론: 우리가 제공해야 할 건 "맵(전체 지도)"이 아니라 "탐색 도구"입니다. 책을 통째로 읽혀주는 게 아니라, 색인을 잘 만들어두면 됩니다.

(이 원칙은 코드베이스 개요 외에도 ① 사용하지 않을 가능성이 높은 함수 시그니처 ② 바뀌지 않는 라이브러리 문서 ③ 일반적인 코딩 베스트 프랙티스에도 적용됩니다.)

🧩 왜 이 3-부분 구조가 효과적인가
원칙
에이전트가 요점을 빠르게 파악합니다. 한 줄이라야 컨텍스트 비용이 적고, 매 작업마다 다시 읽어도 부담이 없습니다. 길어지면 무시당하기 시작합니다.
why
에이전트가 새로운 상황에 원칙을 일반화할 수 있게 합니다. 예: "종결어미는 명사형만"이라는 규칙은 이유 없으면 슬라이드 타이틀에만 적용되지만, "한눈에 읽혀야 하므로"라는 이유가 붙으면 모든 텍스트(버튼, 카드 제목, 알림)에 자동 확장됩니다.
예시
모호한 경우의 정확한 판단 기준을 제공합니다. "친근한 톤" 같은 주관적 규칙도 ✅ "어서 와요!" ❌ "환영합니다" 같은 예시 한 쌍이면 명확해집니다. 단, 예시는 1~2개로 제한 — 10개 이상은 오히려 주의 산만만 야기합니다.
💡 반대 패턴(피해야 할 것): ① 원칙만 있고 이유 없음 → 일반화 못 함 ② 이유만 있고 원칙 없음 → 너무 추상적 ③ 예시만 있고 원칙 없음 → 외워야 함. 셋이 다 있어야 비로소 "전이 가능한 규칙"이 됩니다.
🚦 승인 게이트 워크플로우 — "계획 없이 구현 금지"

📜 규칙 체계 — Norms vs Guardrails

에이전트를 통제하는 것은 두 종류의 규칙 파일입니다.

📏 Norms (행동 규범)

에이전트의 판단 기준이 되는 원칙들. "어떻게 판단할 것인가"

• "코드에서 확인 가능한 사실은 되묻지 말고 판단하라"
• "사용자 의견에 순응하지 말고, 논리적 문제는 근거로 반론하라"
• "구현 완료 후 배포 전에 규범을 다시 읽고 자체 점검하라"

🚫 Guardrails (금지 사항)

에이전트가 넘어서는 안 되는 하드 바운더리. "무엇을 절대 하지 말 것인가"

• "코드 바깥의 맥락은 코드만 보고 단정하지 마라"
• "재검토 요청 시 이전 결론에 끌려가지 말고 처음부터 다시"
• "사용자 승인 없이 외부 시스템에 쓰기 요청을 보내지 마라"
🔄 피드백 루프 — "하지마", "그거 말고"가 규칙이 되는 과정

사용자가 교정 신호를 보낼 때마다 하네스가 이를 자동으로 기록하고, 압축하고, 규칙으로 승화시키는 자기 학습 회로.

모델이 더 똑똑해지면 하네스의 일부가 모델에 흡수될 것이다. 하지만 프롬프트 엔지니어링이 여전히 가치 있듯, 잘 구성된 환경·적절한 도구·지속적 상태·검증 루프는 어떤 에이전트든 더 효과적으로 만든다.

05

조합하는 패턴 — Plan-Critic-Build, Ralph Loop

도구를 알았으니, 조합하는 패턴입니다. 에이전트가 가장 많이 실패하는 지점: 계획 없이 바로 코드 작성. 해결책은 단순 — 계획 단계를 쓰기 권한 없는 모드로 분리.

Plan–Critic–Build — 계획과 실행의 분리
1. PLAN
읽기만 · 쓰기 금지
파일 읽기·검색·질문만. 결과: 편집 가능한 계획 문서.
2. CRITIC
다른 눈으로 본다
리뷰 전용 에이전트. 작성자의 blind spot을 드러내기.
3. BUILD
승인 후 실행
승인된 계획 링크만 컨텍스트로. 쓰기 권한 개방.
핵심: Planner와 Critic을 같은 컨텍스트에 두지 않기. 자기 계획 → 자기 비평 시 blind spot 잔존.

"뭘 보여줘야 하지?" — 답은 이미 머릿속에. 리뷰할 때 말하던 것, 버그 겪고 반사적으로 체크하던 것 → 파일로 추출.

①결정의 "이유"를 쓴다 — 결과만 → 금방 낡음. 이유까지 → 일반화 가능
②Post-Mortem → 규칙 — 계획과 구현의 분기점을 한 줄로 승화

④취향 → 전용 에이전트 — 글로 쓰기 어려운 판단을 에이전트로 encode

POISONING · 중독

LLM Wiki — 지식의 컴파일 NEW · 2026.04

2026년 4월, Andrej Karpathy가 GitHub gist 하나를 던졌습니다. 제목은 단 두 단어 — "LLM Wiki". 며칠 만에 AI 커뮤니티에서 폭발적 화제가 됐습니다. 우리가 Ch.04~08까지 배운 하네스 엔지니어링의 자연스러운 확장이며, Part 1에서 본 Memory 패턴의 야심찬 진화형입니다.

💡 이전 챕터와의 연결

Ch.04~05의 하네스 엔지니어링이 "한 번의 작업에서 컨텍스트를 잘 관리하는 법"이었다면, LLM Wiki는 "프로젝트 전체의 수명 동안 지식을 누적하는 법"입니다. 같은 사고방식의 다른 시간 척도.

🔥 문제 — RAG는 매번 처음부터 시작한다
"LLM이 매 질문마다 처음부터 지식을 재발견하고 있다. 누적이 없다."
— Andrej Karpathy, 2026.04 (LLM Wiki gist)
😫 RAG의 비유
매일 같은 책 5권을 처음 보는 사람처럼 답한다. 어제 답했던 질문이 와도 책을 처음부터 다시 펼친다.
✨ LLM Wiki의 비유
책을 읽고 노트를 정리한 학생처럼 답한다. 같은 질문이 오면 노트만 본다. 새 책이 오면 노트에 추가한다.
핵심 통찰: "검색"이 아니라 "기억의 운영"이다.
3계층 아키텍처
1
Raw Sources
read-only
원본 문서. PDF, PPT, 논문, 강의 자료, 이전 회의록. LLM은 읽기만 하고 절대 수정하지 않습니다.
2
Wiki
LLM-maintained
LLM이 만들고 갱신하는 markdown 페이지들. 요약 · entity 페이지 · 상호참조 링크로 구성. 사람도 직접 읽을 수 있는 형태.
3
Schema
human-curated
위키 운영 규칙. CLAUDE.md의 지식 관리 버전입니다. 사람이 작성하고, LLM이 따른다.
💡 사람이 하는 것: 원본 자료 큐레이팅, Schema 작성, 질문 던지기  ·  LLM이 하는 것: Wiki 유지 보수의 모든 것
3가지 핵심 운영 — Ingest · Query · Lint
OPERATION 1
📥 Ingest
새 자료가 들어오면:
① 자료를 읽는다
② 등장하는 entity 추출
③ 기존 페이지 10~15개 갱신
④ 새 entity 페이지 생성
⑤ index와 log에 기록
OPERATION 2
🔍 Query
사용자가 질문하면:
① index에서 카테고리 찾기
② 관련 entity 페이지 읽기
③ 답변 합성
④ 새 통찰을 wiki에 편입
⑤ 다음 질문은 더 빠르고 풍부
OPERATION 3
🧹 Lint
주기적 점검:
① 모순 검출 (같은 사실 다른 설명)
② 고아 페이지 (index에 없는 것)
③ 죽은 링크
④ stale entity (오래된 것)
⑤ 사람의 검토 요청
💡 Query → Ingest 피드백 루프가 핵심입니다. 답변 자체가 다음 질문을 위한 자산이 됩니다. 그래서 시간이 지날수록 위키가 더 똑똑해집니다.
RAG vs LLM Wiki — 무엇이 다른가
관점 RAG LLM Wiki
지식 처리 시점 매 질문마다 (검색 시점) 한 번 컴파일, 계속 유지
지식 누적 없음 — 매번 새로 발견 있음 — 자산화
상호참조 매번 새로 발견 이미 연결되어 있음
모순 처리 그때그때 (불일관) 추적 · 플래그 · 사람 리뷰
사람이 직접 읽기 어렵다 (벡터 임베딩) 쉽다 (markdown)
갱신 비용 낮음 (인덱싱만) 중간 (LLM이 정리)
💭 비유: 인터프리터 vs 컴파일러. RAG는 매번 해석, LLM Wiki는 미리 컴파일.
CLAUDE.md → Wiki Schema — 같은 사고방식의 시간 척도 확장
CH.05에서 배운 것
CLAUDE.md
대상: 한 번의 작업
갱신: 가끔 (사람 수동)
결과: 코드 변경
시간 척도: 분~시간
→
CH.09 새 패턴
Wiki Schema
대상: 장기 지식 누적
갱신: 자주 (LLM 자동)
결과: wiki 페이지 갱신
시간 척도: 주~년
핵심: 둘 다 "사람이 LLM에게 지속적인 컨텍스트를 미리 제공한다"는 같은 사고방식입니다. CLAUDE.md가 "이번 task에서 어떻게 일할지"를 알려준다면, Wiki Schema는 "이 프로젝트가 살아있는 동안 지식을 어떻게 키울지"를 알려줍니다.
🎯 살아있는 데모 — 이 사이트가 직접 LLM Wiki를 운영합니다

이 강의를 만들면서, 우리는 사이트 자체에 LLM Wiki를 적용했습니다. c:/AI/ai-novice/wiki/ 디렉토리에 실제 위키가 살아있고, 새 자료(블로그, 논문, 새 챕터)가 추가될 때마다 Claude Code가 ingest해서 entity 페이지를 갱신합니다.

⚠️ 한계와 주의점 — 만능은 아니다
⚠️ 잘못된 요약의 굳기
사람의 검토 없이 LLM이 만든 요약이 "공식 지식"으로 굳을 위험. 사람이 주기적으로 lint를 돌리고 검증해야 함.
⚠️ 모순 누적
자료 출처가 서로 모순이면 위키가 혼란스러워짐. Lint로 잡아내지만, 해결은 사람이 해야 함.
⚠️ 휘발성 자료
대상이 너무 자주 바뀌면 갱신 비용 > 검색 비용. 안정적이고 누적되는 지식에 적합하다.
