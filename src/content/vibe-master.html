🤔 왜 이것이 역설인가 — 3가지 이유
①
검증 능력
AI 제안을 검증하려면 코드를 읽을 수 있어야 합니다. AI가 미묘하게 틀린 걸 못 잡아내면 그대로 프로덕션에 들어갑니다. SQL 인젝션, 메모리 누수, 동시성 문제 — 모두 "그럴듯해 보이지만 잘못된" 코드의 함정입니다.
②
설계 능력
아키텍처 설계(DB 스키마, 캐싱 전략, API 경계, 보안 모델)는 AI가 못합니다. 기초가 있어야 "AI가 무엇을 해야 하는가"를 명확히 정의할 수 있습니다. 정의가 모호하면 AI는 모호한 결과만 생성합니다.
③
방향잡이 가치
결국 기초 있는 사람이 AI를 방향잡고 검증합니다. AI는 도구일 뿐이고, 도구를 잘 쓰는 사람의 가치는 도구가 강력해질수록 함께 올라갑니다. "AI가 코딩을 해주니 코딩 못 해도 된다"는 정확히 그 반대입니다.
🔑 한 줄 요약: "AI가 코드 작성을 해주니 우리는 코드 이해를 더 잘해야 한다." 작성에서 해방된 시간을 검증·설계·방향잡이에 투자하는 사람이 새 시대의 승자가 됩니다.

TDD (Test-Driven Development) — 테스트 주도 개발

테스트를 먼저 쓰고, 통과하는 코드를 나중에 쓴다. AI 시대에도 원칙은 동일.

1. RED — 인간이 실패하는 테스트 작성
2. GREEN — AI가 통과 코드 구현
3. REFACTOR — 인간이 리뷰 → AI가 정리
버그 40~80% 감소 — 2025 DORA
AI의 치팅에 주의
· 테스트 삭제/skip으로 통과
· assert 약화 (값 → 타입 검증)
· 구현 먼저, 거기 맞춘 테스트
→ 테스트 수정 금지 필수 규칙
SDD (Spec-Driven Development) — 스펙 주도 개발

코드가 아닌 스펙이 진실의 원천. 무엇을(WHAT) 먼저 정의하고, 어떻게(HOW)는 AI에게.

1. SPECIFY — 인간이 "무엇을" 정의
요구사항 · 수용 기준 · 제약조건
언제 쓰나: 신규 기능 · 대규모 설계 · 여러 세션에 걸친 작업 · 팀 협업 (스펙 = 커뮤니케이션)
2. PLAN — AI가 "어떻게" 제안
기술 스택 · 아키텍처 → 인간 승인
3. TASKS — 분해 후 병렬 실행
작업↔요구사항 매핑 유지. 모호한 부분 → [NEEDS CLARIFICATION] 마커로 구현 중단.
핵심: 각 단계가 파일로 저장 · 커밋 · 참조 → 컨텍스트 소실 방어의 정석
결론: Spec + TDD

🎯 하네스가 해결하는 두 가지 근본 문제

하네스의 존재 이유를 이해하려면 딱 두 가지 문제만 기억하면 됩니다.

🧠
문제 ①
컨텍스트 부패
정보량의 문제. AI가 한 번에 볼 수 있는 "페이지 수"(컨텍스트 창)가 꽉 차면, 앞에서 한 얘기를 잊기 시작합니다. 작업이 길어질수록 점점 멍청해집니다.
하네스의 해결책: CLAUDE.md
새 세션마다 AI가 가장 먼저 읽는 온보딩 문서. 컨텍스트가 꽉 차서 앞을 잊어도 이 파일은 항상 다시 읽음.
🚧
문제 ②
규칙 · 울타리 부재
구조적 제약의 문제. AI가 정보를 다 알면서도 엉뚱한 짓을 합니다. 예: 결제 시스템 만들라고 시켰는데 DB 테이블을 삭제. "절대 하면 안 돼"가 구조로 없기 때문.
하네스의 해결책: Hooks
작업 완료 직전 자동 실행되는 스크립트. 타입체크·테스트 에러 시 저장 자체가 막힘. 부탁이 아닌 강제.
정리: 하네스는 이 두 문제를 풀기 위해 등장했습니다. 정보량 문제 → CLAUDE.md로 컨텍스트를 매번 복구. 규칙 문제 → Hooks로 제약을 자동 강제. 이 둘만 있어도 AI 에이전트의 안정성이 극적으로 올라갑니다.

🔄 Human In / On / Out of the Loop — 사람의 역할이 바뀐다
👤→🤖
Human-in-the-Loop
일일이 지시 · 일일이 확인
사람이 AI에게 매 단계마다 지시를 내리고, 결과를 확인하고, 직접 수정합니다.
마음에 안 들면? → 결과물을 직접 고친다
👤⟳🤖
Human-on-the-Loop ⭐
가이드를 주고 · AI가 스스로
사람이 가이드(하네스)를 설계하면, AI가 스스로 판단하고 실행합니다. 사람은 결과를 모니터링.
마음에 안 들면? → 하네스를 고친다
🤖→🤖
Human-out-of-the-Loop
AI가 하네스까지 스스로 진화
On the Loop의 진화형. 사람이 초기 하네스를 설정하면, AI가 기획→개발→테스트 루프를 반복하며 스스로 하네스를 보강해 나갑니다.
핵심: AI가 실패에서 학습 → 하네스 자체를 개선
Ralph Loop — Geoffrey Huntley (오픈소스 개발자, .NET 커뮤니티 리더)가 소개한 자기 진화 루프 모델.

AI에게 목표를 주면, 단일 프로세스 안에서 실행 → 검증 → 수정을 반복하며 스스로를 진화시켜 문제를 끝까지 해결합니다.

🎯 목표 → ⚡ 실행 → ✅ 검증 → 🔧 수정 → 🔄 반복
즉, AI가 하네스를 스스로 보강하며 진화하는 개념. 사람은 최초 목표와 초기 하네스만 제공합니다.
결과물이 아니라, 결과물을 만들어내는 시스템을 개선하는 것.
Claude Code, Cursor, Windsurf 같은 코딩 에이전트가 바로 Human On the Loop의 대표 사례입니다. 사람은 CLAUDE.md를 쓰고, 테스트를 설계하고, 하네스를 개선합니다. 코드는 에이전트가 씁니다.

🧱 하네스의 5대 구성요소

실무에서 반복 등장하는 5가지 핵심 축. 각각이 없을 때 어떤 리스크가 발생하는지가 설계의 출발점.

1
지식 베이스 CLAUDE.md · AGENTS.md
레포지토리 안에 저장된 아키텍처 결정, 프로젝트 컨텍스트, 실행 규칙. 에이전트의 "헌법".
없으면: 컨벤션 무시, 결정사항 재질문
2
실행 환경 샌드박스 · 파일시스템
모든 행동을 미리 도구로 만들 수 없다 → bash와 코드 실행이 범용 도구. 단, 격리된 샌드박스에서 실행.
없으면: 위험한 명령 실행
3
컨텍스트 관리 — 핵심 전장 ⭐
컨텍스트가 채워질수록 에이전트가 멍청해진다 (Context Rot). 대응: 컴팩션(요약·덜어내기) · 도구 출력 오프로딩 · 서브에이전트로 컨텍스트 격리
없으면: Context Rot, 점점 멍청해짐
4
검증 게이트 테스트 · 타입체크 · 린터
에이전트가 작업 완료를 선언하기 전 반드시 통과해야 하는 테스트·체크·승인 루프. "완료"의 기준이 에이전트의 판단에만 의존하지 않도록.
없으면: 미완성이 완료 처리
5
피드백 루프 Hook · Post-Mortem · Memory
에이전트가 동일한 실패를 반복할 때 이를 감지하고 경로를 수정하는 자기 교정 메커니즘. 실수 → 규칙으로 인코딩.
없으면: 같은 실수 반복
🛠️ 하네스 설계 5단계
1
관찰
에이전트가 실패하는 패턴을 모읍니다. "어디서 틀렸나?"를 기록합니다.
2
분류
실패를 Guide로 해결할 것(사전에 알려주면 되는 것) vs Sensor로 해결할 것(사후에 잡아야 하는 것)으로 분류합니다.
3
Guide 구현
CLAUDE.md에 규칙 추가, 아키텍처 제약 문서화, 프롬프트 템플릿 설계.
4
Sensor 구현
테스트 스위트, 린터 규칙, Git Hooks(pre-commit, post-edit) 설정.
5
반복
새로운 실패 패턴 발생 → 1단계로 돌아감. 하네스는 한 번 만들고 끝이 아닙니다.

조합하는 패턴 — Plan-Critic-Build, Ralph Loop

도구를 알았으니, 조합하는 패턴입니다. 에이전트가 가장 많이 실패하는 지점: 계획 없이 바로 코드 작성. 해결책은 단순 — 계획 단계를 쓰기 권한 없는 모드로 분리.

Plan–Critic–Build — 계획과 실행의 분리
1. PLAN
읽기만 · 쓰기 금지
파일 읽기·검색·질문만. 결과: 편집 가능한 계획 문서.
2. CRITIC
다른 눈으로 본다
리뷰 전용 에이전트. 작성자의 blind spot을 드러내기.
3. BUILD
승인 후 실행
승인된 계획 링크만 컨텍스트로. 쓰기 권한 개방.
핵심: Planner와 Critic을 같은 컨텍스트에 두지 않기. 자기 계획 → 자기 비평 시 blind spot 잔존.

컨텍스트의 4가지 실패 모드 — Drew Breunig
Drew Breunig — 데이터·AI 분야의 인디 연구자/블로거. 2024년 "How Long Contexts Fail" 글에서 컨텍스트 윈도우 실패 패턴을 4가지로 정리해 LLM 운영 담론의 표준이 됨.
POISONING · 중독
환각 1회 → 이후 오류 누적
초반 오답 1개로 평균 39% 성능 저하
DISTRACTION · 산만
히스토리 과의존, 실패한 액션 재시도
CONFUSION · 혼동
무관한 정보 과잉 → 톨 30개+에서 급증
CLASH · 충돌
모순된 정보 → 추론 붕괴
핵심은 "뭘 넣을까"보다 "뭘 뺄까". 컨텍스트 = 길이가 아닌 신호 대 잡음비.
실패 패턴 5가지
01
컨텍스트 과부하
"많이 주면 잘하겠지" — Rot · Distraction · Confusion의 시작
02
불명확한 요구사항
"이거 좀 고쳐줘" → LLM은 학습 데이터 기반 추측 구현. 맞아도 재현성 없음
03
검증 없는 자동수락
"AI가 만들었으니 맞겠지" — 가장 위험한 생각. 자기 작업 평가 시 병리적 낙관주의자
04
긴 세션 드리프트
시점 편향 · 오류 중첩 · Mode Collapse — 초반 규칙을 잊고 방향 상실
05
권한 과다
"바이패스가 편하니까" — 프롬프트 인젝션 · 토큰 유출 · 프로덕션 사고의 단일 경로

LLM Wiki — 지식의 컴파일 NEW · 2026.04

2026년 4월, Andrej Karpathy가 GitHub gist 하나를 던졌습니다. 제목은 단 두 단어 — "LLM Wiki". 며칠 만에 AI 커뮤니티에서 폭발적 화제가 됐습니다. 우리가 Ch.04~08까지 배운 하네스 엔지니어링의 자연스러운 확장이며, Part 3에서 본 Memory 패턴의 야심찬 진화형입니다.

💡 이전 챕터와의 연결

Ch.04~05의 하네스 엔지니어링이 "한 번의 작업에서 컨텍스트를 잘 관리하는 법"이었다면, LLM Wiki는 "프로젝트 전체의 수명 동안 지식을 누적하는 법"입니다. 같은 사고방식의 다른 시간 척도.

🔥 문제 — RAG는 매번 처음부터 시작한다
"LLM이 매 질문마다 처음부터 지식을 재발견하고 있다. 누적이 없다."
— Andrej Karpathy, 2026.04 (LLM Wiki gist)
😫 RAG의 비유
매일 같은 책 5권을 처음 보는 사람처럼 답한다. 어제 답했던 질문이 와도 책을 처음부터 다시 펼친다.
✨ LLM Wiki의 비유
책을 읽고 노트를 정리한 학생처럼 답한다. 같은 질문이 오면 노트만 본다. 새 책이 오면 노트에 추가한다.
핵심 통찰: "검색"이 아니라 "기억의 운영"이다.

🎯 LLM Wiki — 전체 플로우 한 장
Raw Sources · The Wiki · Schema (아키텍처 3계층) 사이에서, Ingest · Query · Lint (3가지 운영)가 Human과 LLM Agent에 의해 돌아갑니다. 화살표를 따라가면 데이터가 어떻게 흐르는지 보입니다.
아키텍처·Lint 흐름
Ingest 쓰기 흐름
Query 읽기/쓰기
사람↔운영 연결
① 아키텍처 3계층
상단의 Raw Sources는 외부에서 들어오는 원본(PDF·논문 등, 읽기 전용). The Wiki는 중앙의 LLM 전용 markdown 저장소. Schema(CLAUDE.md, schema.md)는 사람이 만든 운영 규칙으로 모든 작업을 "이렇게 해라"고 가이드합니다. 화살표: Raw → Wiki (자료 유입), Schema → Wiki (규칙 주입, 점선).
② Ingest — 쓰기 루프 (주황)
Human이 새 자료를 던지면 LLM Agent가 ① 읽고 → ② entity 추출 → ③ 요약 작성 → ④ 상호참조 → ⑤ index 갱신 → ⑥ log 기록. 이 과정에서 wiki의 markdown 파일들이 실제로 써지고, 지식이 누적됩니다.
③ Query — 읽기 + 피드백 루프 (청록)
Human이 질문 → LLM이 ① index 검색 → ② 관련 entity 읽기 → ③ 답변 합성. 핵심은 ④ "file back": 답변 과정에서 얻은 새 통찰을 다시 wiki에 써넣어, 다음 질문이 더 빨라지고 풍부해집니다. RAG와 달리 매번 제로에서 시작하지 않습니다.
④ Lint — 건강 검진 (보라)
LLM Agent가 주기적으로 wiki 전체를 스캔 — 모순(같은 사실의 다른 설명) · 고아 페이지 · 죽은 링크 · stale entity를 찾습니다. 자동 수정 가능한 건 바로 fix & patch, 나머지는 "review?" 화살표로 Human에게 승인 요청.
⑤ 참여자 분담 — 누가 뭘 하나
Human은 시작점과 종점만 맡습니다 — 원본 자료 큐레이팅, Schema 작성, 질문 던지기, 최종 승인. LLM Agent는 그 사이 모든 반복 작업을 자동화합니다 — Ingest · Query · Lint 실행, entity 생성, 상호참조, 요약, 제안. 이것이 Karpathy가 말한 "지식의 컴파일"입니다 — 사람은 의도만, LLM은 집행.
3가지 핵심 운영 — Ingest · Query · Lint 상세
OPERATION 1
📥 Ingest
새 자료가 들어오면:
① 자료를 읽는다
② 등장하는 entity 추출
③ 기존 페이지 10~15개 갱신
④ 새 entity 페이지 생성
⑤ index와 log에 기록
OPERATION 2
🔍 Query
사용자가 질문하면:
① index에서 카테고리 찾기
② 관련 entity 페이지 읽기
③ 답변 합성
④ 새 통찰을 wiki에 편입
⑤ 다음 질문은 더 빠르고 풍부
OPERATION 3
🧹 Lint
주기적 점검:
① 모순 검출 (같은 사실 다른 설명)
② 고아 페이지 (index에 없는 것)
③ 죽은 링크
④ stale entity (오래된 것)
⑤ 사람의 검토 요청
💡 Query → Ingest 피드백 루프가 핵심입니다. 답변 자체가 다음 질문을 위한 자산이 됩니다. 그래서 시간이 지날수록 위키가 더 똑똑해집니다.
RAG vs LLM Wiki — 무엇이 다른가
관점 RAG LLM Wiki
지식 처리 시점 매 질문마다 (검색 시점) 한 번 컴파일, 계속 유지
지식 누적 없음 — 매번 새로 발견 있음 — 자산화
상호참조 매번 새로 발견 이미 연결되어 있음
모순 처리 그때그때 (불일관) 추적 · 플래그 · 사람 리뷰
사람이 직접 읽기 어렵다 (벡터 임베딩) 쉽다 (markdown)
갱신 비용 낮음 (인덱싱만) 중간 (LLM이 정리)
💭 비유: 인터프리터 vs 컴파일러. RAG는 매번 해석, LLM Wiki는 미리 컴파일.

⚠️ 한계와 주의점 — 만능은 아니다
⚠️ 잘못된 요약의 굳기
사람의 검토 없이 LLM이 만든 요약이 "공식 지식"으로 굳을 위험. 사람이 주기적으로 lint를 돌리고 검증해야 함.
⚠️ 모순 누적
자료 출처가 서로 모순이면 위키가 혼란스러워짐. Lint로 잡아내지만, 해결은 사람이 해야 함.
⚠️ 휘발성 자료
대상이 너무 자주 바뀌면 갱신 비용 > 검색 비용. 안정적이고 누적되는 지식에 적합하다.
