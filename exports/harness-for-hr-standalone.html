AI 에이전트와 사람의 협업, 그리고 조직/직무의 미래
개발자 중심 용어를 HR 관점의 협업 시나리오로 번역합니다. 기술 용어의 원래 정의는 정확히 유지하되, 각 개념이 조직과 직무에 무엇을 의미하는지를 함께 봅니다.

DEFINITION

하네스 엔지니어링 (Harness Engineering)

AI 모델(LLM)이 실제 업무를 수행할 수 있도록, 모델 외부에서 도구, 지식, 검증, 제약, 피드백을 하나의 실행 구조로 설계하는 활동.

AI 팀원 한 명이 조직에서 일하려면, 그를 둘러싼 시스템(지침 문서, 도구 접근권, 결재선, 품질 기준, 회고 절차)이 반드시 함께 있어야 합니다. 하네스는 바로 이 "둘러싼 시스템"을 의미합니다.

WHY "HARNESS"?

왜 "Harness"라는 이름이 붙었을까

"Harness"는 영어로 마구(馬具). 말의 힘을 억누르는 것이 아니라 올바른 방향으로 집중시켜 빠르고 정확하게 달리게 하는 장비입니다. 야생말을 경마장에서 의도대로 달리게 하려면 반드시 필요하죠. AI도 똑같습니다.

1-3. 시대의 중첩

Prompt ⊂ Context ⊂ Harness (안쪽 정의가 바깥에 포함됨)
Harness Engineering
에이전트가 일할 수 있는 전체 시스템을 어떻게 설계하는가
Context Engineering
모델이 어떤 정보를 기반으로 판단하도록 설정하는가
Prompt Engineering
모델에게 무엇을 어떻게 질문하는가
각 시대가 이전 시대를 대체가 아닌 포함합니다. 직원이 쌓은 프롬프트 기술은 컨텍스트 엔지니어링에서, 컨텍스트 설계 역량은 하네스 엔지니어링에서 그대로 활용됩니다. 안쪽 능력 없이 바깥만 잘하는 것은 불가능합니다.

2대 근본 문제와 해결 원리

왜 모델만으로는 부족한가
🧠
모델만
텍스트 입력 → 텍스트 출력 함수.
  • 상태 유지 불가 (세션 간 기억 없음)
  • 실행 불가 (파일, 시스템 접근 없음)
  • 환경 접근 불가 (외부 도구 호출 없음)
  • 검증 없음 (자기 결과 확인 불가)
🧠+🔧
모델 + 하네스
실제 업무 수행 가능.
  • 지식 문서로 상태 유지
  • 샌드박스로 안전 실행
  • 도구 호출로 환경 접근
  • 검증 게이트로 품질 보장

2-3. 프롬프트는 부탁, 하네스는 강제

같은 상황, 다른 결과. 구체적 예시로 보는 근본적 차이
🙏
APPROACH A
프롬프트 = 부탁
말로 지시 · 의지에 의존
💬 "민감정보 공유하지 마세요"
1️⃣사람이 프롬프트에 주의사항 기재
↓
2️⃣AI가 매번 스스로 판단해서 지킬지 결정
↓
⚠️잊거나 오해 → 실수 반복 가능
📊 재발률: 높음
VS
🛡️
APPROACH B
하네스 = 강제
구조로 차단 · 의지와 무관
🔒 시스템이 민감정보 전송 차단
1️⃣사람이 Hook 규칙을 한 번 정의
↓
2️⃣매 작업마다 시스템이 자동 검증
↓
✅위반 시도 → 실행 자체 차단
📊 재발률: 0 (원천 불가)
🎯 3가지 구체적 상황에서의 차이
상황 🙏 프롬프트 방식 (부탁) 🛡️ 하네스 방식 (강제)
① 채용 공정성 "성별·나이로 필터링하지 마세요"를 프롬프트에 기재. AI가 잊거나 우회할 수 있음. 민감 속성 필드 자체를 AI 입력에서 제거. 접근할 수 없으므로 편향 물리적 불가능.
② 연봉 데이터 유출 "외부 공유 금지"를 CLAUDE.md에 적음. 긴 대화 중 AI가 놓칠 수 있음. 외부 도메인 전송 시 Pre-Send Hook이 자동 차단·보안팀 알림. AI 판단 불필요.
③ 퇴사 결정 자동 실행 "퇴사 처리는 사람 확인 후에만"이라 지시. AI가 에이전틱 모드에서 스스로 실행할 위험. 퇴사 API 호출 권한을 AI에게 부여 안 함. 오직 HR 담당자 승인 후에만 활성화.

2-4. 하네스 전체 아키텍처

5대 구성요소가 Guides(사전) · Sensors(사후) 축으로 작동하는 전체 그림
📖 이 그림을 읽는 법
좌측(Guides)은 행동 전에 방향을 잡아주는 영역, 우측(Sensors)은 행동 후 결과를 검증하는 영역입니다. 중앙의 Model은 양쪽 통제를 받으며 작동합니다. 실패가 발생하면 빨간 루프를 타고 다시 지식 베이스로 돌아와 규칙이 업데이트됩니다. 즉, 하네스는 정적인 문서가 아니라 학습하는 시스템입니다.
🎯 하나의 예시로 보는 5대 구성요소 — "AI 채용 스크리너"
가상 시나리오: HR이 이력서 1차 스크리닝용 AI 에이전트를 운영합니다. 다섯 가지 구성요소가 어떻게 함께 작동하는지 봅니다.

3-2. 단계별 협업 풍경

각 단계에 HR 직무 3건 예시
🔹

In the Loop (현재)

원 정의: 사람이 AI의 매 단계 출력을 확인하고 수정하는 협업 방식.
💻 개발자 예시: 엔지니어가 AI 출력을 한 줄씩 "이렇게 고쳐" 지시.
📊 HR Analyst
AI 결과를 매번 검토하고 수정
🎯 채용 담당자
AI가 요약한 이력서를 하나하나 재확인
📝 L&D 매니저
AI가 초안한 교육 자료를 문장 단위 편집
결정적 순간: 결과물을 직접 수정.
🔸

On the Loop (전환 지점) ⭐

원 정의: 사람이 하네스(지침, 규칙, 검증)를 설계하고, AI는 그 안에서 자율 실행. 사람은 모니터링과 예외 처리를 담당.
💻 개발자 예시: CLAUDE.md + Hooks 설계 → AI가 자율 구현.
📊 HR Analyst
분석 SOP + 데이터 품질 규칙(Hooks) 설계 → AI가 매주 자동 리포트
🎯 채용 담당자
공정성과 평가 기준(CLAUDE.md) 설계 → AI가 1차 스크리닝 자율
📝 L&D 매니저
교육 품질 기준(Sensors) 설계 → AI가 콘텐츠 자율 생성, 기준 미달만 반려
결정적 순간: 하네스(시스템) 수정. 직무 본질의 전환점.
🔷

Out of the Loop (미래)

원 정의: AI가 스스로 Ralph Loop(실행 → 검증 → 수정 반복)로 하네스를 진화시킴. 사람은 최초 목표와 초기 하네스만 제공.
🤝 HR 협업 예시
AI 에이전트가 자기 실패 케이스를 분석해 자기 온보딩 문서를 자동 업데이트. 예: 신입 사원 FAQ 챗봇이 답변 만족도 저조 질의를 주간 단위로 집계하여 자기 지식 문서를 스스로 보강.
HR 함의: 직원 역할이 설계자에서 큐레이터이자 감독으로 한 단계 더 이동.

"모두가 In → On → Out으로 순서대로 간다"기보다, 조직 안에서도 업무 성격에 따라 각 단계가 병렬적으로 운영될 수 있습니다. 예를 들어 같은 HR 조직 안에서도 다음과 같이 분리되는 것이 합리적일 수 있습니다.
🔹 In the Loop 유지: 고위험 업무 (해고, 보상 결정, 법적 대응)
🔸 On the Loop: 반복 업무 (데이터 리포트, 1차 스크리닝, 정기 공지)
🔷 Out of the Loop: 지속 개선 영역 (FAQ 챗봇, 문서 관리, 온보딩 자료)

우리 Buffalo 프로젝트에 하네스 엔지니어링은 어떤 시사점을 주는가
Q1.
하네스 설계가 필요한 프로세스는 무엇일까?
💭 AI가 더 잘 뛰도록 "둘러싼 환경"을 정성껏 다듬어줄 프로세스는 어디인가
💡 모범 답안 보기 ▼
프레이밍 전환: "어디부터 하지?"라는 우선순위 질문이 아니라, "환경 설계가 가장 큰 차이를 만드는 곳은 어디인가?"라는 레버리지 질문입니다. 자동화 대상이 아니라 "조직의 지식이 가장 빛날 곳"을 찾는 일입니다.
4가지 시그널이 동시에 성립하는 영역
  1. 조직 맥락 의존도가 높은 일 일반 LLM 지식만으로는 풀 수 없고, 우리 조직만의 원칙·관행이 결정적인 영역
  2. 같은 종류의 판단이 반복되는 일 첫 회는 사람이 정의하고 그다음부터 시스템이 누적되는 구조가 가능
  3. 결과의 일관성·공정성이 사람마다 흔들리는 영역 하네스가 사람보다 더 일관된 표준이 됨
  4. 인수인계가 잦아 맥락이 자주 끊기는 일 하네스가 "조직의 영구 기억"으로 작동
HR 영역에서의 후보 매핑

Q2.
우리 조직의 지금 규정·SOP는 AI가 바로 따라 할 수 있을 만큼 명확한가?
💭 하네스보다 먼저 손볼 곳은 없는지
💡 모범 답안 보기 ▼
본질의 재정의: 이 질문은 사실 "우리 조직의 암묵지가 얼마나 형식지화되어 있는가?"와 같은 질문입니다. 노나카의 SECI 모델로 보면, 하네스는 "암묵지 → 형식지" 전환의 산물입니다. SOP의 명확성을 점검한다는 것은 우리 조직의 지식 관리 성숙도를 거울에 비추는 일입니다.
SOP 점검 4층위
  1. 존재 문서화된 형태가 있긴 있는가
  2. 명확성 케이스별 판단 기준까지 적혀 있는가, 아니면 "상황에 맞게 처리"로 끝나는가
  3. 최신성 마지막 업데이트가 언제이고, 실제 운영과 일치하는가
  4. 구조성 사람이 읽기 좋은 문서와 기계가 따라 할 수 있는 문서는 다릅니다
대부분의 조직이 마주하는 진실
  • "우리는 SOP가 잘 정리되어 있다"고 답하지만, 실제로는 베테랑의 머릿속에 있는 경우가 다수
  • 문서화된 SOP조차 "예외 처리 로직"이 빠져 있어 신입은 그대로 따라 하지 못함
  • 즉, AI 도입 시도가 우리 조직의 지식 관리 성숙도를 드러내는 진단이 됩니다
📌 핵심 통찰: 하네스 프로젝트의 첫 번째 가치는 "AI가 일을 잘하게 하는 것"이 아닙니다. "우리가 무엇을 알고 있는지를 처음으로 글로 적어보는 것"이 더 큰 가치입니다. 이 과정 자체가 조직 학습이며, 베테랑이 떠나도 살아남는 조직 자산을 만듭니다. AI 도입은 명분일 뿐, 진짜 산출물은 SOP의 재발견입니다.
Q3.
"실행자 → 조직 설계자"로 전환할 때 각 직무는 무엇이 늘고 무엇이 줄어들까?
💭 역량 모델·JD가 어떻게 다시 쓰여야 할지
💡 모범 답안 보기 ▼
변화의 본질: 단순히 "할 일 목록의 교체"가 아니라 "자기 노동의 레버리지 비율이 바뀌는 것"입니다. 실행자의 시간은 "지금 이 한 건"에 흐르고, 설계자의 시간은 "앞으로 천 건"에 흐릅니다. 시간의 방향이 현재에서 미래로 바뀝니다.
늘어나는 것메타 인지("이 판단의 기준이 무엇인가") · 케이스 패턴 분석(1건 아닌 100건) · 의사결정 기준의 명문화 · 예외 발견과 학습 · 시스템 디버깅
줄어드는 것반복 실행과 정형 응대 · 개인 경험에 의존한 개별 판단 · "내가 직접 해야 안심"이라는 심리
가장 어려운 변화 (정체성 차원)
"나의 가치는 더 이상 처리량으로 측정되지 않는다"라는 사실은 직무 정체성의 근본적 흔들림입니다. 평생 "잘하는 사람 = 빨리 많이 처리하는 사람"이라고 학습해온 직원에게는 가장 큰 심리적 장벽이 됩니다. HR이 가장 무겁게 다뤄야 할 영역입니다.
현실적 시사점
  • 모든 직원이 설계자 역량을 갖추는 것은 비현실적
  • 일부는 "감독·설계 역할"을 부담스러워하거나 거부할 가능성
  • 따라서 이중 트랙(실행 전문가 트랙 / 설계자 트랙) 운영을 검토해야 함
  • JD 변화: "월 N건 처리"에서 "N개 케이스의 패턴 정의"로
  • 역량 모델 변화: 시스템 사고, 케이스 패턴 분석, 명문화 능력, 메타 인지가 새로 추가
📌 핵심 통찰: 직무 변화는 표면적으로는 "역량 모델 업데이트"이지만, 본질은 "성과 정체성의 재구성"입니다. 처리량으로 자기를 측정하던 사람이 시스템 품질로 자기를 측정하기까지의 심리적 전환을 HR이 디자인해야 합니다.
Q4.
하네스를 만들고 유지하는 주체는 누가 되어야 할까?
💭 HR·IT·현업의 역할 분담과 오너십
💡 모범 답안 보기 ▼
역사적 관점: 이 질문의 본질은 "도메인 지식과 기술 구현이 분리될 때 누가 책임지는가"입니다. BPM, RPA, 워크플로우 시스템 모두 같은 질문에 부딪혔습니다. 하네스는 그중에서도 가장 도메인 의존도가 높습니다.
3가지 모델 비교
A. IT 주도 ❌IT가 도구를 만들고 HR이 사용. 도메인 뉘앙스·예외 처리·조직 정치를 IT가 알 수 없음 → "작동은 하지만 맞지 않는" 시스템이 됨
B. HR 주도 ⚠️HR이 SOP·규정 작성하면 그게 곧 하네스. 기술적 강제 한계를 모르므로 "종이 규정 수준"에서 멈춤
C. 공동 오너십 ✅HR=What/Why의 오너, IT=How의 오너. 그 사이를 잇는 신규 역할(하네스 큐레이터 / Agent Operations Lead)이 핵심
오너십 매트릭스
정책·원칙 정의HR 단독
기술 구현·인프라IT 단독
케이스 라이브러리·규칙 진화HR + 현업 (실패 사례를 가장 먼저 만나는 사람)
보안·감사·윤리 게이트HR + IT + 법무 공동
성능 모니터링·운영IT 단독
거버넌스·KPI 정의HR (CHRO 라인)
📌 핵심 통찰: 이 변화는 단순한 "협업 모델"이 아니라 새로운 직군의 등장을 의미합니다. 데이터 사이언티스트가 통계와 도메인 사이에서 등장한 것처럼, 하네스 큐레이터는 HR과 시스템 설계 사이에서 등장합니다. 현실적 권고: 초기에는 소규모 cross-functional 스쿼드(HR 1~2명 + IT 1~2명 + 현업 1명)로 시작하고, 1년 안에 "Agent Ops" 정식 직군 신설을 검토하는 것이 안전합니다.
Q5.
성공했다는 걸 무엇으로 확인할까?
💭 활용률 같은 입력 지표 말고, 진짜 가치가 드러나는 결과 지표
💡 모범 답안 보기 ▼
전제: Goodhart's Law — "측정값이 목표가 되는 순간 더 이상 좋은 측정값이 아니다". KPI는 단순할수록 게임화되기 쉽습니다. 좋은 KPI는 결과의 본질을 우회하기 어려운 것이어야 합니다.
잘못된 KPI (게임화에 약함)
  • AI 활용률 (모두가 사용한다 ≠ 가치 창출)
  • 처리 건수 (양만 늘면 의미 없음)
  • 자동화율 (자동화 자체는 목적이 아님)
좋은 KPI 4층위
L1 학습 능력 ⭐같은 실수의 재발률 · 신규 규칙 추가 빈도 · 하네스 업데이트 주기 → 시스템이 학습하고 있는가
L2 결과 품질사람이 다시 판단하기 위해 개입한 비율 · 의사결정 일관성(편차) · 예외 케이스 처리 시간
L3 시간·비용 재배치절감된 시간이 어디로 갔는가(실행→설계로 이동했나) · 시스템 설계 활동에 쓴 시간 비율 증가
L4 조직 신뢰AI 결정에 대한 직원·고객 신뢰도 설문 · 사람이 AI 결정을 뒤집은 비율(높으면 신뢰 부족, 낮으면 검토 무의미) · 컴플레인·이의제기 건수
L1이 가장 중요한 이유: 하네스의 본질은 "학습하는 시스템"이기 때문입니다. 학습이 멈춘 하네스는 죽은 자산입니다.
📌 핵심 통찰: 하네스의 성과는 "AI가 무엇을 했는가"가 아니라 "조직이 무엇을 학습했는가"로 측정해야 합니다. 이 관점이 일반 자동화 프로젝트와 결정적으로 갈리는 지점입니다. 같은 실수가 두 번 일어나지 않는 조직, 시간이 흐를수록 더 똑똑해지는 조직 자산이 만들어지고 있다면, 그것이 곧 성공입니다.
